---
title: Monitoring API
description: Read-only HTTP API for evaluation results, leader status, container logs, and round history.
---

The monitoring API is a read-only HTTP surface that exposes the validator's on-disk state. Miners can check their evaluation results, see why they were disqualified, and pull raw container logs without needing SSH access to the validator.

**Base URL:** `https://api.cacheon.ai`

**Interactive docs (Swagger UI):** [`https://api.cacheon.ai/docs`](https://api.cacheon.ai/docs)

## Endpoints

### Health and status

| Method | Path          | Description                                                                 |
| ------ | ------------- | --------------------------------------------------------------------------- |
| GET    | `/api/health` | Liveness check. Returns `{"status": "ok"}` if the API is up                 |
| GET    | `/api/status` | Validator overview: current leader, eval counts, time since last evaluation |

```bash
curl https://api.cacheon.ai/api/health
curl https://api.cacheon.ai/api/status
```

### Leader

| Method | Path                  | Description                                          |
| ------ | --------------------- | ---------------------------------------------------- |
| GET    | `/api/leader`         | Current leader record (UID, score, image, won block) |
| GET    | `/api/leader/history` | Chronological list of overtakes                      |

### Evaluations

| Method | Path                             | Description                          |
| ------ | -------------------------------- | ------------------------------------ |
| GET    | `/api/evaluations`               | All evaluation records, newest first |
| GET    | `/api/evaluations?status=dq`     | Disqualified evaluations only        |
| GET    | `/api/evaluations?status=active` | Non-disqualified evaluations only    |
| GET    | `/api/evaluations/{uid}`         | All evaluations for a specific UID   |

```bash
curl https://api.cacheon.ai/api/evaluations/5
```

Each evaluation record includes `score`, `disqualify_reason` (if DQ'd), `ttft_improvement`, `throughput_improvement`, `token_match_rate`, and `per_prompt` stats when available.

### Container logs

| Method | Path                         | Description                           |
| ------ | ---------------------------- | ------------------------------------- |
| GET    | `/api/container-logs`        | List available container log labels   |
| GET    | `/api/container-log/{label}` | Raw Docker stdout/stderr (plain text) |

Labels follow the pattern `uid{N}_{hotkey8}_{block}` for challengers and `baseline_{hash}` for baseline runs.

```bash
curl https://api.cacheon.ai/api/container-log/uid5_5CG7KRPk_7105042
```

This is the fastest way to debug why your container failed. The log contains the full Docker output from your container's startup attempt.

### Validator logs

| Method | Path                         | Description                                      |
| ------ | ---------------------------- | ------------------------------------------------ |
| GET    | `/api/validator-logs`        | List available validator log files (CPU and GPU) |
| GET    | `/api/validator-log/{label}` | Raw validator process log (plain text)           |

Labels follow the pattern `cpu_validator_{YYYYMMDD}_{HHMMSS}` for CPU validator runs and `gpu_eval_{YYYYMMDD}_{HHMMSS}` for GPU eval runs.

```bash
curl https://api.cacheon.ai/api/validator-log/gpu_eval_20260518_112858
```

### Rounds

| Method | Path            | Description                                                  |
| ------ | --------------- | ------------------------------------------------------------ |
| GET    | `/api/rounds`   | Evaluation rounds grouped by block, with challenger outcomes |
| GET    | `/api/eval-job` | Current pending eval job (queued challengers)                |

## Debugging a failed evaluation

1. Find your UID's evaluation: `GET /api/evaluations/{your-uid}`
2. Check `disqualify_reason` in the response
3. Look up the code in [Disqualification reason codes](/docs/evaluation/scoring#disqualification-reason-codes)
4. Pull your container log: `GET /api/container-log/{label}` (format: `uid{N}_{hotkey8}_{block}`)
5. For Pass 2 infra failures, also check validator logs: `GET /api/validator-log/gpu_eval_{timestamp}` and `GET /api/container-log/baseline_scoring`

## Data freshness

The API reads from the validator's local state files, which are synced from S3 after each GPU eval round. Data updates after each eval completes (typically every few hours when challengers are queued).

## Rate limits

The API is rate-limited to 60 requests per minute per IP. All responses include `Cache-Control` headers; respect them to avoid hitting the limit.
