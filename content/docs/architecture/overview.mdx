title: Architecture overview
description: Single GPU machine running chain interaction, Docker container orchestration, vLLM baseline, and evaluation. 4x H200, mounted model weights, dual-network isolation.

Cacheon runs on a single GPU machine. The validator runs as a Docker container that orchestrates miner evaluations via the host Docker socket. Miner servers run in isolated sibling containers on an internal network. Model weights are mounted as a read-only volume.

## High-level topology

```
Bittensor chain
    |
    | (reads commitments, sets weights)
    v
GPU machine (4x H200, fixed NVLink topology)
    |
    |-- Validator container (chain, eval loop, weight-setting)
    |       connected to: cacheon-internal + cacheon-egress
    |-- vLLM baseline container (on-demand, tensor-parallel, cached)
    |       connected to: cacheon-internal only
    |-- Miner container (started per eval, isolated)
    |       connected to: cacheon-internal only
    |-- Model weights volume (/models, read-only)
```

## The validator container

The validator runs as a Docker container on the GPU machine with the Docker socket mounted:

- Holds the wallet keys and interacts with the Bittensor chain (via `cacheon-egress` network)
- Scans on-chain commitments for new challengers every block
- Validates commitment format (Docker image reference, sha256 digest)
- Starts miner Docker containers on the `cacheon-internal` network and reaches them by container IP
- Compares results against the current king
- Sets one-hot weights (`winner = 1.0`, everyone else `= 0.0`)

Everything happens on one machine. There is no SSH, no SFTP, no remote dispatch.

## Container orchestration

Each miner evaluation follows this lifecycle:

1. **Pull** -- Docker image is pulled by digest from the registry
2. **Start** -- Container starts with `--gpus all`, model volume mount, on the `cacheon-internal` network (no egress)
3. **Health check** -- Poll `/health` until 200 (timeout: 10 min)
4. **Warmup** -- 1-2 warmup prompts sent and discarded
5. **Evaluate** -- Two-pass: speed (streaming) then correctness (logprobs), 10 prompts each
6. **Teardown** -- Container is stopped and removed
7. **Cleanup** -- GPU state is reset across all 4 GPUs between evaluations

Containers run without `--privileged`, on an internal Docker network that blocks all outbound traffic, with model weights mounted read-only at `/models`.

## Model weight delivery

Model weights are stored on the machine at `/workspace/models/Qwen2.5-72B-Instruct` and mounted read-only into every container at `/models`. This guarantees:

- **Weight identity**: every miner and the baseline load the exact same bytes
- **Practical image sizes**: miner Docker images are 1-20 GB (runtime only), not 140 GB+
- **Fast pull times**: no model weights in the registry

## Data flow

```
Validator process (on GPU)
    |
    |-- scan chain for commitments
    |-- parse commitment (image + digest)
    |-- pull Docker image
    |-- start container (mount /models:ro, internal network)
    |-- warmup (1-2 prompts, discarded)
    |-- evaluate (speed + correctness, 10 prompts)
    |-- stop container, GPU cleanup
    |-- compare against king (decaying moat)
    |-- set_weights() to chain
    |-- sleep(poll_interval)
    |-- repeat
```
