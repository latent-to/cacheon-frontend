Cacheon's architecture separates chain interaction (CPU) from inference evaluation (GPU). Miner servers run in isolated Docker containers on the GPU pod. Model weights are mounted as a read-only volume.

## High-level topology

```
Bittensor chain
    |
    | (reads commitments, sets weights)
    v
CPU validator host
    |
    | (SSH/SFTP: sends eval jobs, receives results)
    v
GPU pod (4x H200, fixed NVLink topology)
    |
    |-- vLLM baseline (always-on, tensor-parallel)
    |-- Miner container (started per eval, isolated)
    |-- Model weights volume (/models, read-only)
    |-- Evaluation orchestrator
```

## CPU validator host

The CPU server is the chain-facing component:

- Scans on-chain commitments for new challengers every epoch (~360s)
- Validates commitment format (image reference, digest)
- Dispatches evaluation jobs to the GPU pod
- Receives results and compares against the king
- Sets one-hot weights (`winner = 1.0`, everyone else `= 0.0`)
- **Holds the wallet keys.** The GPU pod has no chain access.

The CPU host runs on commodity hardware. It does not need a GPU.

## GPU evaluation pod

The GPU pod runs the actual inference evaluation on **4x H200** with fixed NVLink topology:

- Hosts the vLLM baseline (always-on, tensor-parallel across all 4 GPUs)
- Stores model weights at a known path, mounted read-only into containers
- Pulls and starts miner Docker containers
- Runs the two-pass evaluation (speed + correctness) with warmup
- Returns structured results to the CPU host

The GPU pod has **no wallet keys** and **no chain access**. If it is compromised, the attacker gets model weights (which are public) and evaluation results, not control over subnet weights.

## Container orchestration

Each miner evaluation follows this lifecycle:

1. **Pull** -- Docker image is pulled by digest from the registry
2. **Start** -- Container starts with `--gpus '"device=0,1,2,3"'`, model volume mount, port mapping, no network
3. **Health check** -- Poll `/health` until 200 (timeout: 10 min)
4. **Warmup** -- 1-2 warmup prompts sent and discarded
5. **Evaluate** -- Two-pass: speed (streaming) then correctness (logprobs), 10 prompts each
6. **Teardown** -- Container is stopped and removed
7. **Cleanup** -- GPU state is reset across all 4 GPUs between evaluations

Containers run without `--privileged`, with restricted capabilities, read-only root filesystem, and no network access.

## Model weight delivery

Model weights are stored on the GPU pod at `/workspace/models/Qwen2.5-72B-Instruct` and mounted read-only into every container at `/models`. This guarantees:

- **Weight identity**: every miner and the baseline load the exact same bytes
- **Practical image sizes**: miner Docker images are 1-20 GB (runtime only), not 140 GB+
- **Fast pull times**: no model weights in the registry

## Why CPU/GPU split

The split is a security decision:

- **GPU pod runs arbitrary code.** Miner containers are sandboxed but the GPU is a real attack surface. Isolating it from wallet keys limits blast radius.
- **CPU host is lightweight.** Chain scanning, job dispatch, and weight-setting are cheap operations that don't need GPU access.
- **SSH narrows the interface.** The GPU pod's only external interface is the SSH connection to the CPU host.

## Data flow

```
CPU host                          GPU pod (4x H200)
   |                                 |
   | -- EvaluationJob (JSON) ------> |
   |    (image ref, prompts,         |
   |     block hash, params)         |
   |                                 | -- pull image
   |                                 | -- start container (mount /models:ro)
   |                                 | -- warmup (1-2 prompts, discarded)
   |                                 | -- evaluate (speed + correctness)
   |                                 | -- stop container, GPU cleanup
   |                                 |
   | <-- EvaluationResult (JSON) --- |
   |     (median ttft, median tps,   |
   |      token_match, score)        |
   |                                 |
   | -- set_weights() to chain       |
```

See [Data contracts](/docs/architecture/data-contracts) for the wire format.
