Cacheon runs on a single GPU machine. The validator process handles chain interaction, Docker orchestration, evaluation, and weight-setting. Miner servers run in isolated Docker containers on the same machine. Model weights are mounted as a read-only volume.

## High-level topology

```
Bittensor chain
    |
    | (reads commitments, sets weights)
    v
GPU machine (4x H200, fixed NVLink topology)
    |
    |-- Validator process (chain, eval loop, weight-setting)
    |-- vLLM baseline (always-on, tensor-parallel)
    |-- Miner container (started per eval, isolated)
    |-- Model weights volume (/models, read-only)
```

## The validator process

The validator is a single long-running process on the GPU machine:

- Holds the wallet keys and interacts with the Bittensor chain
- Scans on-chain commitments for new challengers every block
- Validates commitment format (Docker image reference, sha256 digest)
- Pulls miner Docker images and runs containerized evaluations locally
- Compares results against the current king
- Sets one-hot weights (`winner = 1.0`, everyone else `= 0.0`)

Everything happens on one machine. There is no SSH, no SFTP, no remote dispatch.

## Container orchestration

Each miner evaluation follows this lifecycle:

1. **Pull** -- Docker image is pulled by digest from the registry
2. **Start** -- Container starts with `--gpus '"device=0,1,2,3"'`, model volume mount, port mapping, no network
3. **Health check** -- Poll `/health` until 200 (timeout: 10 min)
4. **Warmup** -- 1-2 warmup prompts sent and discarded
5. **Evaluate** -- Two-pass: speed (streaming) then correctness (logprobs), 10 prompts each
6. **Teardown** -- Container is stopped and removed
7. **Cleanup** -- GPU state is reset across all 4 GPUs between evaluations

Containers run without `--privileged`, with restricted capabilities, read-only root filesystem, and no network access.

## Model weight delivery

Model weights are stored on the machine at `/workspace/models/Qwen2.5-72B-Instruct` and mounted read-only into every container at `/models`. This guarantees:

- **Weight identity**: every miner and the baseline load the exact same bytes
- **Practical image sizes**: miner Docker images are 1-20 GB (runtime only), not 140 GB+
- **Fast pull times**: no model weights in the registry

## Why a single machine

The original design split chain interaction (CPU host) from evaluation (GPU pod) over SSH, with the rationale that the GPU pod runs arbitrary code and should not hold wallet keys. This added real operational complexity (SSH tunnels, SFTP file transfers, detached processes, timeout polling) for a threat that is already mitigated by Docker container isolation.

In the collapsed design:

- **Miner code runs inside containers**, not on the host. The container cannot access the host filesystem, wallet keys, or chain connection.
- **One fewer failure mode.** No SSH timeouts, no SFTP transfer errors, no detached-process polling.
- **Simpler deployment.** One machine, one setup script, one process.

## Data flow

```
Validator process (on GPU machine)
    |
    |-- scan chain for commitments
    |-- parse commitment (image + digest)
    |-- pull Docker image
    |-- start container (mount /models:ro, --network none)
    |-- warmup (1-2 prompts, discarded)
    |-- evaluate (speed + correctness, 10 prompts)
    |-- stop container, GPU cleanup
    |-- compare against king (decaying moat)
    |-- set_weights() to chain
    |-- sleep(poll_interval)
    |-- repeat
```
