The validator is the CPU-side component that drives the competition loop.

## The main loop

```
while True:
    sleep(poll_interval)
    commitments = scan_chain_for_new_commitments()
    challengers = filter_unevaluated(commitments)

    if not challengers:
        re_set_weights(king)
        continue

    for challenger in challengers:
        result = dispatch_eval_to_gpu_pod(challenger)
        record_result(result)

    king = determine_king()
    set_weights(king)
```

### Commitment scanning

Every epoch (~360s), the validator reads on-chain commitments. Each commitment contains a Docker image reference and digest. The validator filters for commitments it has not yet evaluated.

### Challenger detection

A commitment becomes a challenger when:

1. It has been revealed on-chain (past the commit-reveal delay)
2. It has not been evaluated by this validator instance
3. Its image digest is not identical to the current king's (copycat DQ)

### Evaluation dispatch

For each challenger, the validator sends an `EvaluationJob` to the GPU pod over SSH/SFTP:

```json
{
  "challenger_hotkey": "5F...",
  "image": "docker.io/miner/server:v1",
  "digest": "sha256:abc...",
  "prompts": ["<passage1>", "<passage2>", "..."],
  "block_hash": "0xabc...",
  "max_tokens": 256,
  "schema_version": 2
}
```

The GPU pod pulls the image, runs the evaluation, and returns an `EvaluationResult`.

### King comparison

After evaluation, the result is compared against the king using the decaying moat:

```python
threshold = king_score * (1 + epsilon(age))
if result.score > threshold and result.correctness_pass:
    new_king = challenger
```

Where `epsilon(age)` starts at 1% and decays linearly to 0 over ~7 days.

### Weight-setting

Weights are one-hot:

```python
weights[king_uid] = 1.0
weights[everyone_else] = 0.0
subtensor.set_weights(netuid, weights, version_key=VERSION_KEY)
```

## State persistence

The validator maintains state across restarts:

- **King record** -- current king's hotkey, score, image digest, crowned-at block
- **Evaluation history** -- all past evaluation results
- **Commitment cache** -- which commitments have been seen and evaluated

State is stored as JSON files in a configurable state directory.

## Dry-run mode

The validator supports `--dry-run` mode which runs the full loop (chain scanning, commitment parsing, state management) without dispatching real GPU evaluations. Useful for testing the chain integration.

## Configuration

Key configuration via CLI flags or environment variables:

| Flag                     | Env var                        | Default  | Description              |
| ------------------------ | ------------------------------ | -------- | ------------------------ |
| `--network`              | `CACHEON_NETWORK`              | `finney` | `finney` or `test`       |
| `--netuid`               | `CACHEON_NETUID`               | `14`     | Subnet UID               |
| `--poll-interval`        | `CACHEON_POLL_INTERVAL`        | `360`    | Seconds between scans    |
| `--gpu-pod-ssh-host`     | `CACHEON_GPU_POD_SSH_HOST`     | --       | GPU pod SSH host         |
| `--gpu-pod-eval-timeout` | `CACHEON_GPU_POD_EVAL_TIMEOUT` | `1800`   | Max seconds per eval     |
| `--dry-run`              | --                             | `false`  | Skip real GPU evaluation |
