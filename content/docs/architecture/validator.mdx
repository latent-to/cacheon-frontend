The validator is a single process running on the GPU machine. It owns the wallet, reads the chain, orchestrates Docker-based evaluations, and sets weights.

## The main loop

```
while True:
    sleep(poll_interval)
    commitments = scan_chain_for_new_commitments()
    challengers = filter_unevaluated(commitments)

    if not challengers:
        re_set_weights(king)
        continue

    for challenger in challengers:
        result = evaluate_in_docker(challenger)
        record_result(result)

    king = determine_king()
    set_weights(king)
```

### Commitment scanning

Every block, the validator reads on-chain commitments. Each commitment contains a Docker image reference and digest. The validator filters for commitments it has not yet evaluated.

### Challenger detection

A commitment becomes a challenger when:

1. It has been revealed on-chain (past the commit-reveal delay)
2. It has not been evaluated by this validator instance
3. Its image digest is not identical to the current king's (copycat DQ)

### Evaluation

For each challenger, the validator runs a local Docker evaluation:

1. Pull the miner's Docker image by digest
2. Start the container with GPU access, model volume mount, internal network (no egress)
3. Wait for `/health` (timeout: 10 min)
4. Send 1-2 warmup prompts (discarded)
5. Run two-pass evaluation: speed (streaming, 10 prompts) then correctness (logprobs)
6. Stop the container, clean up GPU state
7. Compute score: `0.5 * max(0, ttft_improvement) + 0.5 * max(0, throughput_improvement)`

### King comparison

After evaluation, the result is compared against the king using the decaying moat:

```python
threshold = king_score * (1 + epsilon(age))
if result.score > threshold and result.correctness_pass:
    new_king = challenger
```

Where `epsilon(age)` starts at 1% and decays linearly to 0 over ~7 days.

### Weight-setting

Weights are one-hot:

```python
weights[king_uid] = 1.0
weights[everyone_else] = 0.0
subtensor.set_weights(netuid, weights, version_key=VERSION_KEY)
```

## State persistence

The validator maintains state across restarts:

- **King record** -- current king's hotkey, score, image digest, crowned-at block
- **Evaluation history** -- all past evaluation results
- **Commitment cache** -- which commitments have been seen and evaluated

State is stored as JSON files in a configurable state directory.

## Dry-run mode

The validator supports `--dry-run` mode which runs the full loop (chain scanning, commitment parsing, state management) without running Docker evaluations or setting weights. Useful for testing chain integration before the GPU machine is ready.

## Configuration

Key configuration via CLI flags or environment variables:

| Flag                | Env var                   | Default                   | Description                          |
| ------------------- | ------------------------- | ------------------------- | ------------------------------------ |
| `--network`         | `CACHEON_NETWORK`         | `finney`                  | `finney` or `test`                   |
| `--netuid`          | `CACHEON_NETUID`          | `14`                      | Subnet UID                           |
| `--poll-interval`   | `CACHEON_POLL_INTERVAL_S` | `360`                     | Seconds between main-loop iterations |
| `--model-volume`    | `CACHEON_MODEL_VOLUME`    | `/models`                 | Path to model weights on host        |
| `--gpus`            | `CACHEON_GPUS`            | `device=0,1,2,3`          | Docker `--gpus` device string        |
| `--baseline-image`  | `CACHEON_BASELINE_IMAGE`  | `vllm/vllm-openai:latest` | Baseline Docker image                |
| `--baseline-digest` | `CACHEON_BASELINE_DIGEST` | (empty)                   | Baseline digest (required for eval)  |
| `--dry-run`         | `CACHEON_DRY_RUN`         | `false`                   | Skip Docker eval + weights           |
