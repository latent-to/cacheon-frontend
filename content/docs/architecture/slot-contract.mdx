> This page is the invariant — the one thing that must never change. Everything
> *around* it (which slots exist, which sglang chokepoints we patch, correctness
> modes, eval knobs) is free to churn. The waist's job is to be **boring and
> stable**. If a feature can't satisfy the four rules below, it is not a core
> slot.
>
> The committed source is `docs/SLOT_CONTRACT.md` in the repo; the live code is
> `optima/slots.py` (`SlotSpec`) and `optima/seams.py` (the adapter table).

## Why a "waist"

Long-lived extensible systems are hourglasses: a *thin, stable, minimal* waist,
with a proliferation of things above it (slots) and below it (sglang seam
adapters) that change freely. The waist's job is to be **boring and ossified** —
that is the feature, not a bug. Cf. the Internet's IP layer, Unix syscalls, the
PyTorch dispatcher contract. We keep the waist thin so a "seemingly impossible"
submission becomes "add a slot / add an adapter / stage it in the escape hatch,"
never "rewrite everything."

## The four invariants (inviolable)

Every core slot must satisfy all four. If it can't, it is not a core slot — it
goes in the [fenced escape hatch](#the-escape-hatch-where-exotic-submissions-go-first) (experimental, framework
tier), or it doesn't exist.

### 1. The validator owns the boundary

The validator allocates **every output tensor** (shape / dtype / device / stride)
and owns the call site. The miner kernel only *fills* validator-allocated
outputs; it never allocates or returns the result the model consumes. Some slots
have one output (`out`); others have multiple outputs (`out_norm`,
`out_residual`) or an additional process-group argument for collectives. The
common rule is not the spelling of the last argument — it is validator-owned
storage and no miner-returned model tensor.

This is enforced literally in the dispatchers (`optima/dispatch.py`): the
dispatcher allocates the outputs, then calls `impl.entry(*inputs, *outputs, ...)`.
A miner cannot hand back a tensor for the model to consume; the validator takes
the buffers it allocated.

### 2. Strictly upstream of the sampler

A slot sits **above** the logprobs/sampler. Its output feeds the residual stream
→ downstream layers → sampler, all stock. There is therefore **no final output to
substitute** — this is what defeats the "run gibberish, fetch the real answer
from an API" attack.

This is a hard line. A slot that crosses the sampler line (e.g. "submit a whole
sampler") reintroduces the output-substitution attack: a miner could run
gibberish internally and fetch the real answer over the network. Keeping every
slot upstream of the logprobs is what keeps op/block/collective slots safe.

### 3. Gated against a validator-owned reference

The cheap slot-level correctness gate is measured against an **fp32 / dequant
reference**, **not against the stock kernel**. A faster, slightly-different
kernel can pass; a wrong one cannot.

The *metric* may vary per slot — `allclose`, `matched_ratio`, `cosine`,
`topk_overlap` — but the *verify reference* is always high-precision ground
truth. This matters because legitimate optimized kernels *do* differ from the stock kernel at
the ULP level (reordered softmax, fp8, weight-absorbed forms). If the cheap gate
were tied to stock bit patterns, those kernels would be false-failed; if it were
loose without a trusted reference, a wrong kernel could sneak through.

The production model-level fidelity gate is separate. In clean deterministic
arenas, `evaluate --fidelity-mode kl` compares rollout distributions against
stock. In noisy arenas, `evaluate --fidelity-mode audit` samples live dispatcher
calls and re-runs the captured stock baseline on cloned pre-call inputs under
the slot's tolerances.

### 4. No trust in miner-reported numbers

The score never trusts a value the miner's code could have produced.

- **Throughput** is timed out-of-process. The driver/timer process never imports
  the miner module (`seam.mark_driver()`); the kernel runs only in the spawned
  scheduler, which the driver times over IPC. A malicious kernel cannot reach the
  clock.
- **Fidelity** is judged by the validator, not by miner-reported summaries. In
  KL mode the driver compares emitted-token distributions from stock and
  candidate launches. In audit mode the validator re-runs the captured stock
  baseline itself on cloned inputs.
- The token count is **driver-known** under the scoring default (`ignore_eos` +
  a set `max_new_tokens` → the count is `len(prompts) * max_new_tokens` a
  priori), never a scheduler-reported field the miner could inflate.

## Collective slots — the wider surface

`kind="collective"` slots span GPUs. Decode at TP/EP scale is **comms-bound**
(the all-reduce / all-to-all is the single largest category, ~32–43% of GPU time,
and it's *latency*-bound). Capturing that lever needs a slot that spans GPUs, so
the kernel is handed the **process group** — a wider capability than the
op/block "fill a tensor" contract.

The four invariants still hold (validator owns the output buffer + the group +
the call site; the reduce is mid-network, upstream of the sampler; gated vs the
trusted fp32 cross-rank result; timing is out-of-process). But the larger surface
makes two things **mandatory, not optional**:

- **Distributed verification.** A collective can't be checked on one GPU.
  `optima.verify_collective.verify_collective` spawns `world_size` ranks, runs
  the miner kernel as the real collective, and compares each rank's output to a
  `torch.distributed` reduce of the **fp32** partials (gloo/CPU for a numeric
  check, nccl/GPU for the real path).
- **The end-to-end gate.** Per-collective error compounds across every layer, so
  the op-level cosine/matched_ratio is necessary-but-not-sufficient — the
  model-level fidelity gate and any configured benchmark gate are required.

### Overlap needs a block that owns both

A "hide the reduce behind the producer GEMM" optimization is *not* a reduce-only slot —
it needs a **block that owns both** the producer and the reduce. This is realized
as **`moe.fused_experts_reduce`**: a `(prepare, forward)` block handed the
process group whose `forward(...)` fills `out` with the *already-reduced* expert
output — the validator does **not** replay a stock all-reduce after it. This
lifts the structural ceiling: the plain `moe.fused_experts` slot can't express
the overlap, because there the reduce is severed onto a separate stock call.

## Graph-safety is part of the contract

Scoring runs with **CUDA graphs ON**. Graphs-off cripples the baseline ~4.5–6.5×,
so a graphs-off speedup is not a valid Optima result — beating
sglang/vLLM/TensorRT graphs-on is the entire point.

- The **op seams** (`silu`, `rmsnorm`) are graph-captured directly.
- A **block/collective** kernel must **declare `graph_safe: true`** in its
  metadata to run *inside* the graph. Otherwise the seam falls back to the
  trusted baseline under capture — an un-capturable kernel cannot wedge the
  graph.

`graph_safe` means: static shapes, no host syncs (`.item()`/`.cpu()` that read
back), no data-dependent Python control flow, writes only the
validator-allocated buffer. A kernel that lies either errors at capture (→
fallback) or is caught by the fidelity gate. See
[Graph-safe kernels](/docs/miners/graph-safety).

The one structurally-eager seam is `attention.decode`'s gather MVP (a per-step
`max_len` host-sync); its graph-safe form is a paged-direct contract (the next
rung).

## Evolution rules

- **Additive only.** `abi_version` gates the bundle format; new `SlotSpec` fields
  are optional; never break a bundle that parsed before (Linux's "don't break
  userspace"). A miner who wrote a bundle against `abi_version =
  "optima-op-abi-v0"` last year still parses today.
- **Seam adapters are version-pinned glue, not the contract.** Each sglang
  chokepoint (`SiluAndMul.forward`, `RadixAttention.forward`,
  `FusedMoE.forward_impl`, …) is a *lower-bell* adapter, re-validated on every
  `PINNED_SGLANG` bump via `optima compat`. Proliferate them freely; they never
  touch the four invariants.
- **Extraction trigger (do NOT do early).** `SlotSpec` is a dataclass today, and
  that is correct while it's small. When the next slot forces *yet another*
  optional field/callable — or the `lambda` soup stops being readable — extract
  `Slot` into an interface (one class per slot, mirroring sglang's
  `AttentionBackend` base class). Not before: premature abstraction is its own way
  projects die. The refactor is local — it touches neither the invariants nor any
  miner's bundle.

## The escape hatch (where exotic submissions go FIRST)

Some optimizations can't be expressed as tensor-in/tensor-out (a backend swap, a source
recompile). Those do **not** get special-cased into the core. They go through the
**framework / rebuild tier**, which is experimental and fenced:

- It runs **only validator-shipped, reviewed patchers** (`rebuild.json`
  `repo_python` steps) — never miner-supplied code (`bundle_python` is rejected).
  Like PyTorch: you submit a patch to core to add a backend; you don't ship
  arbitrary code into the dispatcher.
- It is gated by the **token / task-accuracy** path plus **no-egress isolation**,
  because a patched engine's self-reported numbers aren't trusted (invariant 4).
- When a pattern recurs there *and* can be made to satisfy the four invariants,
  **promote it** to a clean core slot. The escape hatch is a staging area, not a
  permanent home.

## What's just churn (don't)

A broad "robustness PR" or load-time weight-conversion framework is premature
until its trigger fires (a real second contributor, a real exotic submission,
memory becoming the bottleneck). The sglang plugin hook and GPU canaries now
exist because the runtime needed them; they are still implementation churn below
the waist, not new miner-facing contract.
