## Where we are

Phase 1 is complete. Cacheon's top-performing miners are outperforming the vLLM baseline by 35%+. More importantly, we have shown that a market for inference optimization can discover real improvements against a strong baseline faster than any single team could.

With those foundations in place, we are preparing the next stage of the subnet. To make that transition, competition will pause on June 20 while we rebuild for Phase 2.

## Phase 1 - Single-model inference arena

**Status:** complete.

One model, one hardware target, one mechanism. Prove that a containerized inference server can be evaluated on-chain and the best one wins emission.

- Model: `Qwen2.5-72B-Instruct`, hardware: 8xH200/B200, baseline: pinned vLLM
- Miners submitted Docker containers; the harness evaluated against baseline on the same GPU
- Scoring: median end-to-end speedup vs baseline (10 prompts, correctness gate required)
- Competition with leader and runner-up emission split (80/20)
- CPU/GPU split validator: always-on CPU VPS for chain interaction, ephemeral GPU pod for eval

**What it proved:** inference optimization is a search problem that a competitive network can solve faster than any single team.

## Phase 2 - Optimization pipeline

**Status:** In design. Rolling out from July 2026.

Phase 1 was intentionally narrow to prove the mechanism. Phase 2 expands the optimization surface and redesigns the incentive mechanism to move closer to a continuously improving optimization baseline.

**What is changing:**

- **Incentives mechanism**: redesigned reward structure to better align contributors with long-term optimization discovery
- **Miner contributions**: auditable code contributions (patches to vLLM) targeting specific bottlenecks, rather than full opaque containers
- **Evaluation**: expanded surface covering concurrency, multi-turn workloads, and throughput under SLA; automated GPU provisioning removes the manual operator loop
- **Optimization surface**: beyond single-request speed, into a full serving profile across workload types

We will be rolling out continuous updates as things evolve. Follow on [X](https://x.com/cacheon_ai) or join [Discord](https://discord.gg/invite/cacheon) for announcements.

## Phase 3 - Production inference provider

Winning configurations deployed as live endpoints serving real traffic. The subnet stops being just a competition and starts being a product.

- Agent, RAG, and multi-turn workloads with P99 latency targets
- **Product Market Fit:** teams route traffic to Cacheon because nothing else serves this model as efficiently

## Phase 4 - Intelligence layer

Routing intelligence above inference providers. Given a workload, determine the optimal provider, model, hardware, and serving configuration.

The optimization network continuously generates the knowledge that powers this routing layer. See [Vision](/docs/vision) for the long-term thesis.
