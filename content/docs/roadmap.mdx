## Where we are

Phase 1, the Cacheon container arena, is complete: it proved that a market for inference optimization can discover real improvements against a strong baseline faster than any single team. Phase 2, Optima, is the active mechanism. It replaces opaque container submissions with inspectable GPU kernels for typed slots, the referee is validated end to end on real GPUs, and the first submitted kernel has surpassed the pinned sglang baseline.

## Phase 1: single-model container arena (Cacheon)

**Status:** complete.

One model, one hardware target, one mechanism. Miners submitted whole containerized inference servers; the harness evaluated each against a pinned vLLM baseline on the same GPU and scored end-to-end speed under a correctness gate, with a leader and runner-up emission split.

**What it proved:** inference optimization is a search problem that a competitive network can solve faster than any single team. Its limitation is what motivated Phase 2: an opaque container is not portable IP and cannot be inspected, composed, or folded into a library.

## Phase 2: kernel competition (Optima)

**Status:** active, current mechanism.

Miners submit a **bundle** (`manifest.toml` + kernel source) for a typed **slot** inside a fixed, validator-controlled model. The validator swaps the kernel into a pinned sglang engine and scores throughput gated by fidelity.

Validated end to end on real GPUs (single H100 up to gpt-oss-120b; 4×B300 for MiniMax-M3-NVFP4):

- **Typed slots (ten today):** ops such as `activation.silu_and_mul` and `norm.rmsnorm`, fused blocks such as `attention.sdpa`, `attention.decode`, and `moe.fused_experts`, and cross-GPU collectives such as `collective.all_reduce` and `moe.fused_experts_reduce` (the block that owns its trailing all-reduce). See the [Slot catalog](/docs/reference/slots-table) for the full list.
- **The seam** that swaps an untrusted kernel into the spawned sglang scheduler process, with tamper-resistant, out-of-process timing.
- **Scoring under CUDA graphs on** (graphs-off cripples the baseline, so it is never used to score), with noise-robust bracketed measurement that returns no-decision on a box too noisy to reproduce.
- **Fidelity gates:** per-token KL against the stock reference plus GSM8K and MMLU benchmark accuracy.
- **Commit-reveal** anti-copy and **per-slot king-of-the-hill** settlement.

### What is proven, and what is not (honest status)

- **A submitted kernel has beaten sglang.** On the MiniMax-M3-NVFP4 arena (4×B300), fused decode epilogues measured **1.044×–1.074×** through the full scorer at equal audited fidelity, reproduced on independent prompt seeds and once through the on-chain loop. This is the first validated win, not the ceiling: most example kernels are still correctness demos (faithful ones reproduce the model; broken ones are caught by the gate).
- **Host isolation for untrusted miners is not built.** The kernel is Python running in the model process. Today the process boundary is a tripwire (a static policy scan), not a wall; namespaces, no network egress, per-eval CUDA context, and a watchdog are required before opening to untrusted miners.
- **Chain integration is live on testnet, not mainnet.** Commit-reveal, content-addressed bundle fetch, and the validator loop are built, merged, and live-validated on Bittensor testnet (netuid 307, 2026-07-08). Weights stayed dry-run there because that subnet rejects our stake, so a real weight push and mainnet economics need our own subnet. The Ledger is durable JSON, not yet a database.
- **No public leaderboard or monitoring dashboard yet.** It is roadmap.
- **More slots.** MLA and weight-absorbed attention, dense FP8/FP4 GEMM, a graph-safe paged-attention contract, and further comms-overlap blocks are the highest-value targets.

## Phase 3: production inference serving

Winning kernels deployed as a managed service (`optima-serve`) running the `optima-kernels` library on a rented engine, serving real traffic. The subnet stops being just a competition and becomes a product.

- Agent, RAG, and multi-turn workloads with latency targets
- **Product-market fit:** teams route traffic to Optima because nothing else serves this model on this hardware as efficiently

## Phase 4: intelligence layer

Routing intelligence above inference providers. Given a workload, determine the optimal kernel, engine, and hardware configuration. The optimization network continuously generates the knowledge that powers this routing layer. See [Vision](/docs/vision) for the long-term thesis.
