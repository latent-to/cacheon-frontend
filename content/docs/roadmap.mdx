## V1 - Single-model inference arena

**Status:** complete.

One model, one hardware target, one mechanism. Prove that a containerized inference server can be evaluated on-chain and the best one wins emission.

- Model: `Qwen2.5-72B-Instruct`, hardware: see [GPU table](/docs/validators/overview#gpu-requirements), baseline: pinned vLLM
- Miners submit Docker containers; the harness evaluates against baseline on the same GPU
- Scoring: median end-to-end speedup vs baseline (10 prompts, correctness gate required)
- Competition with runner-up sharing and score-scaled emission
- CPU/GPU split validator: always-on CPU VPS for chain interaction, ephemeral GPU pod for eval, state synced via Hippius S3
- No speculative decoding or weight quantization in V1

## V1.5 - Automated eval pipeline

**Status:** planned.

V1 evaluation is manually triggered: the operator checks GPU availability, provisions a pod, SSHs in, and runs the eval. V1.5 closes this loop.

- **Automated GPU provisioning**: CPU host checks for challengers, queries `validator.providers.TIERS` (8-GPU tier configs only), provisions a pod via provider API when suitable hardware is available, and tears it down after eval completes
- **CPU-to-GPU SSH dispatch**: CPU SSHs into the GPU pod to trigger eval instead of the operator doing it manually

V1.5 makes the validator fully hands-off. An operator spins up the CPU host once and the system handles everything from challenger detection through eval and weight setting.

## V2 - Expanded optimization surface

After V1.5 stabilizes, restrictions on what miners can do are relaxed:

- Speculative decoding (draft models, Medusa heads, lookahead)
- Weight quantization (FP8, INT4, AWQ, GPTQ) if they pass a tightened correctness gate
- Concurrency benchmarks: multiple simultaneous requests, throughput under SLA
- Stricter correctness: full logprob KL validation

## V3 - Production inference provider

The subnet stops being a competition and starts being a product:

- Winning servers deployed as live endpoints serving real traffic
- Agent, RAG, and multi-turn workloads with P99 latency SLA targets
- PMF test: teams route traffic to Cacheon because it is the fastest production backend for this model

## V4 - Multi-model expansion

- Multiple arenas: one per important open-source model
- OpenRouter integration for serving real traffic at scale
- Buyback with revenue

## End state

Cacheon becomes the default high-performance inference provider for important open-source models, discovered through open competition and deployed into production infrastructure.
