Four versions. Each one expands what the subnet optimizes. No version ships before the previous one is stable.

## V1 -- Single-model inference arena

**Status:** in progress.

Big model, simple mechanism:

- One model: `Qwen2.5-72B-Instruct`
- One hardware target: 4x H200 (fixed NVLink topology)
- One baseline: pinned vLLM version (tensor-parallel)
- One endpoint: `/v1/chat/completions`
- Model weights: mounted read-only, not baked into images
- Miners submit Docker containers with full inference servers
- Instruct-style prompts: curated templates + PG19 context, block-hash seeded
- Correctness gate: 99%+ greedy token match + logprob verification
- Scoring: `0.5 x TTFT + 0.5 x throughput` (median across 10 prompts, floored at 0)
- No speculative decoding, no weight quantization in V1
- King-of-the-hill, winner-take-all
- Centralized eval on a single validator-operated GPU machine

The point of V1 is to prove the loop: someone submits a containerized server on-chain, the harness evaluates it against vLLM on 4x H200, the best server wins emission. Everything else follows from making this work.

## V2 -- Expanded optimization surface

After V1 stabilizes:

- **Allow speculative decoding** -- draft models, Medusa heads, lookahead become first-class
- **Allow weight quantization** -- FP8, INT4, AWQ, GPTQ, if they pass a tightened correctness gate
- **Concurrency benchmarks** -- multiple simultaneous requests, throughput under SLA
- **Batch throughput scoring** -- requests/sec at a given latency target
- **Stricter correctness** -- full logprob KL validation if API supports it
- **Verifiable eval rollouts** -- publish eval artifacts to Cloudflare R2

V2 is where the optimization surface opens fully. Speculative decoding and quantization require more nuanced correctness checking, which is why they are gated behind a proven V1 pipeline.

## V3 -- Production inference provider

This is where the subnet stops being a competition and starts being a product.

- Winning servers deployed as **live endpoints** serving real traffic
- Agent, RAG, and multi-turn workloads with P99 latency SLA targets
- Teams route traffic to Cacheon because nobody else serves this model this fast
- The product-market fit: **"If you need this model, we are the highest-performance production backend for it."**

V3 is the PMF test. If users say "we switched because your latency makes our agent actually usable" or "we depend on you for this model," the subnet is working. The mechanism (competition, scoring, king-of-the-hill) is invisible to the customer. They see an endpoint that is faster than everything else.

## V4 -- Multi-model expansion

- Multiple arenas: one per important open-source model
- OpenRouter integration for serving real traffic at scale
- Revenue sharing between subnet and winning miners
- Any model worth running fast gets a Cacheon arena

V4 scales the proven V3 pattern across models. Each arena discovers the fastest server for its model, and the winning server goes live as a production endpoint.

## End state

Cacheon becomes the default high-performance inference provider for important open-source models, discovered through open competition, deployed into production infrastructure serving real users.
