There are two rule sets. The **sandbox** governs what code your kernel may contain. The **four slot invariants** govern the boundary the validator injects it at. Both are enforced by the harness, not by convention.

## The sandbox

Submitted kernels are statically scanned before they are loaded (`python -m optima.cli scan`). The scan runs no code; it rejects a bundle whose kernel source contains a banned construct.

| may NOT write | examples | why |
| --- | --- | --- |
| network | `socket`, `urllib`, `requests` | a kernel must not exfiltrate or fetch anything; the eval box is isolated |
| process / filesystem escape | `subprocess`, `multiprocessing`, `os.system`, `ctypes`, `shutil`, `tempfile` | a kernel is math on tensors, not a shell; escaping the process is out of scope |
| dynamic code | `eval`, `exec`, `compile`, `__import__`, `open`, `globals` | code that builds code at runtime defeats static inspection |
| deserializers | `pickle.loads`, `torch.load`, `marshal` | deserializing untrusted bytes is arbitrary code execution |
| introspection escapes | `__globals__`, `__subclasses__`, `__class__` | these reach out of the sandboxed namespace |
| dynamic `getattr` / `setattr` | with a non-literal attribute name | a computed attribute name hides what is being touched |

**Allowed:** `torch`, `triton`, `triton.language`, normal Python, `math`, and `from sglang... import ...` (you may reuse sglang's own helpers).

<Callout type="warn" title="Isolation is not fully built yet">
  The sandbox scan narrows the attack surface, but full host isolation for untrusted miners is not built.
  The VM is the trust boundary today, and the harness still runs kernel launch code in-process. Untrusted
  miners are not yet supported. Do not treat the scan as a complete security boundary.
</Callout>

## The four slot invariants

Every slot the validator injects your kernel into preserves these four invariants. They are the narrow waist that keeps the subnet cheat-resistant.

### 1. The validator owns the boundary

The validator allocates every output tensor (shape, dtype, device, stride) and owns the call site. Your kernel only *fills* a validator-allocated output; it never allocates or returns the result the model consumes. **Why:** if the miner owned the output buffer, the miner could return anything. Because the validator owns it, the kernel is confined to filling in the values.

### 2. Strictly upstream of the sampler

A slot sits above the logprobs and sampler. Its output feeds the residual stream, then the downstream layers, then the sampler, all stock. **Why:** there is no final output to substitute, which defeats the "run gibberish, fetch the real answer from an API" attack. This is the single rule that most shapes how you write a kernel: see [Slots](/docs/miners/slots#the-load-bearing-invariant).

### 3. Gated against high-precision ground truth, not the stock kernel

Correctness is measured against an fp32 or dequantized reference, **never** against the stock kernel. **Why:** a faster, slightly-different kernel can legitimately pass, while a wrong one cannot. Gating against the stock kernel would forbid every genuine improvement. The metric may vary per slot (`allclose`, `matched_ratio`, `cosine`); the reference is always high-precision ground truth.

### 4. No trust in miner-reported numbers

The score never trusts a value the miner's code could have produced. Throughput is timed out-of-process (the driver never imports miner code); fidelity is judged by the validator from emitted tokens and the high-precision comparison, not from an engine's self-reported logprobs when the engine itself was patched. **Why:** any number the miner could influence is a number the miner could forge.

<Callout type="info" title="Graph-safety is part of the contract">
  Scoring runs with CUDA graphs ON. A block or collective kernel must declare `graph_safe: true` in its
  metadata to run inside the graph; otherwise the seam falls back to the trusted baseline under capture,
  and a kernel outside the graph cannot win. See
  [Submission format](/docs/miners/submission-format#metadatajson).
</Callout>
