<Callout title="Inference Engine Requirements" type="info">
  The **inference engine** (harness, `setup.sh`, `python -m inference_engine`) is what the
  **centralized validator’s GPU side** uses to score policies. Miners can run the same stack to test
  locally before submitting. That environment is **Linux with CUDA and enough VRAM**.
</Callout>

## 0. Prerequisites

- **Linux** (Debian/Ubuntu-style, or a GPU VM/pod where `apt-get` is available). For local dev on macOS or Windows, use a remote Linux GPU box or your own Docker/Linux setup; we do not document a first-class Mac path for the harness installer.
- **Python 3.11+**.
- **CUDA-capable GPU** with ≥24 GB VRAM for harness runs (RTX 4090 is enough for 7B inference at moderate context). Production eval runs on H100 80GB.
- A **Bittensor wallet** registered on subnet 14. See the [Bittensor wallet docs](https://docs.bittensor.com/getting-started/wallets).
- A **public Hugging Face repo** with a `policy.py` at the repo root to host your policy code.

## 1. Clone the harness

```bash
git clone https://github.com/latent-to/cacheon.git
cd cacheon
bash inference_engine/setup.sh
```

`setup.sh` installs system deps, the harness, grabs `Qwen2.5-7B-Instruct` weights, and sets up a venv.

Once it finishes, run:

```bash
python -m inference_engine
```

This loads the model, runs `Harness.verify()` (the passthrough-equals-HF correctness check), and prints baseline latency + peak memory on a handful of prompts. If this doesn't work cleanly, stop and fix your environment — nothing else will work until it does.

## 2. Implement `KVCachePolicy`

Create `my_policy.py` in a fresh repo (not inside `cacheon/`). The minimum viable skeleton:

```python
import torch
from inference_engine.policy import KVCachePolicy, CacheConfig, AttentionOutput


class MyPolicy(KVCachePolicy):
    def setup(self, config: CacheConfig) -> None:
        self.k_cache = [[] for _ in range(config.num_layers)]
        self.v_cache = [[] for _ in range(config.num_layers)]

    def write(self, keys, values, layer_idx, positions) -> None:
        self.k_cache[layer_idx].append(keys)
        self.v_cache[layer_idx].append(values)

    def attend(self, query, layer_idx) -> AttentionOutput:
        K = torch.cat(self.k_cache[layer_idx], dim=2)
        V = torch.cat(self.v_cache[layer_idx], dim=2)
        scores = torch.matmul(query, K.transpose(-1, -2)) / (query.shape[-1] ** 0.5)
        weights = torch.softmax(scores, dim=-1)
        output = torch.matmul(weights, V)
        return AttentionOutput(output=output, attention_weights=weights)

    def memory_bytes(self) -> int:
        total = 0
        for k in self.k_cache:
            for t in k:
                total += t.element_size() * t.numel()
        return total * 2  # K + V
```

This is a naive passthrough that mirrors the reference implementation. It will produce a score of roughly 0 — no memory savings, no latency gain — but it's valid and the harness will run it.

See [Example policies](/docs/miners/example-policies) for patterns that actually save memory (INT8 quantization, H2O eviction, TurboQuant fused scoring).

## 3. Test locally

Run the harness against your policy before committing anything on-chain:

```python
from inference_engine import Harness, PassthroughPolicy, sample_prompts
from inference_engine.scoring import score
from my_policy import MyPolicy

harness = Harness("Qwen/Qwen2.5-7B-Instruct")
prompts = sample_prompts(block_hash="0x" + "aa" * 32, n=3)

baseline = harness.run(PassthroughPolicy(), prompts)
miner    = harness.run(MyPolicy(),         prompts)

result = score(baseline, miner)
print(result)
```

Look for:

- `disqualified == False` — you haven't broken output quality
- `score > 0` — you're actually winning at least one axis
- `memory_reduction > 0` or `latency_improvement > 0`

If `disqualified == True`, check `result.disqualify_reason`. KL gate failure (`> 0.1` nats) means your policy is producing substantially different tokens from baseline. NaN/Inf means you have a numerical bug.

## 4. Push to a public Hugging Face repo

Use the HuggingFace CLI or `huggingface_hub` to upload your `policy.py`:

```bash
huggingface-cli upload you/my-kv-policy policy.py policy.py
```

Or from Python:

```python
from huggingface_hub import upload_file

upload_file(
    path_or_fileobj="policy.py",
    path_in_repo="policy.py",
    repo_id="you/my-kv-policy",
)
```

Note the **commit SHA** returned by the upload (or visible in the repo's commit history). You need the exact 40-character hex SHA — the validator pins to it for reproducibility. Branch names and short SHAs are rejected.

## 5. Register on-chain

```python
import json
import bittensor as bt

wallet   = bt.wallet(name="my_wallet", hotkey="my_hotkey")
subtensor = bt.subtensor(network="finney")

commit_data = json.dumps({
    "repo":     "you/my-kv-policy",          # HF repo id
    "revision": "abc123def456...",           # full 40-char hex SHA
})

subtensor.set_reveal_commitment(
    wallet=wallet,
    netuid=14,
    data=commit_data,
    blocks_until_reveal=1,
)
```

<Callout type="warn" title="One commitment per hotkey">
  This hotkey now permanently points to this commit. To submit a new policy, register a new hotkey.
  See [Competition model](/docs/concepts/competition-model#one-commitment-per-hotkey) for why.
</Callout>

## 6. Wait for eval

The validator scans commitments roughly every 360 seconds. Within a few epochs, your policy will be picked up as a challenger, evaluated on the GPU pod, and scored against the current king.

Check your status:

- **Leaderboard API** (when live): `GET /scores/{your_hotkey}`
- **On-chain**: your weight will show `1.0` if you're the king, `0.0` otherwise. Intermediate states are transient between rounds.

## Common pitfalls

- **Forgot RoPE.** Qwen applies rotary position embeddings to Q/K _before_ your `write()` is called. If your policy internally rotates them again, attention breaks. The harness delivers post-RoPE tensors — just store them.
- **GQA shapes.** `Qwen2.5-7B` has `num_kv_heads=4` (not 28). K and V come in with `num_kv_heads` on dim 1, Q with `num_heads=28`. Don't assume they match.
- **`seq_len=1` on decode.** During decode, `write()` receives a single new token's K/V. Your cache needs to grow, not replace. `attend()` is called with `q_len=1` and needs to attend over the full accumulated cache.
- **Self-reported memory is not the score.** `memory_bytes()` is informational; the actual memory score uses `torch.cuda.max_memory_allocated()`. A large discrepancy flags the submission.
- **No network, no filesystem, no `os`.** Your policy runs inside a sandbox. See [Allowed operations](/docs/miners/allowed-operations).
- **Don't copy the king.** The validator computes a SHA-256 hash of your `policy.py` and rejects byte-identical copies of the current king's source with `duplicate_of_king`. Even a re-upload of the exact same file won't score — you need a genuinely different policy. See [Competition model](/docs/concepts/competition-model) for how the king-of-the-hill rules work.
