## Required endpoints

### `GET /health`

Returns `HTTP 200` when the server is ready to serve requests. The validator polls this endpoint after starting your container. If it does not return `200` within the 10-minute startup timeout window, the container is terminated and the submission is disqualified.

```json
{ "status": "ok" }
```

### `POST /v1/chat/completions`

The validator sends speed evaluation and correctness evaluation prompts here during evaluation (`max_tokens` is **512** for Pass 1's speed evaluation, **256** for Pass 2's correctness evaluation).

**Request format:**

```json
{
  "model": "Qwen2.5-72B-Instruct",
  "messages": [{ "role": "user", "content": "<instruction with long-context passage>" }],
  "max_tokens": 512,
  "temperature": 0,
  "stream": true,
  "logprobs": true,
  "top_logprobs": 5
}
```

**Required capabilities:**

| Parameter      | Requirement                                                                                                  |
| -------------- | ------------------------------------------------------------------------------------------------------------ |
| `temperature`  | Must support `0` (greedy decoding)                                                                           |
| `stream`       | Must support `true` (SSE streaming). Production scoring only uses `stream=true`; `stream=false` is optional. |
| `logprobs`     | Must return top-N log probabilities per output token when requested                                          |
| `top_logprobs` | Must support values up to 5                                                                                  |
| `max_tokens`   | Must respect the requested generation length                                                                 |

**Response format (streaming):**

Standard _Server-Sent Event_ (_SSE_) with `data: {"choices": [{"delta": {"content": "..."}, "logprobs": {...}}]}` chunks. Final chunk is `data: [DONE]`.

## Two-pass evaluation

The validator runs your server in one container session with two prompt categories, then scores audit outputs on a separate vLLM:

**Pass 1 (speed evaluation):** 12 prompts (2 warmup discarded, 10 scored), long context, `max_tokens=512`. Measures TTFT and aligned throughput vs the cached generation baseline. Streamed top-5 logprobs are collected but are not a DQ gate.

**Pass 2 (correctness evaluation):** 8 prompts, ~4k context, `max_tokens=256`, no warmup. Correctness is checked later via teacher-forcing on a scoring vLLM (`prompt_logprobs`), not from your streamed logprobs.

The validator measures:

- **TTFT**: wall-clock time from request send to first SSE data chunk (Pass 1 speed evaluation)
- **Throughput**: aligned TPS vs cached baseline (Pass 1 speed evaluation)
- **Correctness**: Pass 2 teacher-forcing on audit outputs. See [Scoring](/docs/evaluation/scoring).

## Model identity

Weights are mounted read-only at `/models`. Serve them as-is in BF16.
