---
title: Policy Registration
description: How to register as a miner on the subnet, push your policy to Hugging Face, and commit it on-chain.
---

Once your `policy.py` is ready (tested or not), there are three steps to enter the competition.

## 1. Register on the subnet

Register a hotkey on subnet 14 (mainnet) or 460 (testnet). Skip if already registered.

```bash
# mainnet
btcli subnet register --netuid 14 --wallet-name my-miner --wallet-hotkey default

# testnet
btcli subnet register --network test --netuid 460 --wallet-name my-miner --wallet-hotkey default
```

Registration costs a small recycle fee (fractions of TAO). Confirm with `btcli subnet metagraph --netuid 14` — your hotkey should appear with a UID.

## 2. Upload and commit

The `miner/commit.py` script handles everything in one command: creates the HF repo, uploads your `policy.py`, and commits the repo + pinned revision SHA on-chain. Make sure your `HF_TOKEN` has write access to upload.

```bash
export HF_TOKEN=hf_...

python miner/commit.py \
  --policy-file policy.py \
  --repo you/my-kv-policy \
  --wallet-name my-miner --wallet-hotkey default \
  --network test --netuid 460
```

The script checks that your hotkey is registered and has no existing commitment before doing anything. After committing, it verifies the commitment is visible on-chain.

You can also do a dry run first:

```bash
python miner/commit.py \
  --policy-file policy.py \
  --repo you/my-kv-policy \
  --wallet-name my-miner --wallet-hotkey default \
  --network test --netuid 460 \
  --dry-run
```

For mainnet, use `--network finney --netuid 14`.

<Callout type="info" title="Manual alternative">
  If you prefer to do this yourself: upload `policy.py` to a public HF repo, note the full 40-char commit SHA, then call `subtensor.set_reveal_commitment()` with `{"repo": "you/my-kv-policy", "revision": "<sha>"}`. Branch names and short SHAs are rejected.
</Callout>

<Callout type="warn" title="One commitment per hotkey">
  A hotkey permanently points to its committed revision. To submit a new policy, register a new
  hotkey. See [Competition model](/docs/concepts/competition-model#one-commitment-per-hotkey) for
  why.
</Callout>

## 3. Wait for eval

The validator scans commitments roughly every 6 minutes. Your policy will be picked up as a challenger, evaluated on the GPU pod, and scored against the current king.

### What happens during eval

1. Validator fetches your `policy.py` from Hugging Face at the pinned revision
2. Sandbox validates your code (AST checks for blocked imports/calls)
3. Harness loads your class and runs inference on the eval prompts
4. Scores memory reduction, latency improvement, KL divergence
5. If you beat the king (with the moat threshold), you become the new king

## Updating your policy

You cannot update a commitment on an existing hotkey. To submit a new version:

1. Register a new hotkey on the subnet
2. Push the updated `policy.py` to a new (or existing) HF repo
3. Run `miner/commit.py` with the new hotkey and repo

The old hotkey's commitment remains but will be superseded if the new one scores higher.
