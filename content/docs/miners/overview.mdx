This page is the plain-language on-ramp. No prior subnet knowledge assumed. If
you can write a Triton kernel, you can compete here.

## What Optima is, in one paragraph

Optima is a competition where you earn by making LLM inference faster. You pick
**one slot** in a fixed model — an activation, a normalization, an attention
block, a fused-experts MoE, a cross-GPU collective — and you write a kernel for
it. A validator swaps your kernel into the model it controls, runs it, and scores
you on **throughput** — but only if your kernel passes the **fidelity** gate and,
in production scoring, the arena's task-accuracy gate. If your kernel is
genuinely faster than what the engine ships, at equal quality, you earn.

## How you earn

You submit a kernel as a **bundle** (a directory with a `manifest.toml` and your
kernel source). You commit it on-chain via a timelock commit-reveal (so nobody
can copy you and claim priority). The validator evaluates it out-of-process and
records a score. **King-of-the-hill per slot** (the current best kernel per slot
holds the title; a challenger takes it only by beating it by a margin). Emission
is split across slots, so a specialist who owns one slot earns — not just the
single best end-to-end bundle.

Your score is your throughput speedup vs the stock baseline, **but only if every
gate passed**. If any gate failed, your score is `0.0`.

## The honest state of play

- **The mechanism is proven.** Real kernels have been evaluated end-to-end on
  real GPUs, the gates demonstrably catch cheats (a broken silu scores 0; a
  faithful one passes), and commit-reveal + copy detection work.
- **A submitted optimization has surpassed sglang through the referee.** On
  2026-07-07, fused epilogue kernels for MiniMax-M3-NVFP4/4×B300 measured 1.044×
  and 1.074× vs the pinned sglang baseline at equal audited fidelity, reproduced
  across independent prompt seeds and once through the on-chain loop.
- **Every *other* example bundle in the repo is a correctness demo** — faithful
  but slower than sglang's tuned kernels. That's expected: the vendor already
  tuned the obvious kernels. A real throughput improvement is narrow and
  model-specific, often a few percent end-to-end rather than 2× across the board.

The realistic shape of an improvement: you fuse two ops the vendor shipped
separately, overlap a collective with its producer GEMM, or write a better
epilogue for one quantized MoE variant. You measure before you reason, size the
lever by its WALL-time fraction (not kernel-time sum), and accept that a
faithful-but-slower kernel is the default outcome.

## The five-minute mental model

1. **Pick a slot.** Look at [the slot catalog](/docs/reference/slots-table). You
   target exactly one (or a small related set). The slot is a typed input/output
   contract — e.g. `activation.silu_and_mul` is `entry(x, out)`.
2. **Write the kernel.** Your function **fills `out`** — a tensor the validator
   allocated — and returns nothing. You never return tensors, never allocate the
   output, and never touch the sampler. See [The kernel ABI](/docs/miners/kernel-abi).
3. **Test locally, cheaply.** `optima scan` (static check) and `optima verify`
   (correctness vs an fp32 reference) run on CPU in seconds. Loop here.
4. **Test on a GPU.** `python -m optima.cli evaluate` (throughput + fidelity,
   graphs-on) and `python -m optima.cli bench` when the arena uses a benchmark
   floor. This is where you find out if you're actually faster.
5. **Commit.** Package the bundle with `chain-package`, host the archive, then
   post hash+URL on-chain with `chain-submit` via the timelock commit-reveal.

## The gates, in order

| Gate | What it checks | Cost |
|---|---|---|
| **scan** | static policy scan of your source (banned imports/calls) | seconds, CPU |
| **verify** | op-correctness vs an **fp32 reference** under the slot's tolerance | seconds, CPU/GPU |
| **fidelity** | the output distribution is still correct (in-engine **audit**, or rollout-KL on quiet arenas) | a GPU launch |
| **benchmark** | configured task accuracy, e.g. GSM8K/MMLU/long_math, doesn't regress | a GPU launch |
| **throughput** | `B,C,B'` bookended speedup vs a noise-derived bar | three+ GPU launches |
| **settle** | king-of-the-hill: beat the champion by a margin | off-chain |

`scan` + `verify` are cheap sanity checks. The **load-bearing** anti-cheat is
fidelity + benchmark, on the real model. A faster-but-wrong kernel scores zero
there even if it sailed through `verify`. See
[Why did my kernel fail?](/docs/miners/diagnostics).

## What you give up by competing here

The slot is **upstream of the sampler**. You never produce the final tokens and
you never see them. This is not a limitation you work around — it's the whole
reason the competition can't be cheated (no "run gibberish, fetch the answer from
an API" attack is possible). Your kernel fills one tensor; sglang does the rest.

This also means: you can't earn by changing decoding strategy, batching, or
scheduling. You earn by making one validator-owned tensor-fill faster. That's a
narrow lever, but it is the lever the referee can measure without trusting miner
code.

## The advanced path (read these later, not first)

[Override points](/docs/miners/override-points) and [Dependency patches](/docs/miners/dep-patches)
are for two specific quantized-MoE / deep-fusion slots, not the default way to
submit. Get one plain bundle through `scan` + `verify` + `evaluate` first (see
[Your first kernel](/docs/miners/your-first-kernel)); when something fails, start at
[Why did my kernel fail?](/docs/miners/diagnostics) rather than assuming you need the
advanced path. Both pages carry their own scope warning up top — they're worth
opening only once a normal `entry(..., out)` kernel is not what your slot needs.
