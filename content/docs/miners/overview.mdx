## What you're building

A containerized inference server that serves `Qwen2.5-72B-Instruct` via an OpenAI-compatible chat completions API. The validator pulls your Docker image, starts it with 4x H200 GPU access, sends instruct-style prompts, and measures how fast your server responds compared to a vLLM baseline.

You build a Docker image, push it to a container registry, and commit the image reference on-chain. That's it.

## Prerequisites

| Requirement                 | Details                                                                                                                             |
| --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **Bittensor wallet**        | Registered on subnet 14 (mainnet) or 460 (testnet). See [Wallets in Bittensor](https://docs.bittensor.com/getting-started/wallets). |
| **Container registry**      | Your Docker image must be publicly pullable. Docker Hub, GitHub Container Registry, or any public registry works.                   |
| **GPU (for local testing)** | 4x H200 (or multi-GPU setup with enough VRAM for 72B in BF16). Needed to verify your server works and to benchmark locally.         |
| **Model weights**           | `Qwen2.5-72B-Instruct` from Hugging Face. On the real validator, weights are provided as a read-only volume mount.                  |

<Callout type="info" title="You don't need your own GPU to register">
  Registration is just a commitment on-chain pointing to your Docker image. The validator's GPU pod
  runs your container with model weights mounted. Local testing is strongly recommended but not
  required.
</Callout>

## High-level flow

1. Understand the API contract. See [API contract](/docs/miners/api-contract)
2. Build your inference server and Dockerfile. See [Submission format](/docs/miners/submission-format)
3. (Recommended) Test locally against vLLM. See [Testing locally](/docs/miners/testing-locally)
4. Push your Docker image to a public registry
5. Commit the image reference on-chain. See [Registration](/docs/miners/registration)
6. Validator picks you up, pulls your image, evaluates, scores

<Callout type="info" title="Required steps">
  Steps 4 and 5 are the only required actions. Everything else is preparation.
</Callout>

## Scoring axes (quick summary)

Your server is scored on three axes:

- **TTFT improvement**: how much faster your server produces the first token vs. vLLM (median across 10 prompts, floored at 0)
- **Throughput improvement**: how many more output tokens/sec your server achieves vs. vLLM (median, floored at 0)
- **Correctness**: your outputs must match the baseline (99%+ greedy token match + logprob verification). Fail this and your score is zero.

Full details in [Scoring](/docs/evaluation/scoring).

## V1 constraints

V1 deliberately restricts the optimization surface to keep correctness checking simple:

- **No speculative decoding** -- draft models / lookahead are banned
- **No weight quantization** -- must load exact BF16 weights from the mounted volume
- **No alternate models** -- same model, same weights as the baseline

These restrictions will be relaxed in V2. See [Roadmap](/docs/roadmap).

## Common pitfalls

- **Wrong model weights.** Your server must load the exact `Qwen2.5-72B-Instruct` BF16 weights from the mounted volume at `/models`. Modified, quantized, or distilled weights will fail the correctness gate.
- **Slow startup.** The validator gives your container 10 minutes to respond on `/health`. 72B model loading on 4x H200 takes 2-5 minutes. Budget accordingly.
- **Non-compliant API.** Your server must implement `/v1/chat/completions` with logprobs support. See [API contract](/docs/miners/api-contract).
- **Don't copy the king.** The validator computes a digest of your Docker image. Byte-identical copies of the current king's image are rejected.
