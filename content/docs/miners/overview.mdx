## What you're building

A containerized inference server that serves `Qwen2.5-72B-Instruct` via an OpenAI-compatible chat completions API. The validator pulls your Docker image, starts it with GPU access, sends instruct-style prompts, and measures how fast your server responds compared to a vLLM baseline.

You build a Docker image, push it to a container registry, and commit the image reference on-chain. That's it.

## Prerequisites

| Requirement                           | Details                                                                                                                                                           |
| ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Bittensor wallet**                  | Registered on subnet 14 (mainnet) or 470 (testnet). See [Wallets in Bittensor](https://docs.bittensor.com/getting-started/wallets).                               |
| **Container registry**                | Your Docker image must be publicly pullable. You can use Docker Hub, GitHub Container Registry, or any public registry.                                           |
| **GPU (for local testing)**           | 8-GPU NVLink/SXM pod (e.g. 8x H200 or 8x B200). On-chain eval always runs on 8 GPUs (`TP=8`). See [GPU requirements](/docs/validators/overview#gpu-requirements). |
| **Model weights (for local testing)** | `Qwen2.5-72B-Instruct`. On the real validator, weights are provided as a read-only volume mount.                                                                  |

<Callout type="info" title="You don't need your own GPU to register">
  Registration on mainnet is just an on-chain commitment that points to your Docker image. The
  validator machine runs your container with model weights mounted. You only need a GPU for local
  testing before submission.
</Callout>

## High-level flow

1. Understand the API contract. See [API contract](/docs/miners/api-contract)
2. Build your inference server and Dockerfile. See [Submission format](/docs/miners/submission-format)
3. Test locally before submitting. See [Local testing](/docs/miners/local-testing)
4. Push your Docker image to a public registry
5. Run `miner/commit.py` to pay the submission fee and commit on-chain. See [Registration](/docs/miners/registration)
6. Validator verifies payment, pulls your image, evaluates, scores

## Common pitfalls

- **Wrong model weights.** Your server must load the canonical `Qwen2.5-72B-Instruct` weights from the mounted volume at `/models`. Fine-tuned, distilled, or otherwise altered weights are likely to fail the text-similarity pre-filter or teacher-forcing gate.
- **Slow startup.** The validator gives your container 12.5 minutes to respond on `/health`. 72B model loading takes 3-7 minutes; compile-heavy cold starts can need more. Budget accordingly.
- **Non-compliant API.** Your server must implement `/v1/chat/completions` with logprobs support. See [API contract](/docs/miners/api-contract).
- **Tensor parallelism.** On-chain eval always runs on 8 GPUs (`TP=8`). Your startup script should still detect GPU count via `nvidia-smi` so local smoke tests on smaller machines work. See the example in [Submission format](/docs/miners/submission-format).
- **Don't copy the leader.** The validator computes a digest of your Docker image. Byte-identical copies of the current leader's image are rejected.
