## What you're building

You write a **GPU kernel**: a small piece of code for one operation inside a fixed, public model that the validator runs on its GPUs. Not a container, not an inference server, not a fine-tune. The validator swaps your kernel into a pinned sglang engine, runs the model, and scores two things:

1. **Throughput** — tokens/second, versus the model running stock (unmodified sglang).
2. **Fidelity** — does the model still produce the *same* outputs? Measured by per-token KL divergence against the stock run, plus real-benchmark accuracy (GSM8K / MMLU).

You earn if and only if your kernel is **both faster and faithful**. A kernel that is 30% faster but changes the model's answers scores zero. A kernel that is perfectly faithful but slower than stock also scores zero. The whole game is a genuine speedup at equal quality.

Your submission unit is a **bundle**: a directory holding a `manifest.toml`, your `kernels/*.py`, and optional `metadata/*.json`. See [Submission format](/docs/miners/submission-format).

<Callout type="info" title="Cacheon to Optima, Phase 2">
  This is the successor mechanism to Cacheon. Cacheon paid for a Docker container that served a whole
  model; Optima pays for a single kernel swapped into a model the validator controls. The subnet is
  netuid 14 on mainnet, 470 on testnet.
</Callout>

## What you need

| Requirement | Details |
| --- | --- |
| **Ability to write a GPU kernel** | PyTorch, Triton, or CuteDSL. This is a performance-engineering competition, not a config exercise. |
| **A machine to develop on** | Everything up to submission runs on CPU. You can write, statically scan, and op-verify a kernel with no GPU at all. |
| **A GPU (yours or rented) for real scoring** | Throughput and fidelity are only meaningful on a GPU. See [Local testing](/docs/miners/local-testing) for the toolchain and pod recipe. |
| **A Bittensor hotkey** | Needed only to submit and to be paid. Everything before commit needs no chain and no wallet. See [Registration](/docs/miners/registration). |

## The high-level flow

1. **Write** a kernel for one [slot](/docs/miners/slots). Copy an example bundle and edit it.
2. **Verify locally** on CPU: `python -m optima.cli scan` then `verify`. No GPU, no cost.
3. **GPU-evaluate**: `verify --device cuda`, then `evaluate` (throughput + KL) and `bench` (real-task accuracy). See [Local testing](/docs/miners/local-testing).
4. **Commit-reveal**: commit a hash of your bundle, then reveal it. This defeats off-chain copying. See [Registration](/docs/miners/registration).
5. **Settle**: the validator scores every original, passing reveal and applies king-of-the-hill per slot.

## The honest reality check

As of this writing, **no submitted kernel has beaten sglang.** sglang's built-in kernels are already heavily tuned. Most faithful kernels you write will be *slower* than the baseline, and that is the expected default outcome, not a bug.

Winning is real work. A realistic win is roughly **1.1x to 1.4x on one slot in one regime**, not a 2x blowout, and the strategy is to stack several regime-specific wins rather than find one giant lever. [Finding a win](/docs/miners/finding-a-win) is about where the actual openings are. This is not a faucet; it is a performance-engineering competition against a tuned opponent.
