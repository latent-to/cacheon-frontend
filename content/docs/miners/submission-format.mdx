## The bundle

A submission is a **directory** (a "bundle") with three parts:

```
my_bundle/
  manifest.toml          # what you're submitting (data only, never executed)
  kernels/
    my_kernel.py         # your kernel code
  metadata/
    my_kernel.json        # optional: eligibility (dtypes, GPU arch, graph_safe)
```

The `manifest.toml` is data and is never executed. Your `kernels/` hold the code. `metadata/` is optional but recommended. All paths in the manifest are relative and must stay inside the bundle.

## manifest.toml

```toml
bundle_id   = "my-silu-v1"              # unique id, [0-9A-Za-z._-]
abi_version = "optima-op-abi-v0"        # must be exactly this

[[ops]]                                 # one [[ops]] block per slot you target
slot   = "activation.silu_and_mul"      # the slot id (from `optima slots`)
source = "kernels/my_kernel.py"         # relative path to your module
entry  = "silu_and_mul"                 # the function name inside it
dtypes = ["bfloat16", "float16"]        # optional eligibility filter
metadata = "metadata/my_kernel.json"    # optional
# prepare = "prepare"                    # for (prepare, forward) slots like MoE
# setup   = "setup"                      # optional one-time engine setup (advanced)
# architectures = ["sm90", "sm100"]      # optional GPU-arch gate (sm90=H100, sm100=B200)
```

One `[[ops]]` block per slot. A single bundle can target several slots at once; add more `[[ops]]` blocks.

| field | required | meaning |
| --- | --- | --- |
| `bundle_id` | yes | unique id, characters `[0-9A-Za-z._-]` |
| `abi_version` | yes | must be exactly `optima-op-abi-v0` |
| `slot` | yes | the slot id, from `python -m optima.cli slots` |
| `source` | yes | relative path to your kernel module inside the bundle |
| `entry` | yes | the function name inside `source` (the per-step forward for MoE) |
| `dtypes` | no | eligibility filter; outside it the validator runs the baseline |
| `metadata` | no | relative path to the metadata JSON |
| `prepare` | no | function name for the once-at-load pass on `(prepare, forward)` slots |
| `setup` | no | one-time engine setup hook (advanced; see the escape-hatch note in [Rules](/docs/miners/rules)) |
| `architectures` | no | GPU-arch gate, e.g. `["sm90", "sm100"]` |
| `base_kernel` / `override_point` | no | compose your code into a validator-owned base kernel at a named override point (advanced) |

## The kernel contract

Your entry function **writes into a pre-allocated output tensor and returns `None`.** The validator owns and allocates `out`; you only fill it. You never return tensors and you never allocate the output. That is the anti-cheat boundary: see [Slots](/docs/miners/slots#the-load-bearing-invariant).

```python
import torch
import torch.nn.functional as F

def silu_and_mul(x: torch.Tensor, out: torch.Tensor) -> None:
    d = x.shape[-1] // 2
    out.copy_(F.silu(x[..., :d].float()).to(x.dtype) * x[..., d:])
```

## (prepare, forward) slots

The MoE slots (`moe.fused_experts`, `moe.fused_experts_reduce`) expose **two** functions. `prepare` runs **once** at load on the raw checkpoint weights: do your weight relayout or quantization-packing here, and the validator caches the result. `forward` (the `entry`) runs every step and receives that cached object as `prepared`.

```python
def prepare(w13, w2):                  # once, at load
    return {"w13": relayout(w13), "w2": w2.contiguous()}

def fused_experts(x, topk_ids, topk_weights, prepared, out):   # every step
    out.copy_(run_experts(x, topk_ids, topk_weights, prepared))
```

The collective variant `moe.fused_experts_reduce` additionally receives the process `group` and must fill `out` with the *already-reduced* expert output, so it can fuse or overlap the expert GEMM with the trailing all-reduce. The validator does **not** replay a stock all-reduce after it.

## metadata/*.json

Optional but recommended. It declares eligibility and graph-safety.

```json
{
  "op": "activation.silu_and_mul",
  "dtypes": ["bfloat16", "float16"],
  "architectures": ["sm90", "sm100"],
  "graph_safe": true,
  "notes": "free text"
}
```

- `dtypes` / `architectures` — eligibility filters. Outside them the validator runs the baseline instead of your kernel, so claim only what you actually support.
- `graph_safe: true` — **required for a block or collective kernel to be scored under CUDA graphs.** Scoring runs graphs-ON (graphs-off cripples the baseline about 5x, so it is never used). Without this flag your block kernel falls back to the baseline in-graph and cannot win. Only declare it if your kernel truly captures: static shapes, no host syncs (`.item()`/`.cpu()`), no data-dependent Python control flow, and it writes only the validator-allocated buffer. A kernel that lies either errors at capture (falls back) or is caught by the fidelity gate.

## The sandbox

Submitted kernels are statically scanned before they are loaded (`python -m optima.cli scan`). If the scan flags you, this is why.

**Banned:**

- network — `socket`, `urllib`, `requests`, and similar
- process / filesystem escape — `subprocess`, `multiprocessing`, `os.system`, `ctypes`, `shutil`, `tempfile`, and similar
- dynamic code — `eval`, `exec`, `compile`, `__import__`, `open`, `globals`
- deserializers — `pickle.loads`, `torch.load`, `marshal`, and similar
- introspection escapes — `__globals__`, `__subclasses__`, `__class__`, and similar
- dynamic `getattr` / `setattr` with a non-literal name

**Allowed:**

- `torch`, `triton`, `triton.language`
- normal Python and `math`
- `from sglang... import ...` — you may reuse sglang's own helpers
