## Docker image requirements

| Requirement         | Value                                        |
| ------------------- | -------------------------------------------- |
| **Max image size**  | 20 GB (weights are NOT in the image)         |
| **Base image**      | Any (Ubuntu, Alpine, scratch, etc.)          |
| **GPU runtime**     | Must work with NVIDIA Container Toolkit      |
| **Exposed port**    | Your server must listen on port `8000`       |
| **Health endpoint** | `GET /health` must return `200` when ready   |
| **Startup timeout** | 12.5 minutes from container start to healthy |

## Model weights: mounted, not baked

Model weights are provided as a **read-only volume** at `/models`. The validator mounts the Hugging Face snapshot:

```bash
docker run --gpus all \
  -v /workspace/models/Qwen2.5-72B-Instruct:/models:ro \
  --network cacheon-internal \
  --shm-size 16g \
  <your-image>
```

Your server must load weights from `/models`. Do not bake 72B weights into your Docker image; at 140GB+, the image would be impractical to distribute and pull.

## Example Dockerfile

<Callout type="info" title="Validator eval uses 8 GPUs (TP=8)">
  On-chain evaluation always runs on 8-GPU pods (`tensor parallelism=8`). Your startup script should
  still detect available GPUs with `nvidia-smi` so that local smoke tests can run unchanged on
  smaller machines.
</Callout>

A minimal example using vLLM as a starting point:

`start.sh`:

```bash
#!/bin/bash
set -e
GPU_COUNT=$(nvidia-smi --query-gpu=name --format=csv,noheader 2>/dev/null | wc -l)
GPU_COUNT=${GPU_COUNT:-1}
[ "$GPU_COUNT" -lt 1 ] && GPU_COUNT=1
exec python3 -m vllm.entrypoints.openai.api_server \
  --model /models \
  --served-model-name Qwen2.5-72B-Instruct \
  --generation-config vllm \
  --host 0.0.0.0 \
  --port 8000 \
  --dtype bfloat16 \
  --tensor-parallel-size "$GPU_COUNT"
```

<Callout type="info" title="Use python3 inside the container">
  The `vllm/vllm-openai` image exposes `python3` on `PATH`; `python` may be missing. Use `exec
  python3 -m vllm.entrypoints.openai.api_server` in `start.sh`, not `python`.
</Callout>

`Dockerfile`:

```dockerfile
FROM vllm/vllm-openai:latest

EXPOSE 8000

COPY start.sh /start.sh
RUN chmod +x /start.sh
ENTRYPOINT ["/start.sh"]
```

<Callout type="info" title="This is the baseline">
  Submitting stock vLLM will get you a score of approximately zero (no improvement over baseline).
  This example exists so you can verify the contract works. Your job is to beat it.
</Callout>

## Registry requirements

Your image must be publicly pullable without authentication. Supported registries:

- Docker Hub (`docker.io/YOUR_USER/cacheon-miner:v1`)
- GitHub Container Registry (`ghcr.io/YOUR_ORG/cacheon-miner:v1`)
- Any registry accessible via `docker pull`

## On-chain commitment format

The commitment you post on-chain is a JSON object:

```json
{
  "image": "docker.io/YOUR_USER/cacheon-miner:v1",
  "digest": "sha256:abc123...",
  "payment_tx": "0xabc123...",
  "payment_block": "0xdef456..."
}
```

| Field           | Required | Description                                                 |
| --------------- | -------- | ----------------------------------------------------------- |
| `image`         | Yes      | Full Docker image reference (registry/repo:tag)             |
| `digest`        | Yes      | Image manifest digest (`sha256:...`)                        |
| `payment_tx`    | Yes\*    | Extrinsic hash of the fee batch when the payment gate is on |
| `payment_block` | Yes\*    | Block hash where that extrinsic was included                |

\*Required when validators set `CACHEON_SUBMISSION_FEE_RAO` `> 0`. Use [`miner/commit.py`](https://github.com/latent-to/cacheon/blob/main/miner/commit.py) to pay and commit in one flow.

The `digest` pins the exact image content. Tag mutations do not affect evaluation. The remark in the fee batch is `cacheon:<hotkey>:<digest>`, binding payment to that digest.
