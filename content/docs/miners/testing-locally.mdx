---
title: Testing locally
description: How to test your inference server locally before committing on-chain. Run the same evaluation the validator runs.
---

Local testing lets you verify your server is compliant and benchmark it against the vLLM baseline before spending a registration fee.

## Prerequisites

- Docker with NVIDIA Container Toolkit (`--gpus` support)
- 4x H200 or equivalent multi-GPU setup (enough VRAM for 72B BF16)
- Model weights downloaded to a local directory
- The vLLM baseline image (for comparison)

## Step 0: Download model weights

```bash
huggingface-cli download Qwen/Qwen2.5-72B-Instruct --local-dir /workspace/models/Qwen2.5-72B-Instruct
```

These weights will be mounted into both baseline and miner containers.

## Step 1: Start the vLLM baseline

```bash
docker run --gpus '"device=0,1,2,3"' -p 8000:8000 \
  -v /workspace/models/Qwen2.5-72B-Instruct:/models:ro \
  vllm/vllm-openai:latest \
  --model /models --host 0.0.0.0 --port 8000 \
  --dtype bfloat16 --tensor-parallel-size 4
```

Wait for `GET http://localhost:8000/health` to return 200.

## Step 2: Run the benchmark prompts

Send the same instruct-style prompts the validator uses:

```bash
python scripts/local_benchmark.py --target http://localhost:8000 --prompts hybrid
```

This records baseline TTFT and throughput numbers (median across 10 prompts).

## Step 3: Start your server

```bash
docker run --gpus '"device=0,1,2,3"' -p 8001:8000 \
  -v /workspace/models/Qwen2.5-72B-Instruct:/models:ro \
  your-image:latest
```

Wait for `GET http://localhost:8001/health` to return 200.

## Step 4: Benchmark your server

```bash
python scripts/local_benchmark.py --target http://localhost:8001 --prompts hybrid --baseline-results baseline.json
```

This compares your server against the baseline and prints:

```
TTFT improvement:       +23.4%  (median)
Throughput improvement:  +31.2%  (median)
Token match rate:        99.6%  (threshold: 99%)
Correctness:             PASS
Final score:             0.273
```

## Step 5: Check correctness

The correctness check is the most important part. If your server fails the 99% token match threshold, it will be DQ'd on the real validator regardless of speed.

Common causes of correctness failure:

- **Wrong model weights.** Use the exact same weights directory mounted into both containers.
- **Weight quantization.** Not allowed in V1. All servers must load BF16 weights.
- **Modified sampling.** Ensure `temperature=0` produces deterministic greedy output.
- **Speculative decoding.** Not allowed in V1.

## Interpreting results

| Metric                 | Good  | Needs work                      |
| ---------------------- | ----- | ------------------------------- |
| TTFT improvement       | > 10% | 0% (you are no faster than vLLM) |
| Throughput improvement | > 10% | 0%                              |
| Token match rate       | > 99.5% | < 99% (will be DQ'd)          |

The local benchmark uses the same scoring formula as the validator (median aggregation, `max(0, ...)` floor). If your score is positive locally, it should be positive on the real evaluation (modulo hardware differences between your GPUs and the validator's 4x H200).
