title: Local testing
description: The inner loop. Scan and verify on CPU with no GPU, then evaluate and bench on a GPU against the real throughput, KL, and accuracy gates.
---

The loop has two halves. The **CPU half** costs nothing and catches most mistakes: a static policy scan and op-correctness against a reference. The **GPU half** is where throughput and fidelity become real. Start on CPU, and only move to a GPU once `verify` passes.

## CPU: scan and verify (no GPU)

Install the CLI, then run the two local gates against an example bundle.

```bash
git clone https://github.com/latent-to/optima && cd optima
pip install -e .            # installs the `optima` CLI + test deps
pytest tests/               # sanity: should be all green
```

```bash
# static policy scan (no code execution) — the sandbox check
python -m optima.cli scan   examples/miner_silu_torch

# op-correctness vs the high-precision reference, on CPU
python -m optima.cli verify examples/miner_silu_torch --device cpu --dtype float32
```

`scan` is the sandbox gate from [Submission format](/docs/miners/submission-format#the-sandbox): it flags banned constructs without running your code. `verify` feeds your kernel deterministic inputs and compares the output to a high-precision reference. This is a sanity check ("is this even the right function?"), not the load-bearing anti-cheat.

Now make it yours: copy the bundle, edit the kernel, re-verify.

```bash
cp -r examples/miner_silu_torch my_silu && $EDITOR my_silu/kernels/silu_and_mul.py
python -m optima.cli verify my_silu --device cpu --dtype float32
```

Compare against the broken examples to see what a *failing* kernel looks like. They exist on purpose.

```bash
python -m optima.cli verify examples/miner_silu_broken --device cpu --dtype float32   # fails op-correctness
```

That is the whole inner loop: edit, `verify`, repeat. No GPU, no cost.

<Callout type="info" title="Collective slots verify differently">
  A collective kernel cannot be checked on one GPU. `verify` refuses `kind="collective"`; the validator
  uses a distributed verifier that spawns `world_size` ranks and compares each rank's output to a
  reference cross-rank reduce of the fp32 partials. Op and block slots verify with the command above.
</Callout>

## GPU: evaluate and bench (the real scoring)

When local `verify` passes, measure on a GPU (your own or rented). Read [DEV_ENVIRONMENT.md in the repo](https://github.com/latent-to/optima/blob/main/docs/DEV_ENVIRONMENT.md) for the pod recipe. {/* TODO: confirm final optima repo home (currently ShivanshuPurohit/optima) */}

```bash
# 1) op-correctness on real shapes/dtypes
python -m optima.cli verify my_bundle --device cuda --dtype bfloat16

# 2) end-to-end throughput + KL smoke (the real gate), on a small model first
python -m optima.cli evaluate my_bundle --model Qwen/Qwen2.5-1.5B-Instruct \
    --num-prompts 64 --max-new-tokens 64

# 3) capability floor on a real task (throughput + KL + accuracy)
python -m optima.cli bench my_bundle --model Qwen/Qwen2.5-1.5B-Instruct \
    --benchmarks gsm8k --samples 128
```

`evaluate` gives you the **two-launch throughput** and a **KL smoke** in one run: it brackets your candidate between baseline runs and reports a speedup against the noise-derived bar. `bench` runs a real task (GSM8K / MMLU) and reports throughput, KL, and **accuracy**, which is the capability floor.

Read the `evaluate` output the way the validator does: a **speedup at or above the noise-derived bar**, **KL under the per-slot threshold**, and **no accuracy regression**. If the two bracketing baselines disagree a lot, the box is too noisy; fix that before trusting any number (lock clocks if you can, warm up, or run two replicas on disjoint GPUs so drift cancels).

## Environment notes

<Callout type="warn" title="Always launch via python -m optima.cli">
  sglang spawns its scheduler with `mp spawn`, and the `__main__` guard matters. Run the GPU commands
  through `python -m optima.cli ...`, not by importing the harness yourself.
</Callout>

- **`TORCH_CUDA_ARCH_LIST`** — set it to your GPU arch: `9.0` = H100, `10.0` = B200.
- **Triton JIT** — sglang JIT-compiles CUDA at runtime, so the box needs `nvcc` and `ninja` on `PATH` (`export CUDA_HOME=/usr/local/cuda`).
- **The seam** — the harness installs the kernel-injection seam into the venv via a `.pth`, because sglang runs the model in a spawned child. Run everything inside the same venv where you `pip install -e .`.

## Why a kernel fails

| symptom | likely cause | fix |
| --- | --- | --- |
| `scan` reports a banned construct | network / file / dynamic-code call in your kernel | remove it; reuse sglang helpers via `import`, not `open()`/`exec()` |
| `verify` op-correctness fails | wrong math, dtype handling, or shape vs the slot contract | check the exact signature in `optima slots`; compare to the matching example bundle |
| `evaluate` runs but the kernel never fires | dtype/arch eligibility excludes the run, or a block kernel is not `graph_safe` so it fell back | widen `metadata`, declare `graph_safe` |
| KL gate fails on a *faithful* kernel | you did not measure the stock-vs-stock noise floor; the "drift" may be the model's own nondeterminism | measure the floor first; for big MoE use deterministic mode and let the accuracy gate carry quality |
| speedup gate fails | your kernel is simply slower than sglang's (the common case) | profile; see [Finding a win](/docs/miners/finding-a-win) |
| score is 0 despite a speedup | a fidelity gate failed (fast but dumb), or it tied the champion and did not clear the margin | check the quality line in the report |
| NO-DECISION | the box was too noisy (baselines disagreed) | quieten the box; re-run |
