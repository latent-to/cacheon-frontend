title: Local testing
description: "Verify your Docker image works correctly before submitting on-chain. Two paths: a quick no-GPU check to catch startup bugs, and a full GPU dry-run against real weights."
---

You have built and tagged your image. Before committing it on-chain, run through any of these checks. They are ordered by cost:

- the no-GPU check takes seconds and catches most submission mistakes
- the GPU dry-run takes 10-20 minutes and confirms the full eval flow

## Path 1: No-GPU smoke test

**What it catches:** broken entrypoints, bad argument parsing, attempts to reach the internet at startup, completely wrong base images. Runs on any machine with Docker installed and does not require a GPU or model weights.

**What it does not catch:** model loading errors, CUDA issues, tensor parallel mismatches, correctness problems. Catching those errors requires a [full GPU dry run](#path-2-full-gpu-dry-run).

### Run it

```bash
docker run --rm -it \
  -v /dev/null:/models:ro \
  -p 8000:8000 \
  <your-image>
```

`/dev/null` is mounted at `/models` so the container does not return an error on a missing directory. The server will start, attempt to load a model, and then fail. This behavior is expected. Focus on what happens before the model loads.

<Callout type="info" title="What you are looking for">
**Pass:** container prints startup logs, vLLM or your server begins initializing, then complains about missing weights or CUDA.

**Fail:** container exits in under a second. Common patterns:

- `error: unrecognized arguments: python -m vllm.entrypoints.openai.api_server` means your ENTRYPOINT and CMD are conflicting. See [Submission format](/docs/miners/submission-format#example-dockerfile).

- `exec: python: not found` means your script calls `python` but the base image only provides `python3`. Use `exec python3 -m vllm.entrypoints.openai.api_server ...` in your startup script.

Any immediate exit with no output usually means your ENTRYPOINT path is wrong or the script is not executable.

</Callout>

<Callout type="warn" title="Image size on Docker Desktop">
  If you are building from `vllm/vllm-openai:latest`, the base image is ~9 GB. On Docker Desktop (Mac/Windows) you may need to increase disk space under Settings > Resources > Disk image size. A Linux machine with native Docker is easier for this.
</Callout>

## Path 2: Full GPU dry-run

**What it catches:** everything above, plus model loading errors, tensor parallel mismatches, `/health` not coming up, logprobs not being returned, model name mismatches, startup timeout.

**Prerequisites**

- a GPU machine (any Tier A or B hardware; see [GPU requirements](/docs/validators/overview#gpu-requirements)) with Docker
- NVIDIA Container Toolkit, and the model weights installed.

### Step 1: Provision the machine

On a fresh Ubuntu 22.04/24.04 GPU machine, run the [validator setup script](https://github.com/latent-to/cacheon/blob/main/validator/setup.sh). This installs Docker, NVIDIA Container Toolkit, downloads `Qwen2.5-72B-Instruct` weights to `/workspace/models/Qwen2.5-72B-Instruct`, and verifies the GPU is accessible from inside containers.

```bash
export HF_TOKEN=...  # optional, speeds up the model download
sudo apt-get install -y curl vim git
curl -fsSL "https://raw.githubusercontent.com/latent-to/cacheon/main/validator/setup.sh" | sudo -E bash
```

If Docker and NVIDIA Container Toolkit are already installed and you only need the weights, you can skip the script and download the model weights directly:

```bash
pip install huggingface_hub
hf download Qwen/Qwen2.5-72B-Instruct \
  --local-dir /workspace/models/Qwen2.5-72B-Instruct
```

### Step 2: Create an air-gapped test network

The on-chain validator runs your container on an internal Docker network with no outbound internet. You must replicate the same constraint in your local tests.

To do this, run the following command:

```bash
docker network create --internal cacheon-test-internal
```

### Step 3: Start your container

Start the Docker container using the following command. If your weights are at a different path, replace `/workspace/models/Qwen2.5-72B-Instruct` accordingly.

```bash
docker run -d \
  --name cacheon-local-test \
  --gpus all \
  --network cacheon-test-internal \
  -v /workspace/models/Qwen2.5-72B-Instruct:/models:ro \
  --shm-size 16g \
  <YOUR-IMAGE>
```

<Callout type="warn" title="Do not use the --rm flag">
  If the container crashes on startup, the `--rm` flag removes the container before you can read
  logs. Omit it so you can always run `docker logs cacheon-local-test` to see what went wrong.
</Callout>

### Step 4: Watch startup logs

```bash
docker logs -f cacheon-local-test
```

Wait until your server prints that it is ready (e.g. `Uvicorn running on http://0.0.0.0:8000`). On a single B200 or 4x H100 this typically takes 3-7 minutes.

### Step 5: Test `/health`

Get the container IP (it is on an internal network, not reachable via `localhost`):

```bash
CONTAINER_IP=$(docker inspect -f \
  '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' \
  cacheon-local-test)
echo "Container IP: $CONTAINER_IP"
```

Ping `/health` from another terminal on the same network:

```bash
docker run --rm --network cacheon-test-internal curlimages/curl:latest \
  curl -sf http://$CONTAINER_IP:8000/health
```

Expected result: `HTTP 200` from `docker logs -f cacheon-local-test`.

### Step 6: Test completions with logprobs

```bash
docker run --rm --network cacheon-test-internal curlimages/curl:latest \
  curl -s http://$CONTAINER_IP:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen2.5-72B-Instruct",
    "messages": [{"role": "user", "content": "What is 1 + 1?"}],
    "max_tokens": 32,
    "temperature": 0,
    "stream": true,
    "logprobs": true,
    "top_logprobs": 5
  }'
```

Check the response:

- You get `data: {...}` SSE chunks
- Each chunk has `logprobs.content[].top_logprobs` with 5 entries
- The stream ends with `data: [DONE]`
- The model field in the request (`Qwen2.5-72B-Instruct`) is not rejected

If the model name is rejected, register it in your container startup. For vLLM, pass `--served-model-name Qwen2.5-72B-Instruct` alongside your other flags in the `start.sh` snippet under [Example Dockerfile](/docs/miners/submission-format) on the Submission format page (same block as `--model /models`, `--dtype`, and `--tensor-parallel-size`).

### Step 7: Clean up

Finally, clean up the state of your Docker container:

```bash
docker stop cacheon-local-test
docker network rm cacheon-test-internal
```

## Pre-submission checklist

Before pushing your image and committing on-chain:

| Check                                      | How to verify                                                         |
| ------------------------------------------ | --------------------------------------------------------------------- |
| Container does not exit immediately        | Path 1 passes with your startup logs visible                          |
| No internet access during startup          | No HuggingFace download attempts in logs when on `--internal` network |
| `/health` returns 200                      | Step 5 above                                                          |
| `/v1/chat/completions` streams SSE         | Step 6 above -- `data:` chunks present                                |
| Logprobs returned                          | Each chunk has `logprobs.content[].top_logprobs`                      |
| Model name `Qwen2.5-72B-Instruct` accepted | Step 6 request not rejected                                           |
| Startup completes within 10 minutes        | Container healthy before timeout                                      |
| `--tensor-parallel-size` not hardcoded     | Startup script uses `nvidia-smi` to detect GPU count at runtime       |
