## Training is no longer the bottleneck

Most of the public attention on LLMs still goes to training. In production, the constraints that actually determine whether a model is usable are different:

- **Latency** — tokens/sec and time-to-first-token. Users feel the delay.
- **Memory** — KV cache scaling with context length. More context means more memory.
- **Cost** — GPU utilization and serving efficiency. Dollars per million tokens.

These are **inference-time** problems. Cacheon treats inference optimization as a search problem and asks the subnet to measure it in the open.

## The memory wall

Every transformer model stores a key-value cache that grows linearly with sequence length. The math is unforgiving:

```
KV cache memory = 2 × num_layers × num_kv_heads × head_dim × seq_len × dtype_bytes
```

At **100k tokens × 80 layers × `head_dim=128`** in FP16, the KV cache alone is **~80 GB**. That's the size of an entire NVIDIA H100 GPU. This means you run out of memory before you start generating.

This is why longer context, larger batches, and cheaper serving all run into the same wall: the cache.

## The benchmark problem

There's no shortage of research on KV-cache optimization. What's missing is a shared, verifiable benchmark:

| Approach     | What it does                               | Typical reported savings |
| ------------ | ------------------------------------------ | ------------------------ |
| H2O          | Evict tokens by attention importance       | 5–10× memory             |
| SnapKV       | Compress by observation windows            | 4–8×                     |
| KIVI         | 2-bit quantization of keys + values        | 4–6×                     |
| TurboQuant   | Rotated 3-bit quantization + QJL scoring   | 8–10×                    |
| StreamingLLM | Keep first + last sliding window of tokens | 10× at long context      |

Each paper uses its own prompts, its own hardware, its own quality metric, so the **numbers don't compare.** Teams shipping production inference have no way to know which approach actually works for their specific workload.
