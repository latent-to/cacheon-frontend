title: What is the KV cache?
description: How transformers reuse past work during decoding, why the cache grows linearly with context, and the two places a miner can intervene.

## The plain-English version

Language models generate text one token at a time. To pick the next token, the model needs access to everything that came before in the sequence. Recomputing the past from scratch every step would be impossibly slow — so the model **caches** some of that work.

For each layer and each position already processed, the model stores two vectors per attention head:

- the **key** — what the token is "offering" to attend to
- the **value** — what the token contributes if attended to

Those stored vectors are the **KV cache**. It grows by one entry per new token, per layer, per head. More context means more cached rows, which is why memory usage explodes on long prompts and long chats.

## The formulas

When a transformer generates token `N`, every attention layer computes:

```python
Q = input @ W_q       # query for this token
K = input @ W_k       # key for this token
V = input @ W_v       # value for this token

scores   = Q @ K_cache.T       # dot product with every cached key
weights  = softmax(scores)
output   = weights @ V_cache    # weighted sum of cached values
```

`K_cache` and `V_cache` are the growing stores — one row per previous token, per layer, per attention head.

## Two places to intervene

A miner can act on two surfaces:

### 1. Storage — compress at write

Take the raw K/V vectors the model produced, compress them however you want, and save the compressed form. Decompress later when scoring attention.

This covers:

- **Quantization** — INT8, FP8, 3-bit Lloyd-Max, SmoothQuant variants
- **Eviction** — H2O-style importance tracking, StreamingLLM attention sinks, SnapKV observation windows
- **Layout / mixed precision** — per-head, per-layer adaptive strategies

### 2. Scoring — fused attention over compressed representations

Skip decompression entirely. Compute `Q @ K_compressed.T` directly, with a mathematical correction term that recovers the uncompressed result (approximately).

TurboQuant is the canonical example: random rotation + 3-bit Lloyd-Max on keys, then `Q @ K_3bit.T + QJL_correction` in one fused step. Never materializes uncompressed keys.

<Callout type="info" title="Which is more powerful?">
  Path 2 strictly dominates Path 1 — it saves both the memory of the decompressed buffer and the
  compute of decompression. But it requires the miner to own the attention math, not just the
  storage format. Cacheon's interface supports both.
</Callout>

## Why the cache is the right target

Two things make the KV cache the right first optimization target:

1. **It's the largest single cost.** At long context, the cache dwarfs weights in memory footprint.
2. **It's a clean interface.** Unlike batching or scheduling, the cache is local to a single request — a miner can reason about it without owning the whole serving stack.

V1 of Cacheon is scoped to this layer only. V2 expands to scheduling, batching, and cross-hardware policy generation. See [Roadmap](/docs/roadmap).

## Where the interface lives

Miners implement the `KVCachePolicy` class. Its two primary methods mirror the two intervention points:

- `write(keys, values, layer_idx, positions)` — receives the Q/K/V after projection and RoPE, stores whatever it wants
- `attend(query, layer_idx)` — receives the query, returns the final attention output (path 2 stays inside this call)

Full API on the [Policy interface](/docs/miners/policy-interface) page.
