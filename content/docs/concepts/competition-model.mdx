title: Competition model
description: King-of-the-hill with winner-take-all emission. Miners submit containerized inference servers. The fastest correct server wins.
---

Cacheon uses a **king-of-the-hill** incentive. There is always one reigning champion—the miner with the fastest inference server on record. **All subnet emissions flow to the king.** Challengers compete to dethrone them.

## How a round works

1. The validator scans on-chain commitments on every block.
2. Any hotkey with a revealed commitment that hasn't been evaluated yet is a **challenger**.
3. If there are no challengers, the king keeps its weight, no GPU runs, and the loop sleeps.
4. If challengers exist, the validator dispatches evaluation jobs and compares scores.
5. A challenger who beats the king's score **by more than a small, time-decaying margin** takes the throne -- see [defender's moat](#the-defenders-moat). Weights are updated: `winner = 1.0`, everyone else `= 0.0`.

## The defender's moat

The dethrone rule is not a plain `>`. It must pass two checks:

1. **Copycats get disqualified.** If a challenger's Docker image hash matches the current king's, but the hotkey differs and the commitment block is later, the challenger is DQ'd (`reason = "duplicate_of_king"`).
2. **Non-copies need a small edge.** A new server must strictly exceed `king_score x (1 + e(age))`, where e starts at **1%** at the block when the king is crowned and **decays linearly to 0** over **~50,400 blocks (~7 days)**. After that, the threshold is just `king_score` -- any strict improvement wins.

Why:

- Running the same server twice can produce small score variance from GPU clock drift, scheduling jitter, and network overhead. Without the moat, rankings could flip on noise.
- Without the copycat DQ, the dominant strategy would be "wait, copy, register."
- The decaying margin means a king cannot sit on a free 1% edge indefinitely.

## One commitment per hotkey

Each hotkey gets **exactly one submission**. To submit a new server, miners must register a new hotkey.

This is deliberate:

- **Commitment pressure.** Every submission costs something (registration fee + time). You will test locally before registering.
- **Clean on-chain record.** One hotkey maps to one Docker image at one pinned digest. No ambiguity about which server produced which score.

## What validators score on

Each evaluation round measures three properties per challenger:

| Axis        | How it's measured                                              |
| ----------- | -------------------------------------------------------------- |
| Correctness | First-divergence logprob analysis against baseline. Hard gate. |
| TTFT        | Time-to-first-token improvement vs. vLLM baseline              |
| Throughput  | Output tokens/sec improvement vs. vLLM baseline                |

The final score is:

```python
if not correctness_pass:
    score = 0.0
else:
    ttft_imp = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
    tps_imp  = max(0, (miner_tps - baseline_tps) / baseline_tps)
    score = 0.5 * ttft_imp + 0.5 * tps_imp
```

See [Scoring](/docs/evaluation/scoring) for the full rationale.
