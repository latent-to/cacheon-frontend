---
title: Competition model
description: Competition with leader and runner-up emission split. Miners submit containerized inference servers, and the fastest correct submissions earn the largest share of emissions.
---

Cacheon uses a competitive incentive model. The leader is the miner with the fastest inference server on record and receives most subnet emissions, while the runner-up earns a smaller share. Challengers compete to overtake the leader.

## How a round works

1. The validator scans on-chain commitments approximately every 5 minutes.
2. Any hotkey with a revealed commitment that hasn't been evaluated yet is a **challenger**.
3. If there are no challengers, the leader keeps its weight, no GPU runs, and the loop sleeps.
4. If challengers exist, the validator dispatches an evaluation job. The current leader and runner-up are re-evaluated in the same round alongside challengers, so all scores are fresh and no cached score carries forward.
5. A challenger who beats the leader's fresh score **by more than 1%** takes the top spot. See [defender's moat](#the-defenders-moat). The leader receives 80% of the competition pool; the runner-up receives 20%. Total pool scales with the leader's improvement score.

## The defender's moat

The overtake rule is not a plain `>`. It must pass two checks:

1. **Copycats get disqualified.** If a challenger's Docker image hash matches the current leader's, but the hotkey differs and the commitment block is later, the challenger is DQ'd (`reason = "duplicate_of_leader"`).
2. **Non-copies need a small edge.** A new server must strictly exceed `leader_score x 1.01`. The 1% moat is fixed.

Why:

- Running the same server twice can produce small score variance from GPU clock drift, scheduling jitter, and network overhead. Without the moat, rankings could flip on noise.
- Without the copycat disqualification, miners are incentivized to wait, copy the leader, and register it themselves.

## Resubmitting on the same hotkey

Each on-chain commitment is identified by **(hotkey, commit block)**. Miners can commit again on the same hotkey after paying a new submission fee. The validator always uses the **latest** revealed commitment per hotkey, but each new block is treated as a separate submission.

- **Payment per attempt.** Every new commitment must include fresh `payment_tx` and `payment_block` fields tied to that image digest via the on-chain remark.
- **Local testing first.** Submission fees and GPU eval time are real costs; validate locally before committing.
- **History remains identified by block.** Prior scores remain under the old `hotkey:commit_block` entry; a new commit starts a new challenger record.

## Submission fee (payment gate)

Before GPU evaluation, the CPU validator verifies that the miner paid the configured submission fee. This is done through the following steps:

1. Reads `payment_tx` and `payment_block` from the commitment JSON.
2. Calls `substrate.retrieve_extrinsic_by_hash` (one RPC lookup; no payment database).
3. Confirms that the extrinsic succeeded, was signed by the miner's coldkey, emitted a `Balances.Transfer` to the treasury with at least the configured fee, and included remark `cacheon:<hotkey>:<digest>` in the batch call.

Miners submit payment via `miner/commit.py`, which atomically runs `Utility.batch_all([transfer, remark])` then reveals the commitment. Utility.batch_all([transfer, remark]) before revealing the commitment. Validators enforce the submission payment gate with `CACHEON_SUBMISSION_FEE_RAO` environment variable (specified in RAO); setting it to 0 disables payment verification.

## What validators score on

Each evaluation round measures three properties per challenger:

| Axis        | How it's measured                                              |
| ----------- | -------------------------------------------------------------- |
| Correctness | First-divergence logprob analysis against baseline. Hard gate. |
| TTFT        | Time-to-first-token improvement vs. vLLM baseline              |
| Throughput  | Output tokens/sec improvement vs. vLLM baseline                |

The final score is:

```python
if not correctness_pass:
    score = 0.0
else:
    ttft_imp = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
    tps_imp  = max(0, (miner_tps - baseline_tps) / baseline_tps)
    score = 0.5 * ttft_imp + 0.5 * tps_imp
```

See [Scoring](/docs/evaluation/scoring) for the full rationale.
