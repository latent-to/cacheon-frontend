---
title: Evaluation harness
description: How the harness orchestrates two-pass containerized evaluation. Baseline cache, miner stress + audit, scoring container teacher-forcing, warmup, and median aggregation.
---

The harness pulls miner containers, runs Pass 1 (stress speed + match) and Pass 2 (audit generation + teacher-forcing scoring), and compares results against a cached vLLM generation baseline.

## Round lifecycle

1. **Sample prompts** from `block_hash`: 12 stress (10 scored + 2 warmup) and 8 audit
2. **Generation baseline once**: start `vllm/vllm-openai:latest`, run stress prompts, cache results, tear down
3. **Per miner** (leader, runner-up, then challengers):
   - `docker stop` the scoring container if it is running (GPU handoff)
   - Start miner container; run **Pass 1 stress** (12 prompts, discard 2 warmup) then **Pass 2 audit generation** (8 prompts, no warmup) in one session
   - Tear down miner container; reset GPUs
   - If Pass 1 aggregate match ≥ 25%: compute speed score from stress prompts; start or resume scoring vLLM (`v0.9.2`, audit-sized context cap) and teacher-force the 8 audit outputs
   - If match fails or Pass 2 fails: `score = 0.0`
4. **Kill scoring container** at end of round

Miner and scoring baseline **never share GPUs**. The scoring container stays alive via `docker stop` / `docker start` between miners.

### Warmup

Before scored stress prompts, the harness sends **2 warmup requests** that are discarded. Both baseline and miner receive the same warmup. The competition measures steady-state serving speed.

### Timeouts

| Timeout                                   | Value       | What happens if exceeded           |
| ----------------------------------------- | ----------- | ---------------------------------- |
| **Startup (miner / generation baseline)** | 10 minutes  | Container terminated, DQ           |
| **Startup (scoring baseline)**            | 15 minutes  | `baseline_scoring_unavailable`, DQ |
| **Per-prompt**                            | 120 seconds | Request cancelled, DQ              |

### GPU state cleanup

Between evaluations, the harness resets GPU state across all GPUs:

- CUDA contexts are destroyed
- GPU clocks are reset to default
- Memory allocator caches are cleared

Each miner starts from a clean GPU state, identical to what the generation baseline saw.

## vLLM images

| Role                           | Default image                             | Context                                                       |
| ------------------------------ | ----------------------------------------- | ------------------------------------------------------------- |
| **Pass 1 generation baseline** | `vllm/vllm-openai:latest` (digest-pinned) | Up to ~128k (`--max-model-len` from GPU count + model config) |
| **Pass 2 teacher-forcing**     | `vllm/vllm-openai:v0.9.2`                 | Audit-sized cap (~4.5k tokens), not full 128k                 |

Generation baseline results are **cached** per block hash under `baseline_cache/`. On a cache miss, baseline runs with the same warmup discards and streaming logprobs as miners.

## Prompt delivery

Stress prompts (Pass 1):

```json
{
  "model": "Qwen2.5-72B-Instruct",
  "messages": [{ "role": "user", "content": "<instruction with long passage>" }],
  "max_tokens": 512,
  "temperature": 0,
  "stream": true,
  "logprobs": true,
  "top_logprobs": 5
}
```

Audit prompts (Pass 2 generation in miner container):

```json
{
  "model": "Qwen2.5-72B-Instruct",
  "messages": [{ "role": "user", "content": "<instruction with ~4k passage>" }],
  "max_tokens": 256,
  "temperature": 0,
  "stream": true,
  "logprobs": true,
  "top_logprobs": 5
}
```

Prompt order is **deterministic** from the block-hash seed (same order for every miner in the round).

Pass 2 correctness scoring uses a non-streaming request with `prompt_logprobs=1` on the separate scoring container. See [Scoring](/docs/evaluation/scoring#pass-2-correctness-teacher-forcing).

## Measurement

### TTFT

Wall-clock time from HTTP request send to the first SSE data chunk. Includes server prefill, first-token sampling, and response serialization. Does not include container startup.

### Throughput (Pass 1 stress only)

**Aligned** output tokens per second on stress prompts (`max_tokens=512`):

```
throughput = N / (time_to_emit_token_N - time_to_emit_token_1)
```

`N` is the baseline's `output_tokens` for that prompt. If the miner stops early, that prompt gets **0 throughput credit** (not an automatic DQ).

### Aggregation

TTFT and throughput use the **median** across the **10 scored stress prompts**.

### Correctness (Pass 2)

Teacher-forcing on the scoring vLLM after miner teardown. See [Scoring](/docs/evaluation/scoring#pass-2-correctness-teacher-forcing).

## Per-prompt metrics

`per_prompt` on an `EvaluationRecord` covers **Pass 1 stress only** (10 entries):

| Field              | Type  | Description                                                  |
| ------------------ | ----- | ------------------------------------------------------------ |
| `ttft_s`           | float | Time to first token for this prompt                          |
| `throughput_tps`   | float | Aligned TPS for this prompt (`baseline_N / miner_time_to_N`) |
| `output_tokens`    | int   | Number of output tokens generated                            |
| `token_match_rate` | float | Fraction of tokens matching baseline for this prompt         |

These are available in `state.json` and uploaded to S3 for dashboards.

## Container log capture

After each evaluation (success or failure), the harness captures stdout/stderr via `docker logs` into `state-mainnet/container_logs/`:

- Generation baseline: `baseline_{cache_key}.log`
- Scoring baseline: `baseline_scoring.log`
- Miner: `uid{N}_{hotkey_prefix}_{block}.log`

Logs are uploaded to S3 with the rest of the state directory.
