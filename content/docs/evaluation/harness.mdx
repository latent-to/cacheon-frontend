The harness is the thing that actually loads the model, runs the policy, and measures the three score axes.

## HuggingFace Transformers, not vLLM

vLLM is faster and more production-grade. It's also the wrong choice for Cacheon's V1.

| Stack                        | Pros                                                                            | Cons                                                                                                                     |
| ---------------------------- | ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| **vLLM / PagedAttention**    | Fast. Production-grade. Handles batching, scheduling.                           | Manages its own KV cache internally. Replacing it requires fighting undocumented internals that change between versions. |
| **HuggingFace Transformers** | Plain Python. Every attention layer is a `forward()` method. Exact hook points. | Slower. Not production-grade. But we're measuring _relative_ improvements, not absolute throughput.                      |

vLLM's `PagedAttention` is deeply intertwined with its paged-memory allocator. You can't insert a custom policy without forking vLLM. Even if you did, the baseline would be measured against vLLM-specific behavior and not generalize.

HuggingFace gives you plain attention layers you can swap at runtime. Baseline and miner both run on the same HF stack, so relative comparisons are valid.

## How attention is patched

For each of the 28 attention layers in `Qwen2.5-7B-Instruct`, the harness replaces `self_attn.forward` with a function that calls the policy:

```python
def _patched_forward(self, hidden_states, attention_mask, position_ids, ...):
    q = self.q_proj(hidden_states)
    k = self.k_proj(hidden_states)
    v = self.v_proj(hidden_states)

    q, k = apply_rotary_pos_emb(q, k, cos, sin)   # RoPE applied BEFORE policy sees K

    policy.write(k, v, layer_idx=self.layer_idx, positions=position_ids)
    attn_out = policy.attend(q, layer_idx=self.layer_idx)

    return self.o_proj(attn_out.output)
```

The original forwards are saved and restored after the run — no permanent model mutation.

## The two execution modes

### `verify()` — correctness check

Run once at harness startup, before any eval:

1. Generate a reference output using the **unpatched** model.
2. Feed that exact token sequence through the **patched** model using `PassthroughPolicy`.
3. Compute per-token KL between the two.
4. Assert `KL < 1e-4` for every token.

If this fails, the monkey-patch has a bug — typically a RoPE ordering mistake or GQA reshape error. Nothing else will work until `verify()` passes.

### `run(policy, prompts)` — production eval

For each prompt:

1. Patch attention layers with the policy's `write`/`attend` hooks.
2. **Prefill** — one forward pass over the full prompt.
3. **Decode loop** — generate tokens one at a time until EOS or `max_new_tokens`.
4. Unpatch attention.
5. Collect latency, peak memory, and logits into a `RunResult`.

Same sequence of calls whether it's the baseline or the miner — that's what makes the comparison valid.

## Prefill vs. decode

The distinction matters because the shapes `policy.write` receives are different:

| Phase   | `policy.write` `seq_len`  | `policy.attend` `q_len` |
| ------- | ------------------------- | ----------------------- |
| Prefill | full prompt length (~32K) | full prompt length      |
| Decode  | 1                         | 1                       |

Your cache must grow on each decode step, not replace. The harness delivers `positions` so you always know where you are in the sequence.

## What the harness returns

```
RunResult
  output_texts        list[str]         — decoded text per prompt
  output_ids          list[list[int]]   — token ids per prompt
  all_logits          list[Tensor]      — [num_generated, vocab_size] per prompt
  latency_s           float             — wall-clock time for the full run
  peak_memory_bytes   int               — torch.cuda.max_memory_allocated()
  policy_memory_bytes int               — policy.memory_bytes() (self-reported)
```

`all_logits` is what the scoring module uses to compute KL against the baseline. It's per-position, so KL is averaged over every generated token.

## Qwen-specific gotchas

These are the places the monkey-patch has to get right:

### Grouped-query attention

`num_attention_heads = 28` but `num_key_value_heads = 4`. When the layer reshapes after projection:

- Q: `[batch, seq_len, 28, 128]`
- K, V: `[batch, seq_len, 4, 128]`

Your `write()` receives K and V with `num_kv_heads = 4` on the head dim. `attend()` receives Q with `num_heads = 28`. You handle the broadcast — typical pattern is `K.repeat_interleave(28 // 4, dim=1)` inside `attend`.

### RoPE ordering

Qwen applies rotary position embeddings to Q and K **after** projection but **before** attention. The harness applies RoPE to K before calling `policy.write()`. If your cache stores pre-RoPE keys and you try to re-apply rotation later, scores are wrong.

Just store what you're given.

### Attention mask

The harness does not pass an attention mask to `policy.attend()`. Causal masking is implicit: your cache only contains tokens at positions `< current position`, because `write` is called before `attend` in each step, and the cache at step `t` only contains keys/values from steps `≤ t`.

If you materialize a score tensor across all cached positions, you don't need to mask — there's nothing illegitimate there.

## Why V1 runs on a single GPU pod

Eval happens on one H100 80GB, run by one operator (the subnet owner). This isn't a decentralization statement — it's operational reality:

- **Cross-hardware reproducibility.** An H100 and an A100 running the same policy produce slightly different floating-point results. If multiple validators ran eval independently, KL values would disagree, and rankings would be inconsistent.
- **Centralized eval eliminates that variance.** Everyone sees the same numbers because everyone's numbers come from the same hardware.

The trust concentration is real. The V2 mitigation is **publishing eval rollouts** — prompts, baseline logits, KL per token — to verifiable storage (Cloudflare R2 is the candidate). Anyone can then recompute scores offline and verify. Verify-after-the-fact, not run-it-yourself.
