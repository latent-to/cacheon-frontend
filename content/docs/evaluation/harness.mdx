title: Evaluation harness

The harness is the system that pulls miner containers, starts them, sends prompts, and measures performance against the vLLM baseline.

## Lifecycle

For each challenger:

```
1. Pull Docker image by digest
2. Start container with --gpus all, model volume mount, internal network
3. Poll GET /health every 5s until 200 (startup timeout: 10 min)
4. Send 2 warmup prompts (discarded, see below)
5. Send 10 streaming requests with logprobs; measure TTFT + throughput and collect tokens + logprobs in the same pass
6. Stop and remove container
7. Clean up GPU state (reset clocks, clear caches, all GPUs)
8. Compute median TTFT/throughput, correctness check, score
9. Compare against king
```

### Warmup

Before scored prompts, the harness sends 2 warmup requests that are discarded. This absorbs one-time costs: CUDA graph compilation, KV cache pre-allocation, GPU clock ramping. Both baseline and miner receive the same warmup treatment. The competition measures steady-state serving speed.

### Timeouts

| Timeout        | Value       | What happens if exceeded |
| -------------- | ----------- | ------------------------ |
| **Startup**    | 10 minutes  | Container terminated, DQ |
| **Per-prompt** | 120 seconds | Request cancelled, DQ    |

### GPU state cleanup

Between evaluations, the harness resets GPU state across all GPUs to prevent one miner's run from affecting the next:

- CUDA contexts are destroyed
- GPU clocks are reset to default
- Memory allocator caches are cleared

This ensures each miner starts from a clean GPU state, identical to what the baseline saw.

## vLLM baseline

The baseline is a pinned version of vLLM serving `Qwen2.5-72B-Instruct`. The validator starts the baseline container on demand when prompts change, runs evaluation, caches the results, then tears it down.

Baseline results are **pre-computed and cached** per prompt set. When the block hash changes (producing new prompts), the baseline runs once. All challengers in that round are compared against the same cached baseline numbers. This saves ~50% of eval wall time per challenger.

On a cache miss, baseline is driven the same way as challengers: streaming completions with logprobs and the same warmup discards.

## Prompt delivery

Prompts are delivered as OpenAI-compatible chat completion requests over HTTP:

```json
{
  "model": "Qwen2.5-72B-Instruct",
  "messages": [{ "role": "user", "content": "<instruction with passage context>" }],
  "max_tokens": 256,
  "temperature": 0,
  "stream": true,
  "logprobs": true,
  "top_logprobs": 5
}
```

The same prompts are sent to both baseline and miner. Prompt order is shuffled per challenger to prevent order-dependent optimizations.

## Measurement

### TTFT

Measured as wall-clock time from the moment the HTTP request is sent to the moment the first SSE data chunk arrives. This includes:

- Server-side prefill computation
- First token sampling
- HTTP response serialization

It does not include container startup (that happens before evaluation begins).

### Throughput

Measured as output tokens per second, computed from SSE chunk timestamps:

```
throughput = total_output_tokens / (last_chunk_time - first_chunk_time)
```

This measures sustained decode speed, not including prefill.

### Aggregation

Both TTFT and throughput are aggregated using the **median** across the 10 scored prompts. Median is robust to outliers (GC pauses, memory pressure spikes) and prevents gaming by being fast on easy prompts but slow on hard ones.

### Correctness

Measured from the same streaming pass that captures speed. Per-prompt first-divergence analysis against baseline output, with logprob verification at the first mismatch position. See [Scoring](/docs/evaluation/scoring) for the full correctness gate specification.

## Per-prompt metrics

Every scored prompt produces a detailed breakdown stored in the evaluation record:

| Field              | Type  | Description                                          |
| ------------------ | ----- | ---------------------------------------------------- |
| `ttft_s`           | float | Time to first token for this prompt                  |
| `throughput_tps`   | float | Tokens per second for this prompt                    |
| `output_tokens`    | int   | Number of output tokens generated                    |
| `token_match_rate` | float | Fraction of tokens matching baseline for this prompt |

These per-prompt results are available in `state.json` (inside each `EvaluationRecord`) and uploaded to S3 for frontend dashboards. Miners can inspect their performance per prompt to understand where their server excels or struggles.

## Container log capture

After each evaluation (success or failure), the harness captures the full stdout/stderr from the container via `docker logs`. Logs are saved to `state/container_logs/`:

- Baseline: `baseline_{cache_key}.log`
- Miner: `uid{N}_{hotkey_prefix}_{block}.log`

Container logs are never truncated. They are uploaded to S3 with the rest of the state directory for post-mortem debugging.
