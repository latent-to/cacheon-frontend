The harness is the system that pulls miner containers, starts them, sends prompts, and measures performance against the vLLM baseline.

## Containerized evaluation, not in-process hooks

Some benchmarks run miner code inside the evaluator's process. Cacheon deliberately avoids this:

| Approach                  | Pros                                                               | Cons                                                           |
| ------------------------- | ------------------------------------------------------------------ | -------------------------------------------------------------- |
| **In-process hooks**      | Simple. Direct measurement.                                        | Limits miners to one language, one framework, one API surface. |
| **Containerized servers** | Any language, any runtime, any optimization technique. Full-stack. | Container overhead. Harder to measure. More infra.             |

Containers let miners use Rust, custom CUDA, custom memory allocators -- anything that makes inference faster. The trade-off is infrastructure complexity, which is the validator's problem, not the miner's.

## Evaluation lifecycle

For each challenger:

```
1. Pull Docker image by digest
2. Start container with --gpus '"device=0,1,2,3"', model volume mount, port mapping, no network
3. Poll GET /health every 5s until 200 (startup timeout: 10 min)
4. Send 1-2 warmup prompts (discarded, not scored)
5. Run speed pass: 10 streaming requests, measure TTFT + throughput
6. Run correctness pass: 10 non-streaming requests with logprobs
7. Stop and remove container
8. Clean up GPU state (reset clocks, clear caches, all 4 GPUs)
9. Compute median TTFT/throughput, correctness check, score
10. Compare against king
```

### Warmup

Before scored prompts, the harness sends 1-2 warmup requests that are discarded. This absorbs one-time costs: CUDA graph compilation, KV cache pre-allocation, GPU clock ramping. Both baseline and miner receive the same warmup treatment. The competition measures steady-state serving speed.

### Timeouts

| Timeout        | Value                         | What happens if exceeded |
| -------------- | ----------------------------- | ------------------------ |
| **Startup**    | 10 minutes                    | Container killed, DQ     |
| **Per-prompt** | 120 seconds                   | Request cancelled, DQ    |
| **Total eval** | Configurable (default 30 min) | Eval aborted, DQ         |

### GPU state cleanup

Between evaluations, the harness resets GPU state across all 4 GPUs to prevent one miner's run from affecting the next:

- CUDA contexts are destroyed
- GPU clocks are reset to default
- Memory allocator caches are cleared

This ensures each miner starts from a clean GPU state, identical to what the baseline saw.

## vLLM baseline

The baseline is a pinned version of vLLM serving `Qwen2.5-72B-Instruct` on the same 4x H200 with tensor parallelism. It runs as an always-on container alongside the evaluation harness.

Baseline results are **pre-computed and cached** per prompt set. When the block hash changes (producing new prompts), the baseline runs once. All challengers in that round are compared against the same cached baseline numbers. This saves ~50% of eval wall time per challenger.

### Baseline warmup

The baseline receives the same 1-2 warmup prompts before its measured run. Baseline measurement follows the same two-pass protocol as miner evaluation.

## Prompt delivery

Prompts are delivered as OpenAI-compatible chat completion requests over HTTP:

```json
{
  "model": "Qwen2.5-72B-Instruct",
  "messages": [{ "role": "user", "content": "<instruction with passage context>" }],
  "max_tokens": 256,
  "temperature": 0,
  "stream": true
}
```

The same prompts are sent to both baseline and miner. Prompt order is shuffled per challenger to prevent order-dependent optimizations.

## Measurement

### TTFT

Measured as wall-clock time from the moment the HTTP request is sent to the moment the first SSE data chunk arrives. This includes:

- Server-side prefill computation
- First token sampling
- HTTP response serialization

It does not include container startup (that happens before evaluation begins).

### Throughput

Measured as output tokens per second, computed from SSE chunk timestamps:

```
throughput = total_output_tokens / (last_chunk_time - first_chunk_time)
```

This measures sustained decode speed, not including prefill.

### Aggregation

Both TTFT and throughput are aggregated using the **median** across the 10 scored prompts. Median is robust to outliers (GC pauses, memory pressure spikes) and prevents gaming by being fast on easy prompts but slow on hard ones.

### Correctness

Measured from the non-streaming correctness pass. Token-by-token comparison against baseline output, plus logprob spot-checks at divergent positions. See [Scoring](/docs/evaluation/scoring) for the full correctness gate specification.

## Why V1 runs on a single GPU pod

Eval happens on one 4x H200 pod, run by one operator (the subnet owner). This is not a decentralization statement; it is operational reality:

- **Cross-hardware reproducibility.** Different GPU generations and interconnect topologies produce different floating-point results and different latency profiles. Multiple validators would disagree on scores.
- **Container overhead variance.** Docker startup, GPU initialization, and network stack behavior vary between hosts. Centralizing eval eliminates that variance.
- **Multi-GPU topology sensitivity.** Tensor parallel performance depends on NVLink topology. A 4-GPU all-connected NVSwitch setup behaves differently from a 2+2 PCIe bridge setup. Fixing the topology is essential for fair comparison.

The V2 mitigation is publishing full eval artifacts (prompts, baseline outputs, miner outputs, scores) to verifiable storage so anyone can audit.
