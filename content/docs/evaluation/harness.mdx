title: Harness
description: How the harness orchestrates one-pass containerized evaluation, including baseline caching, miner generation, batched teacher-forcing, warmup requests, and median aggregation.
---

The harness pulls miner containers, runs evaluation prompts (speed + correctness on the same outputs), and compares speed against a cached vLLM generation baseline.

## Round lifecycle

1. **Sample prompts** from `block_hash`: 12 evaluation prompts (10 scored + 2 warmup), ~6k input / 512 output each
2. **Baseline once**: start `vllm/vllm-openai:v0.22.0`, run evaluation prompts, cache results, tear down
3. **Per miner** (leader, runner-up, then challengers):
   - Start miner container; run **12 eval prompts** (2 warmup discarded, 10 scored)
   - Tear down miner container; reset GPUs
   - If text-similarity pre-filter passes, queue the 10 scored outputs for batched teacher-forcing
4. **Teacher-forcing batch**: start scoring vLLM once (`v0.9.2`, ~7k context cap, 600s startup timeout), teacher-force all queued outputs, tear down scoring container
5. Persist evaluation records and rerank

Miner and scoring baseline **never share GPUs**. Teacher-forcing uses a single scoring container boot per round (no pause/resume between miners).

### Timeouts

| Timeout                           | Value        | What happens if exceeded           |
| --------------------------------- | ------------ | ---------------------------------- |
| **Startup (miner)**               | 12.5 minutes | Container terminated, DQ           |
| **Startup (generation baseline)** | 20 minutes   | Eval abort                         |
| **Startup (scoring baseline)**    | 10 minutes   | `baseline_scoring_unavailable`, DQ |
| **Per-prompt**                    | 120 seconds  | Request cancelled, DQ              |

### GPU state cleanup

Between evaluations, the harness resets GPU state across all GPUs:

- CUDA contexts are destroyed
- GPU clocks are reset to default
- Memory allocator caches are cleared

Each miner starts from a clean GPU state, identical to what the generation baseline saw.

## vLLM images

| Role                                                 | Default image              | Context                                                                                           |
| ---------------------------------------------------- | -------------------------- | ------------------------------------------------------------------------------------------------- |
| **Generation baseline** (speed reference)            | `vllm/vllm-openai:v0.22.0` | 32k on mainnet (`--max-model-len` from mounted model config; capped at `max_position_embeddings`) |
| **Scoring baseline** (correctness / teacher-forcing) | `vllm/vllm-openai:v0.9.2`  | 7168 tokens (`6144` input + overhead + `512` output budget; not the 32k generation cap)           |

Override via `CACHEON_BASELINE_IMAGE` / `CACHEON_SCORING_IMAGE` in [validator config](https://github.com/latent-to/cacheon/blob/main/validator/config.py).

## Prompt delivery

Evaluation prompts (generation in miner container):

```json
{
  "model": "Qwen2.5-72B-Instruct",
  "messages": [{ "role": "user", "content": "<instruction with ~6k passage>" }],
  "max_tokens": 512,
  "temperature": 0,
  "stream": true,
  "logprobs": true,
  "top_logprobs": 5
}
```

Prompt order is **deterministic** from the block-hash seed (same order for every miner in the round).

Correctness scoring uses a non-streaming request with `prompt_logprobs=1` on the separate scoring container over the **same outputs**. See [Scoring](/docs/evaluation/scoring#correctness-teacher-forcing).

## Measurement

### End-to-end speed

Wall time from HTTP request send to the aligned k-th output token:

```
miner_e2e = ttft + decode[k - 1]
baseline_e2e = baseline_ttft + baseline_decode[k - 1]
```

`k = min(baseline_N, miner_N)`. The miner must emit at least `ceil(0.9 * baseline_N)` tokens (minimum 2) or that prompt gets 0 speed credit.

### Aggregation

End-to-end speed improvement uses the **median** across the **10 scored prompts**.

### Correctness

Teacher-forcing on the scoring vLLM after all miners finish generation. See [Scoring](/docs/evaluation/scoring#correctness-teacher-forcing).

## Per-prompt metrics

`per_prompt` on an `EvaluationRecord` covers the **10 scored evaluation prompts**:

| Field              | Type  | Description                                                 |
| ------------------ | ----- | ----------------------------------------------------------- |
| `ttft_s`           | float | Time to first token for this prompt (telemetry)             |
| `e2e_s`            | float | End-to-end wall time to aligned token count for this prompt |
| `output_tokens`    | int   | Number of output tokens generated                           |
| `token_match_rate` | float | Fraction of tokens matching baseline for this prompt        |

These fields are stored in Postgres on each evaluation and exposed by the monitoring API.

## Container log capture

After each evaluation (success or failure), the harness captures stdout/stderr via `docker logs` into `state-mainnet/container_logs/`:

- Generation baseline: `baseline_{cache_key}.log`
- Scoring baseline: `baseline_scoring_{block}_{timestamp}.log` (same `{timestamp}` as `gpu_eval_{timestamp}.log`)
- Miner: `uid{N}_{hotkey_prefix}_{block}.log`

Logs are uploaded to S3 under `logs/` and `container_logs/` only.
