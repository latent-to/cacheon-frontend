The harness is the referee. For each submission it runs the fixed model **twice** in pinned sglang under CUDA graphs (a stock **baseline** and a **candidate** with the miner kernel swapped in) and diffs them. This page is the mechanism: how a kernel that the validator does not trust gets *into* a running model, and how the timing is kept out of the kernel's reach.

## Two launches, everything else equal

```python
baseline  = _run_launch(cfg, prompts, bundle_path="",          active=False)  # stock kernels
candidate = _run_launch(cfg, prompts, bundle_path=bundle_path, active=True)   # miner kernel
```

Same weights, same seed, same sampler, same prompts. The **only** difference is the one op the miner replaced, so any delta in throughput or output distribution is attributable to that kernel. The baseline *is* the gold standard: the same model on the same box with the stock kernels, which is local, exact, free, deterministic, and adversary-independent (unlike a production API, which is non-reproducible, sparse, and breaks consensus).

**Each launch runs in its own fresh process.** sglang plus deterministic mode set process-global CUDA and torch state (deterministic algorithms, the cuBLAS workspace, the sampling backend); in a shared driver, the baseline's state corrupted the candidate on large MoE models (on gpt-oss-120b a no-op kernel appeared to "regress" to 0%). Isolated processes make the two launches independent and free all GPU memory between them.

## The obstacle: sglang runs the model in a separate process

`sglang.Engine` does `mp.set_start_method("spawn")` and launches the model in a separate **scheduler** process. With *spawn*, that child is a brand-new Python interpreter that re-imports sglang from scratch, so patching a class in the parent process does nothing: the model runs in the child, and the patch is in the parent. Naive monkeypatching silently no-ops.

## The seam: a `.pth` + a post-import hook

Optima needs code to run **inside every interpreter in the venv**, including the spawned child, regardless of sglang version. Python offers exactly one such hook: **`.pth` files**. A line in a `.pth` file under site-packages that starts with `import` is executed at interpreter startup, in the parent *and* in every spawned child. The install writes one line:

```bash
echo 'import optima.bootstrap' > $SITE_PACKAGES/optima.pth
```

`optima.bootstrap` does not import sglang at startup (too heavy). Instead it registers a **meta-path finder** that watches for the seam-target modules (activation, layernorm, radix attention, the fused-MoE layer, and the distributed parallel-state) and, the moment one finishes loading, runs `seam.activate()` against it. This is the post-import hook pattern:

```python
class _SeamFinder(MetaPathFinder):
    def find_spec(self, fullname, ...):
        if fullname not in _TARGETS:            # the seam chokepoint modules
            return None
        spec = <real spec from the other finders>
        spec.loader = _wrap_loader(spec.loader)  # run seam.activate() after exec
        return spec
```

The pinned sglang does also ship a hook/plugin framework, so an entry-point hook is available; the `.pth` path stays primary because it is version-independent and known spawn-safe. Migrating to the sanctioned hook is tracked as future work.

### What `seam.activate()` does

1. **Installs the dispatcher** into the target class (for example `SiluAndMul.forward_cuda`), keeping the original for fallback. Patching at the *class* before the model is built means every instance binds the dispatcher when it is constructed.
2. **Decides whether to load the miner kernel**, from env:
   - `OPTIMA_ACTIVE=1` and `OPTIMA_BUNDLE_PATH=<dir>` → re-scan, load, and register the kernel, and enable the registry. This is the **candidate** run.
   - otherwise → the registry stays empty, the dispatcher always falls back to stock. This is the **baseline** run.

The *same* process code serves both launches; the validator picks baseline versus candidate by setting env before each launch. Block and collective seams are opt-in per slot (`OPTIMA_ATTENTION_SEAM`, `OPTIMA_MOE_SEAM`, `OPTIMA_COLLECTIVE_SEAM`).

## The dispatcher: the one place a miner kernel is called

```python
def dispatched(self, x):
    impl = REGISTRY.lookup("activation.silu_and_mul",
                           dtype_name=..., last_dim=x.shape[-1], arch=...)
    if impl is None:
        return baseline_forward(self, x)         # not eligible / not active
    out = torch.empty(<validator-chosen shape>, dtype=x.dtype, device=x.device)
    try:
        impl.entry(x, out)                        # the ONLY miner call
    except Exception:
        return baseline_forward(self, x)          # crash -> fall back
    return out
```

The validator owns the **output allocation**, the **eligibility check** (dtype / arch / size, else fall back), and the **fallback**. The miner's `entry` only ever sees pre-allocated tensors and only fills the output; it never allocates output, never returns tensors, and never sees the final tokens or logprobs. This is the anti-cheat boundary: because the kernel is strictly upstream of the sampler, there is no final output to substitute, so the "run gibberish, fetch the real answer from an API" attack has nothing to attack.

## Tamper-resistant, out-of-process timing (`mark_driver`)

The `.pth` runs in **every** process, including the validator's *driver* (the one that calls `engine.generate()` and holds `time.perf_counter`). If the driver also imported the miner module, a malicious kernel's module-level code could monkeypatch the driver's clock.

The fix: the driver calls `seam.mark_driver()` **before importing sglang**. `activate()` checks that flag and, in the driver, installs a pass-through seam but **never imports the miner module**. The miner kernel is loaded only in the spawned scheduler child (a fresh process where the flag is not set). So:

- **scheduler child** — runs the miner kernel (this is where it executes).
- **driver process** — holds the timer, never touches miner code.

The two are different OS processes communicating over IPC, so the kernel cannot reach the timer. This is what "out-of-process timing" means, and it is why a broken kernel still fails the gate (proving the kernel really runs in the child) while the timer stays clean.

<Callout title="Honest scope" type="info">
Out-of-process timing makes the wall-clock **denominator** genuinely tamper-proof. Everything the scheduler *sends* the driver (token count, logprobs, text) is only as trustworthy as process isolation, which is the largest not-yet-built piece for genuinely untrusted miners. Today the VM is the trust boundary; that is fine for a controlled bootstrap, and the token budget is fixed a priori (`ignore_eos` + set `max_new_tokens`) so the throughput numerator does not depend on a scheduler-reported count.
</Callout>

## One candidate forward pass, end to end

1. Driver: `seam.mark_driver()`; set `OPTIMA_ACTIVE=1`, `OPTIMA_BUNDLE_PATH=...`; `import sglang`; `Engine(...)`.
2. Engine spawns the scheduler child. Child boots → `.pth` runs `import optima.bootstrap` → meta-path finder installed.
3. Child imports the target module → finder fires → `seam.activate()` patches the class, loads and registers the miner kernel, enables the registry.
4. Child builds the model; each target instance binds the dispatcher.
5. Driver: `engine.generate(prompts, return_logprob=True, top_logprobs_num=k)`.
6. Child runs the forward pass; at the seam, the dispatcher calls the miner `entry(x, out)`. The rest of the model (attention, norms, sampler) is stock sglang.
7. Child returns tokens plus per-position top-k logprobs over IPC to the driver.
8. Driver stamps wall-clock around the call and reads the logprobs. **Neither the timer nor the logprob computation ran any miner code.**

## Graphs-ON is the only regime that counts

Scoring runs with CUDA graphs ON because graphs-off cripples the baseline (~4.5-6.5x), and beating sglang graphs-on is the whole point. Op seams capture directly inside the graph; a block or collective kernel must declare `graph_safe: true` in its metadata (no host syncs, no data-dependent shapes inside the capture) or it falls back to the baseline in-graph and cannot win. See [Data contracts](/docs/evaluation/data-contracts) for the metadata schema.
