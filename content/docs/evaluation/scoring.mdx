title: Scoring
description: Two-pass evaluation. Pass 1 speed vs cached baseline with a match gate. Pass 2 teacher-forcing correctness. Median aggregation across stress prompts.
---

Final score is **correctness-gated**: Pass 1 aggregate token match must clear the threshold, Pass 2 teacher-forcing must pass, and only then does the speed score count. Otherwise `score = 0.0`.

## Two-pass overview

| Pass                | Prompts              | Context / output                            | Container                                                                     | Purpose                       |
| ------------------- | -------------------- | ------------------------------------------- | ----------------------------------------------------------------------------- | ----------------------------- |
| **Pass 1 (stress)** | 10 scored + 2 warmup | Long context (up to ~128k cap), **512** out | Miner + cached generation baseline (`vllm/vllm-openai:latest`)                | Speed vs baseline; match gate |
| **Pass 2 (audit)**  | 8, no warmup         | ~**4k** in / **256** out                    | Miner generates outputs; separate scoring vLLM (`v0.9.2`) teacher-forces them | Correctness gate              |

Pass 1 baseline numbers are **pre-computed once per block hash** and cached. Pass 2 does not re-run a generation baseline for speed; a dedicated scoring container runs `prompt_logprobs` over each miner's audit outputs after the miner container is torn down.

## Scoring formula

```python
if pass1_match_fail or not pass2_correctness_pass:
    score = 0.0
else:
    ttft_imp = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
    tps_imp  = max(0, (miner_tps - baseline_tps) / baseline_tps)
    score = 0.5 * ttft_imp + 0.5 * tps_imp
```

<Callout title="Scoring axes" type="info">
Gates and axes, in order of enforcement:

1. **Pass 1 match gate:** aggregate token match below 25% → `pass1_match_fail`, skip Pass 2 scoring.
2. **Pass 2 correctness gate:** teacher-forcing fail → `correctness_fail` or `scoring_infra_fail`; speed zeroed.
3. **TTFT:** 50% of the weighted speed score (stress prompts only).
4. **Throughput:** 50% of the weighted speed score (stress prompts only).

Improvements are floored at zero. If your server is slower than vLLM on one axis, that axis contributes 0, not a negative number.

</Callout>

## Pass 1: speed and match

### Aggregation

Pass 1 sends **12 stress prompts** (2 warmup discarded, **10 scored**) to the miner. Baseline TTFT and aligned throughput use the **median** across those 10 scored prompts.

### Measurement

Stress prompts use **streaming chat completions** with `logprobs` (`stream=true`, `logprobs=true`, `top_logprobs=5`). TTFT and throughput come from SSE timing. Top-5 logprobs are collected and cached with the baseline but are **not** a DQ gate in V1.

### Warmup

Before scored stress prompts, the validator sends **2 warmup prompts** that are discarded. Both baseline and miner receive the same warmup. The competition measures steady-state serving speed, not startup optimization.

### Pass 1 match gate

Aggregate match is the **mean of per-prompt token match rates** across the 10 scored stress prompts (not a single pooled token average).

If aggregate match is below `PASS1_MATCH_DQ_THRESHOLD` (default **25%**), the miner is DQ'd with `pass1_match_fail` and Pass 2 teacher-forcing is skipped. Configurable via `CACHEON_PASS1_MATCH_DQ_THRESHOLD`.

`token_match_rate` on the evaluation record stores this aggregate Pass 1 value for dashboards.

## Pass 2: correctness (teacher-forcing)

After Pass 1 match passes, the validator starts or resumes a **separate scoring vLLM** container (`CACHEON_SCORING_IMAGE`, default `vllm/vllm-openai:v0.9.2`). It boots with a **small `--max-model-len`** sized for audit prompts (~4.5k), not the 128k stress budget, so `prompt_logprobs` stays stable.

The scoring container teacher-forces each of the **8 audit outputs** the miner already generated (`prompt_logprobs=1` on `/v1/chat/completions` with prompt + assistant text). Per prompt, all miner output tokens must satisfy:

- **Mean logprob** ≥ `MEAN_LOGPROB_THRESHOLD` (default `-4.0`)
- **Min logprob** for any single token ≥ `MIN_LOGPROB_THRESHOLD` (default `-12.0`)

Configurable via `CACHEON_MEAN_LOGPROB_THRESHOLD` and `CACHEON_MIN_LOGPROB_THRESHOLD`.

**Every audit prompt must pass.** One failure disqualifies the submission (`correctness_fail`). Empty or missing logprobs (OOM, HTTP 500) → `scoring_infra_fail`.

The ~4k audit pass does **not** prove 128k behavior; that is a known V1 gap.

### Why this catches bad actors

- **Garbage after a valid prefix**: low mean/min logprob on garbage tokens.
- **Wrong or quantized weights**: baseline assigns near-zero probability to miner tokens.
- **Implausible fluent padding**: may pass if every inserted token is high probability (TPS gaming is handled separately; see throughput below).

### Scoring baseline image

Speed eval uses the generation baseline (`vllm/vllm-openai:latest`, digest-pinned). Teacher-forcing uses **`vllm/vllm-openai:v0.9.2`** because newer vLLM builds OOM on long `prompt_logprobs`; audit prompts stay short (~4k) and the scoring server context cap matches that budget.

## TTFT improvement

```
ttft_improvement = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
```

Computed from the **median** TTFT across 10 scored stress prompts. At long input (often tens of thousands of tokens on 8x H200/B200, `TP=8`), TTFT is dominated by prefill attention.

## Throughput improvement

Per stress prompt, aligned throughput is:

```
miner_tps = baseline_output_tokens / miner_time_to_emit_that_many_tokens
baseline_tps = baseline_output_tokens / baseline_time_to_emit_that_many_tokens
```

If the miner emits fewer tokens than the baseline for that prompt, `miner_tps = 0` for that prompt (no speed credit, not a length DQ).

Improvement vs baseline:

```
throughput_improvement = max(0, (miner_tps - baseline_tps) / baseline_tps)
```

The final score uses the **median** across the 10 scored stress prompts (`max_tokens=512`). Extra tokens beyond the baseline length do not help: the TPS numerator is fixed to the baseline token count.

## Baseline: vLLM

The generation baseline is a pinned vLLM instance serving `Qwen2.5-72B-Instruct` on the same hardware as challengers. See [GPU requirements](/docs/validators/overview#gpu-requirements). Baseline results are **pre-computed and cached** per block hash; all challengers in the round share the same cached numbers.

## Explicitly not in V1

- **Speculative decoding** -- banned. Simplifies correctness checking.
- **Weight quantization** -- banned. All servers must load exact BF16 weights.
- **Alternate models** -- banned. Same model, same weights.
- **Approximate attention** -- banned if it changes outputs beyond the gate's tolerance.
- **Sampling-based evaluation** -- all eval is greedy (temperature=0).

These restrictions will be relaxed in V2 once the eval pipeline is proven.

## Reproducibility

Every scored run produces an `EvaluationRecord` with `score`, `ttft_improvement`, `throughput_improvement`, `token_match_rate`, `disqualified`, and `disqualify_reason`. Combined with the block hash that seeded prompts and pinned Docker digests, anyone can replay the eval and verify the result.

## Emission scaling

Total emission allocated to the competition pool scales linearly with the leader's improvement score:

`comp_frac = min(1.0, leader_score / SCORE_EMISSION_TARGET)`

With the default target of 0.10 (10% improvement over baseline):

- A leader scoring 0.05 earns 50% of total emission for the pool
- A leader scoring 0.10 or above earns 100%
- The remainder goes to a burn UID

Within the competition pool, the leader receives 80% and the runner-up receives 20%. When there is no runner-up, the leader receives the full pool.
