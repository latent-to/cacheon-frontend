Scoring is **correctness-gated**: failing correctness yields zero; otherwise _Time to First Token_ (_TTFT_) and throughput improvements versus the pinned vLLM baseline are averaged (50/50).

## Scoring formula

```python
if not correctness_pass:
    score = 0.0
else:
    ttft_imp = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
    tps_imp  = max(0, (miner_tps - baseline_tps) / baseline_tps)
    score = 0.5 * ttft_imp + 0.5 * tps_imp
```

<Callout title="Scoring axes" type="info">
Three axes, in order of enforcement:

1. **Correctness gate:** fail this and nothing else matters.
2. **TTFT:** 50% of the weighted score.
3. **Throughput:** 50% of the weighted score.

Improvements are floored at zero. If your server is slower than vLLM on one axis, that axis contributes 0, not a negative number.

The equal split rewards **prefill latency** (TTFT, perceived wait and interactive steps) and **decode throughput** (cost per GPU-hour and long completions).

</Callout>

## Aggregation

Each evaluation round sends **10 prompts** to both baseline and miner. Per-prompt TTFT and throughput use the **median** across prompts so spikes and cherry-picking easy prompts hurt less.

## Single-pass evaluation

Each prompt is sent once over **streaming chat completions** with `logprobs` (`stream=true`, `logprobs=true`, `top_logprobs=5`). TTFT and throughput come from SSE timing; tokens and top logprobs feed correctness. Extra logprob cost applies equally to baseline and miner, so relative improvements stay comparable.

## Warmup

Before the scored prompts, the validator sends **1-2 warmup prompts** that are discarded. This absorbs one-time costs: CUDA graph compilation, KV cache pre-allocation, GPU clock ramping. Both baseline and miner receive the same warmup. The competition measures steady-state serving speed, not startup optimization.

## Correctness gate

Correctness verifies that the miner's streamed output is **plausible under the baseline model** at `temperature=0`. After phase 1 (speed), a pinned scoring vLLM image runs `prompt_logprobs` over the full prompt plus the miner's assistant text (capped at 256 tokens).

For each miner output token, the baseline assigns a logprob conditioned on the miner's own prefix (teacher-forcing). The prompt passes only if:

- **Mean logprob** across all miner tokens is above `MEAN_LOGPROB_THRESHOLD` (default `-4.0`)
- **Min logprob** for any single token is above `MIN_LOGPROB_THRESHOLD` (default `-12.0`)

Configurable via `CACHEON_MEAN_LOGPROB_THRESHOLD` and `CACHEON_MIN_LOGPROB_THRESHOLD`.

### Prompt-level pass rate

**Every scored prompt must pass.** A single failed prompt disqualifies the submission. Garbage tokens, fabricated streams, and many quantized-weight setups score far below threshold.

`token_match_rate` is recorded for dashboards but is **not** a DQ gate in V1. The subnet goal is same-model inference optimization; at `temperature=0`, honest stacks should still show high match in practice.

### Why this catches bad actors

- **Garbage after a valid prefix**: low mean/min logprob on garbage tokens.
- **Wrong or quantized weights**: baseline assigns near-zero probability to miner tokens.
- **Implausible fluent padding**: may pass if every inserted token is high probability (TPS gaming is handled separately; see throughput below).

### Scoring baseline image

Speed eval uses the latest pinned generation baseline. Teacher-forcing uses a separate pinned image (`v0.6.6.post1`) because large `prompt_logprobs` on long contexts is unstable on newer vLLM builds.

## TTFT improvement

TTFT improvements are measured using the following formula:

```
ttft_improvement = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
```

The final score is computed as the median across all 10 prompts and floored at zero. This means that servers slower than the baseline on TTFT receive no score contribution from this axis.

TTFT captures prefill efficiency; that is, how fast the server processes the full input prompt and produces the first output token. At 32K input tokens on e.g. 8x H200 (`TP=8`), this is dominated by the attention computation during prefill.

## Throughput improvement

Per prompt, aligned throughput is:

```
miner_tps = baseline_output_tokens / miner_time_to_emit_that_many_tokens
baseline_tps = baseline_output_tokens / baseline_time_to_emit_that_many_tokens
```

If the miner emits fewer tokens than the baseline for that prompt, `miner_tps = 0` for that prompt (no speed credit, not a length DQ).

Improvement vs baseline:

```
throughput_improvement = max(0, (miner_tps - baseline_tps) / baseline_tps)
```

The final score uses the **median** across the 10 scored prompts. Extra tokens beyond the baseline length do not help: the client stops at `max_tokens`, and the TPS numerator is fixed to the baseline token count.

## Baseline: vLLM

The baseline is a pinned version of vLLM serving `Qwen2.5-72B-Instruct` on the same hardware as the challengers. See [this section](/docs/validators/overview#gpu-requirements) for minimum hardware requirements. Baseline results are **pre-computed and cached** per prompt set: the baseline runs once when prompts rotate (new block hash), and all challengers in that round are compared against the same cached baseline numbers.

vLLM is a strong baseline. It implements PagedAttention, continuous batching, FlashAttention, and other production optimizations. Beating it requires genuine innovation, not just a different framework.

## Explicitly not in V1

- **Speculative decoding** -- banned. Simplifies correctness checking.
- **Weight quantization** -- banned. All servers must load exact BF16 weights.
- **Alternate models** -- banned. Same model, same weights.
- **Approximate attention** -- banned if it changes outputs beyond the gate's tolerance.
- **Sampling-based evaluation** -- all eval is greedy (temperature=0).

These restrictions will be relaxed in V2 once the eval pipeline is proven.

## Reproducibility

Every scored run produces a `ScoreResult`:

```
ScoreResult
  ttft_improvement     float
  throughput_improvement float
  token_match_rate     float
  correctness_pass     bool
  score                float
  disqualified         bool
  disqualify_reason    str | None
```

Combined with the block hash that seeded the prompts and the pinned Docker image digest, anyone can replay the eval and verify the result.

## Emission scaling

Total emission allocated to the competition pool scales linearly with the leader's improvement score:

`comp_frac = min(1.0, leader_score / SCORE_EMISSION_TARGET)`

With the default target of 0.10 (10% improvement over baseline):

- A leader scoring 0.05 earns 50% of total emission for the pool
- A leader scoring 0.10 or above earns 100%
- The remainder goes to a burn UID

Within the competition pool, the leader receives 80% and the runner-up receives 20%. When there is no runner-up, the leader receives the full pool.
