The scoring algorithm uses a **correctness-gated speed model**. A strict correctness check acts as a primary filter; submissions that fail it receive a zero score. For valid submissions, the final score combines TTFT and throughput improvements against the vLLM baseline.

## Scoring formula

```python
if not correctness_pass:
    score = 0.0
else:
    ttft_imp = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
    tps_imp  = max(0, (miner_tps - baseline_tps) / baseline_tps)
    score = 0.5 * ttft_imp + 0.5 * tps_imp
```

<Callout title="Scoring axes" type="info">
Three axes, in order of enforcement:

1. **Correctness gate** -- fail this and nothing else matters.
2. **TTFT** -- 50% of the remaining score.
3. **Throughput** -- 50%.

Improvements are floored at zero. If your server is slower than vLLM on one axis, that axis contributes 0, not a negative number.

</Callout>

## Aggregation

Each evaluation round sends **10 prompts** to both baseline and miner. Per-prompt TTFT and throughput are computed, then aggregated using the **median** across prompts.

Why median: it is robust to outliers (a single GC pause or memory allocation spike does not tank your score) and prevents gaming by being fast on easy prompts but slow on hard ones.

## Single-pass evaluation

Speed and correctness are measured from the **same streaming request** (`stream=true, logprobs=true, top_logprobs=5`). Each prompt is sent once; the validator measures TTFT and throughput from SSE chunk timing while simultaneously collecting tokens and per-token logprobs for the correctness gate.

This eliminates the exploit surface of a two-pass design where a miner could detect which pass is running and return different outputs (fast garbage for speed, correct tokens for correctness).

Logprob serialization adds a small constant overhead per token. Because the baseline and miner both run with the same request parameters, this overhead cancels out in the relative improvement calculation.

## Warmup

Before the scored prompts, the validator sends **1-2 warmup prompts** that are discarded. This absorbs one-time costs: CUDA graph compilation, KV cache pre-allocation, GPU clock ramping. Both baseline and miner receive the same warmup. The competition measures steady-state serving speed, not startup optimization.

## Correctness gate

Correctness verifies that the miner serves the **exact same model with the same weights** using **greedy decoding** (`temperature=0`).

Because multi-GPU tensor parallelism introduces tiny numerical differences between runs, a strict global token-match rate falsely disqualifies legitimate miners. One near-tie flip at an early token causes the entire rest of the sequence to diverge (the cascade effect).

Instead, the gate uses **per-prompt first-divergence analysis**:

1. For each prompt, the validator walks the baseline and miner outputs token-by-token until the first mismatch.
2. If there is no mismatch, the prompt passes.
3. If there is a mismatch, the validator checks the logprobs at that position:
   - The baseline's token must appear in the miner's **top-5** logprobs
   - The logprob gap between the miner's chosen token and the baseline's token must be **`<= 0.5`**
   - The miner's chosen token must be its own **top-1** (greedy decoding verification)
4. If all three hold, the divergence is classified as tensor-parallel numerical noise and the prompt passes.

### Prompt-level pass rate

At least **9 out of 10 prompts** must pass. A single failed prompt is allowed as a safety margin; a wrong model or quantized weights will typically fail on many prompts.

### Why this catches bad actors

- **Quantized weights** (INT8/INT4/FP8): logprob shifts are typically 0.5-2.0+. The baseline token will not be in the top-5 with a small gap.
- **Modified architecture** (missing layers, different attention): distributions are completely different. Caught immediately.
- **Non-greedy decoding** (`temperature > 0`): the miner's chosen token will not be its own top-1.
- **Fabricated logprobs**: the miner does not know the baseline output (prompts are block-hash seeded). To fabricate correctly they would need to run the exact correct model, defeating any speed advantage.

### Why not full KL divergence

The OpenAI-compatible API returns top-N logprobs, not the full vocabulary distribution. True KL divergence requires the full ~152K-token distribution per position. Returning that through an HTTP API would be:

- ~600KB per token position at float32
- ~150MB per prompt at 256 generated tokens

This is impractical for an API benchmark and would itself distort speed measurements. First-divergence logprob analysis achieves the same goal with minimal overhead.

## TTFT improvement

```
ttft_improvement = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
```

Median across prompts. Floored at zero: if your server is slower than the baseline on TTFT, this axis contributes nothing to your score.

TTFT captures prefill efficiency: how fast the server processes the full input prompt and produces the first output token. At 32K input tokens on e.g. 4x H200, this is dominated by the attention computation during prefill.

### Why TTFT gets 50%

Time-to-first-token is the user-facing latency. In agentic workflows, tool calls and chain-of-thought steps are gated on TTFT. Optimizing prefill directly reduces perceived latency in production.

## Throughput improvement

```
throughput_improvement = max(0, (miner_tps - baseline_tps) / baseline_tps)
```

Median across prompts. Floored at zero.

Throughput captures decode efficiency: how fast the server generates output tokens once prefill is complete.

### Why throughput gets 50%

Throughput determines cost-per-token in production. Higher throughput means more requests served per GPU-hour. For long-form generation (summaries, code, analysis), throughput dominates total response time.

## Disqualification vs. low score

| Outcome          | Cause                                       | What happens                  |
| ---------------- | ------------------------------------------- | ----------------------------- |
| **Disqualified** | Correctness gate fails, timeout, crash, OOM | `score = 0`, not a challenger |
| **Low score**    | Passes correctness but doesn't beat king    | Recorded, not king            |
| **Wins**         | Passes correctness, beats king's score      | New king, takes all emission  |

## Baseline: vLLM

The baseline is a pinned version of vLLM serving `Qwen2.5-72B-Instruct` on the same hardware as the challengers. See [this section](/docs/validators/overview#gpu-requirements) for minimum hardware requirements. Baseline results are **pre-computed and cached** per prompt set: the baseline runs once when prompts rotate (new block hash), and all challengers in that round are compared against the same cached baseline numbers.

vLLM is a strong baseline. It implements PagedAttention, continuous batching, FlashAttention, and other production optimizations. Beating it requires genuine innovation, not just a different framework.

## Explicitly not in V1

- **Speculative decoding** -- banned. Simplifies correctness checking.
- **Weight quantization** -- banned. All servers must load exact BF16 weights.
- **Alternate models** -- banned. Same model, same weights.
- **Approximate attention** -- banned if it changes outputs beyond the gate's tolerance.
- **Sampling-based evaluation** -- all eval is greedy (temperature=0).

These restrictions will be relaxed in V2 once the eval pipeline is proven.

## Reproducibility

Every scored run produces a `ScoreResult`:

```
ScoreResult
  ttft_improvement     float
  throughput_improvement float
  token_match_rate     float
  correctness_pass     bool
  score                float
  disqualified         bool
  disqualify_reason    str | None
```

Combined with the block hash that seeded the prompts and the pinned Docker image digest, anyone can replay the eval and verify the result.
