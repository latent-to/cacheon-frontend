The scoring algorithm uses a **correctness-gated speed model**. A strict correctness check acts as a primary filter; submissions that fail it receive a zero score. For valid submissions, the final score combines TTFT and throughput improvements against the vLLM baseline.

## Scoring formula

```python
if not correctness_pass:
    score = 0.0
else:
    ttft_imp = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
    tps_imp  = max(0, (miner_tps - baseline_tps) / baseline_tps)
    score = 0.5 * ttft_imp + 0.5 * tps_imp
```

<Callout title="Scoring axes" type="info">
Three axes, in order of enforcement:

1. **Correctness gate** -- fail this and nothing else matters.
2. **TTFT** -- 50% of the remaining score.
3. **Throughput** -- 50%.

Improvements are floored at zero. If your server is slower than vLLM on one axis, that axis contributes 0, not a negative number.

</Callout>

## Aggregation

Each evaluation round sends **10 prompts** to both baseline and miner. Per-prompt TTFT and throughput are computed, then aggregated using the **median** across prompts.

Why median: it is robust to outliers (a single GC pause or memory allocation spike does not tank your score) and prevents gaming by being fast on easy prompts but slow on hard ones.

## Two-pass evaluation

Speed and correctness are measured in separate passes to prevent logprob serialization overhead from distorting speed measurements.

### Pass 1: Speed

Prompts are sent with `stream=true` and no logprobs requested. The validator measures:

- **TTFT**: wall-clock time from HTTP request send to first SSE data chunk
- **Throughput**: total output tokens divided by wall-clock time from first token to last token

Both baseline (vLLM) and miner are measured identically.

### Pass 2: Correctness

Same prompts are sent with `stream=false` and `logprobs=5`. The validator collects full output tokens and per-token logprobs. This pass is not timed.

## Warmup

Before the scored prompts, the validator sends **1-2 warmup prompts** that are discarded. This absorbs one-time costs: CUDA graph compilation, KV cache pre-allocation, GPU clock ramping. Both baseline and miner receive the same warmup. The competition measures steady-state serving speed, not startup optimization.

## Correctness gate

Correctness is enforced via greedy token matching with logprob spot-checks. Both the baseline and the miner run with `temperature=0` (greedy decoding) on identical model weights.

### Token match

```
token_match_rate = matching_tokens / total_tokens
```

Across all prompts. Must be >= 99%.

V1 bans quantization and speculative decoding. With identical BF16 weights, greedy decoding, and temperature=0, token divergence should be extremely rare. Different attention kernel implementations can occasionally flip the argmax at positions where the top-2 logprobs are very close, but this should affect fewer than 1% of tokens. The 99% threshold catches any real divergence while allowing the occasional numerical edge case.

### Logprob verification

At positions where the miner's output token differs from the baseline's:

1. The baseline's chosen token must appear in the miner's **top-2** logprobs
2. The logprob gap between the miner's top-1 and the baseline's token must be **< 0.05**

If either check fails at any position, the submission is DQ'd.

This catches:

- Modified model weights (would show large logprob divergence)
- Truncated models (missing layers change the distribution)
- Wrong sampling (non-greedy decoding)
- Fabricated outputs (random tokens won't have correct logprob structure)

### Why token match, not full KL divergence

The OpenAI-compatible API returns top-N logprobs, not the full vocabulary distribution. True KL divergence requires the full ~152K-token distribution per position. Returning that through an HTTP API would be:

- ~600KB per token position at float32
- ~150MB per prompt at 256 generated tokens

This is impractical for an API benchmark and would itself distort speed measurements. Greedy token match + logprob spot-checks achieves the same goal with minimal overhead.

## TTFT improvement

```
ttft_improvement = max(0, (baseline_ttft - miner_ttft) / baseline_ttft)
```

Median across prompts. Floored at zero: if your server is slower than the baseline on TTFT, this axis contributes nothing to your score.

TTFT captures prefill efficiency: how fast the server processes the full input prompt and produces the first output token. At 32K input tokens on 4x H200, this is dominated by the attention computation during prefill.

### Why TTFT gets 50%

Time-to-first-token is the user-facing latency. In agentic workflows, tool calls and chain-of-thought steps are gated on TTFT. Optimizing prefill directly reduces perceived latency in production.

## Throughput improvement

```
throughput_improvement = max(0, (miner_tps - baseline_tps) / baseline_tps)
```

Median across prompts. Floored at zero.

Throughput captures decode efficiency: how fast the server generates output tokens once prefill is complete.

### Why throughput gets 50%

Throughput determines cost-per-token in production. Higher throughput means more requests served per GPU-hour. For long-form generation (summaries, code, analysis), throughput dominates total response time.

## Disqualification vs. low score

| Outcome          | Cause                                                       | What happens                  |
| ---------------- | ----------------------------------------------------------- | ----------------------------- |
| **Disqualified** | Token match < 99%, logprob check fails, timeout, crash, OOM | `score = 0`, not a challenger |
| **Low score**    | Passes correctness but doesn't beat king                    | Recorded, not king            |
| **Wins**         | Passes correctness, beats king's score                      | New king, takes all emission  |

## Baseline: vLLM

The baseline is a pinned version of vLLM serving `Qwen2.5-72B-Instruct` on the same 4x H200 hardware. Baseline results are **pre-computed and cached** per prompt set: the baseline runs once when prompts rotate (new block hash), and all challengers in that round are compared against the same cached baseline numbers.

vLLM is a strong baseline. It implements PagedAttention, continuous batching, FlashAttention, and other production optimizations. Beating it requires genuine innovation, not just a different framework.

## Explicitly not in V1

- **Speculative decoding** -- banned. Simplifies correctness checking.
- **Weight quantization** -- banned. All servers must load exact BF16 weights.
- **Alternate models** -- banned. Same model, same weights.
- **Approximate attention** -- banned if it changes outputs beyond the 99% threshold.
- **Sampling-based evaluation** -- all eval is greedy (temperature=0).

These restrictions will be relaxed in V2 once the eval pipeline is proven.

## Reproducibility

Every scored run produces a `ScoreResult`:

```
ScoreResult
  ttft_improvement     float
  throughput_improvement float
  token_match_rate     float
  correctness_pass     bool
  score                float
  disqualified         bool
  disqualify_reason    str | None
```

Combined with the block hash that seeded the prompts and the pinned Docker image digest, anyone can replay the eval and verify the result.
