---
title: Scoring
description: Two-launch evaluation. A miner kernel scores only if it is both faithful (per-token KL and benchmark accuracy against the stock run) and genuinely faster (bracketed, noise-derived throughput). Fail fidelity and the score is zero.
---

Optima scores one kernel at a time. The validator runs a fixed, public model in pinned **sglang** **twice** with everything held equal (same weights, seed, sampler, and prompts) and the **only** difference the one op the miner replaced: a **baseline** launch with stock kernels and a **candidate** launch with the miner kernel swapped in. It then asks two questions of the candidate:

1. **Fidelity** — did the model still produce the same outputs?
2. **Throughput** — did the kernel make the model faster?

**A kernel earns only if it is both faithful and faster.** A kernel that is 30% faster but changes the model's answers scores **0**. A faithful kernel that is slower than stock also earns nothing (no speedup). The whole game is a genuine speedup at equal quality.

<Callout title="Reality check" type="warn">
As of this writing, **no submitted kernel has beaten sglang.** sglang's built-in kernels are already heavily tuned; most faithful kernels are slower than the baseline. This validates the referee, not any optimization. Winning is a real performance-engineering result, not a faucet.
</Callout>

## The gate ladder

A submission passes through three gates in order. Fail any gate and the score is 0. Pass all three and the score is the measured speedup.

| Gate | What it checks | Cost | Where it runs |
| --- | --- | --- | --- |
| **A — op-correctness** | Is this even the right function? | Cheap, local | Deterministic inputs vs a high-precision reference (`optima verify`) |
| **B — fidelity** | Does the model still produce the same outputs? | Full model | Per-token KL + benchmark accuracy on the real model |
| **C — throughput** | Is the speedup real (not noise)? | Full model | Bracketed two-launch timing under CUDA graphs |

### Gate A — op-correctness (sanity check, not the anti-cheat)

The validator feeds the kernel deterministic inputs and compares its output to a **high-precision reference** with an allclose-style tolerance. This is a cheap local filter that rejects a kernel computing the wrong thing (for example SiLU where the model wants SwiGLU) before the expensive end-to-end run. It is explicitly a sanity check, **not** the load-bearing anti-cheat: a kernel can be op-correct and still drift the model end to end. Miners run it themselves with `optima verify`.

For kernels that legitimately change numerics (flash-style softmax reordering, FP8, MLA weight absorption), the comparison is a **matched-ratio** or **cosine** bound against high-precision ground truth rather than bit-exact allclose. The reference is always high-precision ground truth, never the stock kernel.

### Gate B — fidelity (the load-bearing anti-cheat)

With the kernel swapped into the live model, the validator compares the candidate's **per-token output distribution** to the baseline. When `return_logprob=True`, sglang returns the top-k `(logprob, token_id)` per generated position for both launches; the validator diffs them.

| Check | What it catches | Default threshold |
| --- | --- | --- |
| **mean per-token KL** | the output distribution drifting | `5e-3` (most slots); `3e-2` for attention |
| **argmax-disagreement rate** | a sparse cheat: bit-exact almost everywhere but a few tokens flipped | `~1%` |
| **coverage / tail-mass** | a flattened distribution that fools top-k KL by matching only the head | loose by default |
| **benchmark accuracy** | the model getting dumber (GSM8K / MMLU) | no regression beyond a tolerance (`~2` points) |

The KL threshold is **per-slot** and is calibrated to the model's own nondeterminism floor: running the *same* stock kernel twice is not bitwise identical, so the gate must sit above that floor. Attention's flash-style softmax reorders arithmetic, so its intrinsic floor is higher, which is why `attention.sdpa` and `attention.decode` gate at `3e-2` rather than `5e-3`.

The alignment is subtle. Greedy decoding means the baseline and candidate token sequences can *diverge* once a kernel flips an argmax, and positions after a divergence are no longer comparable (different context). So the validator compares position *i*, then stops if token *i* differs, but it always compares position 0 first (identical prompt context), so a kernel that derails the very first token still receives a large KL instead of "zero comparable positions."

<Callout title="Honest limitation" type="info">
Top-k truncation means this KL **approximates** the true full-vocab KL. It is sharp enough to catch the cheats that matter (calibration collapse, dropped work) at k ≥ 20, but the tightest gate would capture full logits at a reference seam; that is roadmap. Benchmark accuracy is noisy at small n, which is exactly why per-token KL is the dense **primary** gate and benchmark accuracy is a capability **floor**. See [Prompts](/docs/evaluation/prompts) for the benchmark path.
</Callout>

Because the objective is a scalar (throughput), the benchmarks are **AND-gates**, not score components: there is nothing to aggregate. A faithful kernel preserves both KL and accuracy; a kernel that secretly degrades the model blows up KL and/or drops accuracy and scores zero even if it looked fast. On a validated run, a broken silu dropped GSM8K from 62.5% to 0% while being 26% *faster* and correctly scored 0.

### Gate C — throughput (is the speedup real?)

This is **CUDA-graphs-ON** scoring. Graphs-off cripples the baseline (measured ~4.5-6.5x), so it is never used. A block or collective kernel must declare `graph_safe: true` in its metadata to run *inside* the graph; without it, the kernel falls back to the baseline in-graph and cannot win.

The validator runs on rented pods where it cannot lock GPU clocks, so it does not trust a single timing. Instead it **brackets** the candidate between two baseline launches (`B, C, B'`), pairs the candidate's throughput against the *mean* of the two baseline reads, and derives the required speedup from the *measured* baseline noise:

```python
speedup  = candidate_read / mean(baseline_reads)       # paired against the bracket
noise    = relative_spread(baseline_reads)             # measured, not assumed
required = 1.0 + max(min_margin, k * noise)            # min_margin = 0.02, k = 2.0
```

If the two bracketing baselines disagree by more than `max_noise` (default `10%`), the box was too unstable this round to trust: the verdict is **NO-DECISION** (`confident=False`), the round is discarded, and nobody is crowned on a number that cannot be reproduced. Token budgets are held equal (`ignore_eos` on), so a kernel cannot win by emitting fewer tokens, and the throughput numerator is known to the driver a priori rather than reported by the scheduler. Timing uses `time.perf_counter` in the driver with `torch.cuda.synchronize()` on both sides, so async work is fully counted and out-of-process (see [Harness](/docs/evaluation/harness)).

## The score

```python
passed_fidelity = mean_kl <= kl_threshold and num_positions > 0   # per-slot; 5e-3 default
passed_speedup  = confident and speedup >= required               # bracketed, noise-derived

score = (speedup if (passed_fidelity and passed_speedup)          # faithful AND faster
         else 0.0 if not passed_fidelity                          # cheat -> 0
         else speedup)                                            # faithful but slow -> ~1.0, no title
```

Read carefully, this encodes the whole philosophy:

- **Fail fidelity → score 0**, no matter how fast. You cannot trade correctness for speed.
- **Pass fidelity but below the speedup margin → score ~1.0**, so it takes no title but is not punished.
- **Pass both → score is the speedup.**

## King-of-the-hill

The current best kernel for a slot is the **champion** and holds the emission. A challenger takes the title only if it beats the champion by the margin:

```
challenger_score >= champion_score * (1 + margin)     # margin = 0.02
```

A bit-identical copy *ties* the champion, never clears the margin, and so earns **0** (this is what makes copying pointless; see [Data contracts](/docs/evaluation/data-contracts) for commit-reveal and copy detection). There is **one champion per slot**. With `optima settle --per-slot`, emission is split across the slots that have a champion, so a miner can win by *specializing* in one slot rather than beating everyone everywhere. Winner-take-all per slot is the baseline policy; production is expected to smooth it (bounty the most recent distinct improver, decay former champions).

A champion's frozen score is a speedup measured against the stock kernels of the sglang pin it was crowned under. After a `PINNED_SGLANG` bump the stock baseline changes, so that frozen number is no longer comparable; the round is settled against *current* stock and the stale champion is flagged for re-baseline.

## Reproducibility

Every settled round is driven by the same fixed model, the pinned sglang version, a block-hash-derived prompt seed (see [Prompts](/docs/evaluation/prompts)), and the two-launch protocol. Given the bundle and that seed, an independent validator draws the identical prompts and can replay the evaluation. Cross-validator determinism (locked clocks, pinned hardware and driver, deterministic mode) is the remaining calibration work that lets Bittensor consensus catch an outlier score.
