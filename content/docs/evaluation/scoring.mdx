---
title: Scoring
description: One-pass evaluation. Speed (end-to-end wall time) and correctness (teacher-forcing) use the same prompts and miner outputs. Final metrics use median aggregation across 10 scored prompts.
---

Final score is **correctness-gated**: the same miner outputs that earn speed credit must pass teacher-forcing on the scoring vLLM. A cheap token-match pre-filter can short-circuit hopeless submissions before scoring. Otherwise `score = 0.0` when either gate fails.

## One-pass overview

| Stage           | Prompts                | Context / output         | Container                                                     | Purpose                                             |
| --------------- | ---------------------- | ------------------------ | ------------------------------------------------------------- | --------------------------------------------------- |
| **Speed**       | 10 scored + 2 warmup   | ~**6k** in / **512** out | Miner + cached generation baseline (`vLLM v0.22.0`)           | End-to-end time vs baseline; token-match pre-filter |
| **Correctness** | Same 10 scored outputs | Same text                | Separate scoring vLLM (`v0.9.2`) teacher-forces miner outputs | Correctness gate                                    |

Baseline numbers are **pre-computed once per block hash** and cached. After all miners finish generation, one scoring container runs `prompt_logprobs` over the **same outputs** used for speed in a single batch.

## Scoring formula

```python
if pass1_match_fail or not correctness_pass:
    score = 0.0
else:
    # Per prompt: wall time from request start to aligned k-th token
    k = min(baseline_N, miner_N)
    require k >= max(2, ceil(0.9 * baseline_N))  # tolerance band
    miner_e2e = miner_ttft + miner_decode[k - 1]
    baseline_e2e = baseline_ttft + baseline_decode[k - 1]
    improvement = max(0, (baseline_e2e - miner_e2e) / baseline_e2e)
    speed_improvement = median(improvement across 10 scored prompts)
    score = speed_improvement
```

<Callout title="Scoring axes" type="info">
Gates and axes, in order of enforcement:

1. **Token-match pre-filter (Speed gate):** if aggregate token match below 10%, return `pass1_match_fail`; skip teacher-forcing.
2. **Correctness gate:** if teacher-forcing fails, return `correctness_fail` or `scoring_infra_fail`; speed zeroed.
3. **End-to-end speed:** single axis, median relative speedup vs baseline wall time to the aligned token count (same 10 scored prompts).

Improvements are floored at zero. Slower-than-baseline miners score 0, not negative.

</Callout>

## Speed: end-to-end wall time and token match

### Aggregation

The harness sends **12 eval prompts** (2 warmup discarded, **10 scored**) to each miner. Speed uses the **median** end-to-end improvement across those 10 scored prompts.

### Measurement

Eval prompts use **streaming chat completions** with `logprobs` (`stream=true`, `logprobs=true`, `top_logprobs=5`). The validator records request start, time to first token (TTFT), and per-token arrival times. **Score uses total wall time** from request start to the aligned k-th token, not decode-only throughput. This is buffering-invariant: a miner cannot gain credit by generating slowly and flushing tokens in one burst.

Per prompt:

```
k = min(baseline_output_tokens, miner_output_tokens)
miner_e2e   = miner_ttft + miner_decode[k - 1]
baseline_e2e = baseline_ttft + baseline_decode[k - 1]
```

`decode[i]` is seconds from first token to token i. TTFT plus decode elapsed equals request start to that token.

**Tolerance band:** the miner must emit at least `ceil(0.9 * baseline_N)` tokens (minimum 2). Otherwise that prompt gets 0 speed credit. Extra tokens beyond baseline length do not help.

Top-5 logprobs are collected and cached with the baseline but are **not** a DQ gate in V1.

### Warmup

Before sending the scored prompts, the validator sends **2 warmup prompts** that are discarded. Both baseline and miner receive the same warmup. The competition measures steady-state serving speed, not startup optimization.

### Token-match pre-filter

Aggregate match is the **mean of per-prompt token match rates** across the 10 scored prompts (not a single pooled token average).

If aggregate match is below `PASS1_MATCH_DQ_THRESHOLD` (default **10%**), the miner is disqualified with `pass1_match_fail` and teacher-forcing is skipped. Configurable via `CACHEON_PASS1_MATCH_DQ_THRESHOLD` in [validator config](https://github.com/latent-to/cacheon/blob/main/validator/config.py).

`token_match_rate` on the evaluation record stores this aggregate value for dashboards.

## Correctness (teacher-forcing)

After all miners finish generation, the validator starts one **scoring vLLM** container (`CACHEON_SCORING_IMAGE`, default `vllm/vllm-openai:v0.9.2`) and teacher-forces every queued output in a single batch. It boots with `--max-model-len` sized for eval prompts (~7k), not the full 32k generation cap. Startup timeout is **600 seconds** (10 minutes).

Correctness is **text-canonical and cross-engine safe**. The scoring vLLM tokenizer only is used; challenger stream token counts are never compared for DQ.

### Terminology

| Term                                | Source                         | Meaning                                                                                       |
| ----------------------------------- | ------------------------------ | --------------------------------------------------------------------------------------------- |
| **`challenger_output_text`**        | Challenger SSE                 | Exact UTF-8 text reconstructed from the challenger's stream (source of truth for scoring)     |
| **`challenger_stream_token_count`** | Challenger SSE                 | Count of token strings from `logprobs.content[].token` during streaming (telemetry only)      |
| **`scoring_canonical_token_count`** | Scoring vLLM `/tokenize`       | Assistant positions after tokenizing `messages + challenger_output_text`, minus prompt prefix |
| **`scoring_logprob_count`**         | Scoring vLLM `prompt_logprobs` | Teacher-forcing logprobs extracted for assistant positions (M)                                |
| **`scoring_scored_token_count`**    | Validator                      | `min(N, M)` logprobs used for mean/min correctness thresholds                                 |

### `challenger_output_text` reconstruction

When the challenger streams with logprobs:

1. **Primary:** concatenate `delta.content` chunks (OpenAI-compatible).
2. **Fallback:** if logprobs were streamed but content deltas are empty, join streamed token strings.
3. Do not normalize whitespace. The exact bytes are passed to the scoring vLLM.

### Correctness gate

The scoring container teacher-forces each of the **10 scored outputs** the challenger already generated (`prompt_logprobs=1` on `/v1/chat/completions` with prompt + assistant text). For each prompt, all extracted scoring logprobs, up to min(`scoring_canonical_token_count`, `scoring_logprob_count`), must satisfy:

- **Mean logprob** must be at least ≥ `MEAN_LOGPROB_THRESHOLD` (default `-4.0`)
- The minimum logprob (**min logprob**) for any individual token must be ≥ `MIN_LOGPROB_THRESHOLD` (default `-12.0`)

These thresholds are configurable through `CACHEON_MEAN_LOGPROB_THRESHOLD` and `CACHEON_MIN_LOGPROB_THRESHOLD`.

**All scored prompts must pass.** A single failure disqualifies the entire submission with `correctness_fail`. Empty logprobs (for example from OOMs or HTTP 500 errors), or near-empty extraction below 50% of `scoring_canonical_token_count`, result in `scoring_infra_fail`.

Large differences between `challenger_stream_token_count` and `scoring_canonical_token_count` are logged as telemetry (`possible_stream_spam`, `stream_canonical_drift`) but do not trigger disqualification in V1.

### Why this catches bad actors

- **Fast garbage on speed prompts**: fails teacher-forcing on the same output (no separate short audit set to fingerprint).
- **Wrong model weights**: scoring vLLM assigns low probability to implausible miner tokens.
- **Garbage after a valid prefix**: low mean/min logprob on garbage tokens.

### Scoring baseline image

Generation uses the pinned baseline (`vllm/vllm-openai:v0.22.0`). Teacher-forcing uses **`vllm/vllm-openai:v0.9.2`** because newer vLLM builds OOM on long `prompt_logprobs`; Eval prompts stay at ~6k input and the scoring server context cap matches that budget (~7168 tokens).

## Disqualification reason codes

When `disqualified` is true, the evaluation record's `disqualify_reason` field explains why. Format is usually `prefix: details`. Visible on the dashboard (Rounds, Evaluations) and via `GET /api/evaluations/{uid}`.

### Correctness detail codes

Failures include `prompt {N}:` before the detail. These come from teacher-forcing on **scoring canonical tokens** (not challenger stream token count).

| Detail pattern                                                                              | Wrapped as           | Meaning                                                                                                                                                                                                                        | What to do                                                                                                      |
| ------------------------------------------------------------------------------------------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------- |
| `scoring_infra_fail: no scoring logprobs available`                                         | `scoring_infra_fail` | Scoring vLLM returned no extractable logprobs (empty response, tokenize failure, or HTTP error).                                                                                                                               | Check `baseline_scoring_{block}_{timestamp}.log` and matching `gpu_eval_{timestamp}.log`. Validator-side issue. |
| `scoring_infra_fail: extracted {N}/{M} scoring canonical tokens (below 50% coverage floor)` | `scoring_infra_fail` | Scoring vLLM returned too few logprobs vs `/tokenize` count on the **same engine** (partial response, OOM). `{M}` is `scoring_canonical_token_count`; `{N}` is `scoring_logprob_count`. Small suffix gaps are scored normally. | Validator-side issue. Check `baseline_scoring_{block}_{timestamp}.log`.                                         |
| `min_logprob {X} below threshold {Y}`                                                       | `correctness_fail`   | At least one token scored below the per-token floor. Default threshold `Y` is `-12.0`.                                                                                                                                         | Inspect whether specific tokens are corrupted or injected.                                                      |

GPU evaluation logs print counts per prompt:

```
prompt 0: correctness=FAIL mean_lp=-1.200 min_lp=-3.400 challenger_stream_token_count=162 scoring_canonical_token_count=161 scoring_logprob_count=161 scoring_scored_token_count=161
```

`challenger_stream_token_count` is telemetry only; it is **not** used for DQ.

## End-to-end speed improvement

```
speed_improvement = median(max(0, (baseline_e2e - miner_e2e) / baseline_e2e))
```

Computed across 10 scored prompts (~6k input context). `score` equals `speed_improvement` when the miner passes both gates.

The validator may log `buffered stream suspected` when TTFT is much slower than baseline but the decode window is near zero (telemetry only, not a separate DQ).

## Baseline: vLLM

The generation baseline is a pinned vLLM instance serving `Qwen2.5-72B-Instruct` on the same hardware as challengers. See [GPU requirements](/docs/validators/overview#gpu-requirements). Baseline results are **pre-computed and cached** per block hash; all challengers in the round share the same cached numbers.

## Hard requirements

- **Alternate models**: same model and canonical weights from `/models`.
- **Greedy eval**: all eval requests use `temperature=0`.

## Reproducibility

Every scored run produces an `EvaluationRecord` with `score`, `speed_improvement`, `token_match_rate`, `disqualified`, and `disqualify_reason`. Combined with the block hash that seeded prompts and pinned Docker digests, anyone can replay the eval and verify the result.

## Emission scaling

Total emission allocated to the competition pool scales linearly with the leader's improvement score:

`comp_frac = min(1.0, leader_score / SCORE_EMISSION_TARGET)`

With the default target of 0.10 (10% improvement over baseline):

- A leader scoring 0.05 earns 50% of total emission for the pool
- A leader scoring 0.10 or above earns 100%
- The remainder goes to a burn UID

Within the competition pool, the leader receives 80% and the runner-up receives 20%. When there is no runner-up, the leader receives the full pool.
