Prompts are **instruct-style requests** built from curated instruction templates filled with long-context passages from [**PG19** (Project Gutenberg books)](https://www.tensorflow.org/datasets/catalog/pg19). A **block-hash-derived seed** makes them deterministic: same block, same prompts, anywhere, any machine.

## Why instruct format

The target model is `Qwen2.5-72B-Instruct` and the long-term positioning is "fastest backend for agentic workflows." Benchmarking with raw text continuation does not reflect real serving workloads. Instruct-format prompts test what production deployments actually send.

## Prompt design

Each prompt combines two pieces:

1. **Instruction template**: one of 20 curated literary-analysis tasks
2. **Context passage**: a long slice of text sampled from a PG19 book

Both are selected deterministically via the block-hash seed. The template determines the output shape; the PG19 passage provides the input context.

All templates follow the pattern `<instruction>\n\n{context}`. The full list is in the `TEMPLATES` constant in [`validator/prompts.py`](https://github.com/latent-to/cacheon/blob/main/validator/prompts.py).

Because all templates operate on literary passage analysis, they produce **structured, grounded outputs** from fixed input. This keeps greedy decoding stable and the 99%+ token match gate meaningful.

## Why PG19 for context

Inference benchmarking needs long, continuous sequences that stress prefill, attention, and KV cache. Short documents produce trivially fast responses where optimization differences are lost in noise.

| Dataset          | Typical doc length             | Why rejected or chosen                                  |
| ---------------- | ------------------------------ | ------------------------------------------------------- |
| FineWeb          | Median &lt;1K tokens           | Too short: prefill is trivial, no differentiation       |
| MMLU / HellaSwag | ~100-500 tokens                | Evaluation benchmarks, not serving workloads            |
| **PG19**         | **100K+ tokens (full novels)** | **Chosen**: long-form text forces real serving at scale |

PG19 is pinned to a specific Hugging Face dataset revision (`a0e01428956e39868fa36ccfa0ee236ff04e6a6b`). Row ordering is stable across upstream updates.

**PG19 must be pre-downloaded** to the local HuggingFace cache before starting the validator. If the dataset is missing, the validator raises an error at startup with a copy-paste download command rather than attempting an 11 GB download mid-evaluation.

## Why PG19 is hard to game

PG19 has 28,000+ full books. Each prompt pairs a long slice from a random book with one of 20 templates. The search space is enormous (many books, many offsets, many templates), so pre-caching answers for every possible prompt is impractical.

Fixed public benchmarks like **LongBench** are different: the prompt set is small and enumerable. A miner could download it, generate offline, and cache outputs. PG19 plus templates expands the pool enough that enumeration stops being the easy path.

## Passage quality filters

Not all PG19 rows are usable. OCR artifacts, chapter headers, and whitespace-heavy rows are rejected before a passage is accepted:

| Filter                | Threshold    | Rejects                           |
| --------------------- | ------------ | --------------------------------- |
| Minimum length        | 16,000 chars | Rows too short to stress prefill  |
| Alpha character ratio | ≥ 50%        | OCR noise, numeric tables         |
| Whitespace ratio      | ≤ 35%        | Chapter headers, blank-heavy rows |

The sampler retries up to 1,000 times before raising an error. Valid PG19 data should never exhaust this limit in practice.

## Parameter defaults

| Parameter               | Value         | Notes                                                                                                        |
| ----------------------- | ------------- | ------------------------------------------------------------------------------------------------------------ |
| `n` (prompts/round)     | 10            | Balance between statistical signal and eval time                                                             |
| `MAX_CONTEXT_CHARS`     | 131,072       | Caps passage length; long enough for prefill to be a real bottleneck                                         |
| `MIN_CONTEXT_CHARS`     | 16,000        | Every prompt stresses prefill                                                                                |
| `max_tokens`            | 256           | Output length per prompt                                                                                     |
| `MAX_SAMPLE_ATTEMPTS`   | 1,000         | Bounded retries on quality filter failures                                                                   |
| `PROMPT_ENGINE_VERSION` | 1             | Included in the baseline cache key; bump when templates or sampling logic change to force cache invalidation |
| dataset `revision`      | `a0e01428...` | Pinned commit SHA for stable row ordering                                                                    |

## Open-ended prompts and the correctness gate

At `temperature=0` the model is deterministic for fixed weights and input. The risk is not "creative" output, but **numerical ties**: where top-2 token log-probabilities are very close, different kernels or accumulation order can flip the argmax. That is what the 99%+ token match floor and **logprob sanity check** are for: at a mismatch, the baseline token must appear in the miner's top-2 with a tight logprob gap (see [Scoring](/docs/evaluation/scoring)).

Template diversity is kept because real workloads include varied request shapes and diversity makes the benchmark more representative than only rigid extraction-style tasks.
