Prompts are sampled from [**PG19** (Project Gutenberg books)](https://www.tensorflow.org/datasets/catalog/pg19) with a **block-hash-derived seed**. Same block, same prompts — anywhere, any machine.

## Why PG19

KV-cache benchmarking needs long, continuous sequences that force the cache to grow. Short documents produce trivially small caches where memory reduction is lost in noise.

| Dataset                          | Typical doc length             | Why rejected or chosen                                                        |
| -------------------------------- | ------------------------------ | ----------------------------------------------------------------------------- |
| FineWeb                          | Median &lt;1K tokens           | Too short — cache never gets large enough for signal                          |
| MMLU / HellaSwag                 | ~100–500 tokens                | Evaluation benchmarks, not continuation prompts                               |
| Synthetic (infilling, retrieval) | Variable                       | Deferred to [Cacheon V2+](../docs/roadmap#v2--expanded-optimization-surface); want real text for V1                                        |
| **PG19**                         | **100K+ tokens (full novels)** | **Chosen** — long-form text pushes cache to ~12% of peak memory at 32K tokens |

PG19 is pinned to a specific Hugging Face dataset revision. Row ordering is stable regardless of upstream changes — eval rounds from two years ago can be reproduced exactly.

## Seeding from block hash

```python
import hashlib, random

def derive_seed(block_hash: str) -> int:
    # block_hash is the hex string from the Bittensor chain
    h = hashlib.sha256(block_hash.encode()).digest()
    return int.from_bytes(h[:8], "big")

def sample_prompts(block_hash: str, n: int, *, max_chars=131072, min_chars=1000):
    seed = derive_seed(block_hash)
    rng = random.Random(seed)
    indices = list(range(len(PG19_ROWS)))
    rng.shuffle(indices)
    out = []
    for idx in indices:
        passage = PG19_ROWS[idx]
        if len(passage) < min_chars:
            continue
        out.append(_truncate_at_boundary(passage, max_chars))
        if len(out) == n:
            return out
```

Anyone with the block hash can reconstruct the exact prompt set used in that eval round.

## Parameter defaults

| Parameter   | Value             | Why                                                                                                                                       |
| ----------- | ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `max_chars` | 131,072           | ~32K tokens. At this length the Qwen2.5-7B KV cache is ~1.8 GB (~12% of peak memory) — large enough for the memory axis to produce signal |
| `min_chars` | 1,000             | Filters PG19 frontmatter, chapter headers, OCR artifacts                                                                                  |
| `revision`  | pinned commit SHA | Row ordering stable across upstream dataset updates                                                                                       |

Passages are truncated at **sentence boundaries first, word boundaries as fallback** — avoiding mid-word cuts that waste tokens.

## Why long context matters for scoring

Qwen2.5-7B-Instruct uses **grouped-query attention (GQA)** with `num_kv_heads=4` vs. `num_attention_heads=28`. The KV cache is already 7× smaller than a full MHA model. That means at short context, there's almost nothing to optimize:

| Context length | KV cache size | % of peak memory | Memory axis signal |
| -------------- | ------------- | ---------------- | ------------------ |
| 2K tokens      | ~113 MB       | ~0.7%            | Invisible          |
| 8K tokens      | ~450 MB       | ~3%              | Marginal           |
| **32K tokens** | **~1.8 GB**   | **~12%**         | **Clear**          |

The `max_chars` default is tuned for this regime. Submitting at shorter context would mean the memory axis contributes nothing to your score.

## Decode length

The harness generates **256 tokens** per prompt after prefill. This is enough time for eviction and compression policies to diverge from baseline — aggressive compression usually degrades gradually over many decode steps, not instantly.

At 256 tokens × N prompts × 28 layers, `policy.write` is called several thousand times per evaluation. Your policy code runs a lot; micro-optimizations matter.

## V2 direction

If the memory axis becomes noisy after launch, the upgrade path is **Qwen2.5-14B**, not a new dataset. The 14B model's KV cache is ~5× larger per token and still fits on one H100. The harness is already parameterized by `model_name` — swap is one constant.

Additional prompt datasets (infilling, multi-turn, RAG-style) come in V2+ once V1 is running reliably.
