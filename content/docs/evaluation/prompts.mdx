Prompts are **instruct-style requests** built from curated instruction templates filled with [**PG19** (Project Gutenberg)](https://www.tensorflow.org/datasets/catalog/pg19) passages. A **block-hash-derived seed** makes them deterministic: same block, same prompts, anywhere, any machine.

## Eval prompts

One prompt set is used for both speed (vs baseline) and correctness (teacher-forcing on the same miner outputs).

| Parameter              | Value                     |
| ---------------------- | ------------------------- |
| Scored prompts / round | 10 (+ 2 warmup discarded) |
| Input context          | ~6,144 tokens             |
| `max_tokens`           | 512                       |

Every challenger in the round is evaluated against the same workload.

## Why instruct format

The target model is `Qwen2.5-72B-Instruct`. Instruct-format prompts reflect real serving workloads rather than raw text continuation.

## Prompt design

Each prompt combines:

1. **Instruction template**: one of 20 curated literary-analysis tasks
2. **Context passage**: a slice of text sampled from a PG19 book

Both are selected deterministically via the block-hash seed. Templates live in [`validator/prompts.py`](https://github.com/latent-to/cacheon/blob/main/validator/prompts.py).

## Why PG19 for context

Long, continuous sequences stress prefill, attention, and KV cache. Short documents produce trivially fast responses where optimization differences are lost in noise.

| Dataset          | Typical doc length             | Why rejected or chosen                                  |
| ---------------- | ------------------------------ | ------------------------------------------------------- |
| FineWeb          | Median &lt;1K tokens           | Too short: prefill is trivial, no differentiation       |
| MMLU / HellaSwag | ~100-500 tokens                | Evaluation benchmarks, not serving workloads            |
| **PG19**         | **100K+ tokens (full novels)** | **Chosen**: long-form text forces real serving at scale |

PG19 is pinned to Hugging Face revision `a0e01428956e39868fa36ccfa0ee236ff04e6a6b`.

<Callout title="PG19 Setup Requirement" type="info">
  **PG19 must be pre-downloaded** to the local HuggingFace cache before starting the validator. If
  the dataset is missing, the validator raises an error at startup with a copy-paste download
  command rather than attempting an 11 GB download mid-evaluation.
</Callout>

## Passage quality filters

| Filter                | Threshold   | Rejects                            |
| --------------------- | ----------- | ---------------------------------- |
| Minimum length        | 4,000 chars | Rows too short for the eval budget |
| Alpha character ratio | ≥ 50%       | OCR noise, numeric tables          |
| Whitespace ratio      | ≤ 35%       | Chapter headers, blank-heavy rows  |

The sampler retries up to 1,000 times before raising an error.

Passage length uses a conservative chars/token estimate plus overhead and safety margin. Before each request, the validator also calls the server `/tokenize` endpoint and trims user content if input plus `max_tokens` would exceed `max_model_len`.

## Parameter defaults

| Parameter                | Value           | Notes                                           |
| ------------------------ | --------------- | ----------------------------------------------- |
| Scored prompts / round   | 10 (+ 2 warmup) | Warmup discarded before scoring                 |
| `max_tokens`             | 512             |                                                 |
| Context budget           | ~6,144 tokens   | Sized for teacher-forcing on v0.9.2             |
| `EVAL_MIN_CONTEXT_CHARS` | 4,000           |                                                 |
| `PROMPT_ENGINE_VERSION`  | 4               | In baseline cache key; bump on sampling changes |
| `SCORING_MAX_MODEL_LEN`  | 7168            | Scoring container `--max-model-len`             |
| dataset `revision`       | `a0e01428...`   | Pinned commit SHA                               |

## Correctness and prompts

Teacher-forcing via `prompt_logprobs` runs on the **same outputs** as speed scoring. The scoring vLLM uses a matching ~7k context cap. End-to-end speed is measured during generation; the final score requires both the token-match pre-filter and teacher-forcing to pass. Streamed top-5 logprobs are collected for diagnostics and baseline cache.
