Prompts are **instruct-style requests** built from curated instruction templates filled with long-context passages from [**PG19** (Project Gutenberg books)](https://www.tensorflow.org/datasets/catalog/pg19). A **block-hash-derived seed** makes them deterministic: same block, same prompts, anywhere, any machine.

## Why instruct format

The target model is `Qwen2.5-72B-Instruct` and the long-term positioning is "fastest backend for agent workflows." Benchmarking with raw text continuation does not reflect real serving workloads. Instruct-format prompts test what production deployments actually send.

## Hybrid prompt design

Each prompt combines two pieces:

1. **Instruction template** -- curated templates that reflect real workload shapes
2. **Context passage** -- long text sampled from PG19

Both are selected deterministically via the block-hash seed.

### Template categories

| Category           | Example instruction                                                   | Tests                             |
| ------------------ | --------------------------------------------------------------------- | --------------------------------- |
| **Summarization**  | "Summarize the following document in 3 paragraphs."                   | Long-context prefill, generation  |
| **Document QA**    | "Answer the following question based on the document: {question}"     | Prefill + targeted decode         |
| **Agent trace**    | "Given the tool outputs below, decide the next action."              | Short decode, high TTFT pressure  |
| **Code/retrieval** | "Extract all function signatures from the following code listing."    | Structured output decode          |
| **Long text wrap** | "Continue writing in the style of the author, starting from: {text}" | Raw generation from long context  |

~20 curated templates, rotated per block hash. The template determines the workload shape; the PG19 passage provides the context fill.

## Why PG19 for context

Inference benchmarking needs long, continuous sequences that stress prefill, attention, and KV cache. Short documents produce trivially fast responses where optimization differences are lost in noise.

| Dataset                     | Typical doc length             | Why rejected or chosen                                    |
| --------------------------- | ------------------------------ | --------------------------------------------------------- |
| FineWeb                     | Median &lt;1K tokens           | Too short -- prefill is trivial, no differentiation       |
| MMLU / HellaSwag            | ~100-500 tokens                | Evaluation benchmarks, not serving workloads              |
| **PG19**                    | **100K+ tokens (full novels)** | **Chosen** -- long-form text forces real serving at scale |

PG19 is pinned to a specific Hugging Face dataset revision. Row ordering is stable.

## Seeding from block hash

Randomness is derived from the block hash. Each block produces a deterministic seed:

```python
import hashlib, random

def derive_seed(block_hash: str) -> int:
    h = hashlib.sha256(block_hash.encode()).digest()
    return int.from_bytes(h[:8], "big")

def sample_prompts(block_hash: str, n: int = 10):
    seed = derive_seed(block_hash)
    rng = random.Random(seed)
    templates = rng.sample(INSTRUCTION_TEMPLATES, n)
    passages = _sample_pg19_passages(rng, n, max_chars=131072, min_chars=1000)
    return [template.format(context=passage) for template, passage in zip(templates, passages)]
```

Anyone with the block hash can reconstruct the exact prompt set used in that evaluation round.

## Parameter defaults

| Parameter           | Value             | Why                                                          |
| ------------------- | ----------------- | ------------------------------------------------------------ |
| `n` (prompts/round) | 10                | Balance between statistical signal and eval time             |
| `max_chars`         | 131,072           | ~32K tokens. Long enough for prefill to be a real bottleneck |
| `min_chars`         | 1,000             | Filters PG19 frontmatter, chapter headers, OCR artifacts     |
| `revision`          | pinned commit SHA | Row ordering stable across upstream dataset updates          |

## Prompt order randomization

Prompt order is shuffled per challenger, seeded by `block_hash + challenger_hotkey`. This prevents miners from optimizing for "first prompt is always the longest" patterns.

## Why long context matters

At short context, every server is fast. The differences that matter show up at scale:

| Context length | Prefill time (vLLM baseline, 4x H200) | Differentiation potential |
| -------------- | -------------------------------------- | ------------------------- |
| 2K tokens      | ~20ms                                  | Invisible                 |
| 8K tokens      | ~100ms                                 | Marginal                  |
| **32K tokens** | **~1-2s**                              | **Clear**                 |

The `max_chars` default is tuned for the 32K regime. This is where prefill optimization, chunked prefill, attention kernel choice, and KV cache management actually differentiate servers.

## Decode length

The validator requests **256 tokens** per prompt via `max_tokens=256`. This is enough time for decode-phase optimizations (memory layout, scheduling, communication) to differentiate from baseline.
