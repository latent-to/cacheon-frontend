Optima drives both launches (baseline and candidate) with the **same** prompts, sampled fresh each epoch. Two paths exist: a generic corpus for the cheap KL and calibration smoke, and the **real-benchmark** path (`optima bench`) that is the actual scoring workload.

## Per-epoch sampling

A kernel must not be able to special-case a fixed handful of prompts, so the validator samples a fresh subset every epoch from a larger corpus, keyed by an epoch seed:

```python
sample_prompts(n, seed)                 # deterministic subset for this epoch
sample_prompts_for_block(block_hash, n) # seed derived from the chain block hash
```

The seed is derived from the chain **block hash** (folded with `PROMPT_ENGINE_VERSION`). This gives two properties at once:

- **Cross-validator agreement.** Two validators scoring a submission at the same block derive the *same* seed, so they draw the identical prompt set. This is a hard requirement for consensus: they must agree on the weights they set for that submission.
- **Unpredictability across blocks.** The seed rotates per block, so a kernel cannot pre-bake answers for a known prompt set. Sampling is without replacement when `n <= len(CORPUS)`, otherwise with replacement so callers can request large workloads for throughput measurement.

The corpus today is a stand-in of technical instruction prompts (algorithms, systems, databases, ML) diverse enough to exercise varied shapes and stabilize the KL estimate. In production this corpus is drawn from the real serving distribution and rotated and expanded each epoch. `PROMPT_ENGINE_VERSION` is bumped when the corpus or sampling changes, so a change reshuffles prompts even at the same block and the version a score was produced under stays reproducible.

<Callout title="Why fresh prompts" type="info">
Overfitting the eval distribution (great on eval prompts, useless in production) never fully closes; it is a tuning discipline. Fresh per-epoch prompts, plus rotating the corpus toward the real serving distribution, are the defense. Memoize-and-replay is also defeated: activations differ per prompt and token, and the prompt set changes each epoch, so caching does not help at the op level.
</Callout>

## The real-benchmark scoring path (`optima bench`)

The generic corpus is fine as a cheap KL and calibration smoke, but it is a *toy* workload (short generations on filler prompts). The **scoring** workload is `optima bench`, which drives both launches with a few samples from **real benchmarks** at realistic generation lengths, so the throughput being scored reflects the decode-heavy work production actually does, not a 64-token toy task.

Two benchmarks today, both prompted for chain-of-thought so they are *long-generation* workloads rather than a one-token logit rank:

| Benchmark | Domain | Answer format | Decode budget (`max_new_tokens`) |
| --- | --- | --- | --- |
| **GSM8K** | grade-school math | numeric ("The answer is N.") | `256` |
| **MMLU** | knowledge, multiple choice | option letter ("The answer is (X).") | `512` |

Both are sampled per epoch by the same seeded interface (`load(n, seed)`), few-shot-prompted deterministically, and answers are extracted and checked (numeric equality for GSM8K, option-letter match for MMLU). The same `Benchmark` protocol takes SWE-bench, Tau-bench, and KernelBench later; only `check()` changes (run tests or tools in a sandbox instead of extracting an answer).

## Two gates on one realistic run

On that single realistic run the validator gates on **both** (see [Scoring](/docs/evaluation/scoring#gate-b--fidelity-the-load-bearing-anti-cheat)):

- **Per-token KL** vs the baseline, on the same prompts: the dense, low-variance **primary** gate. A faithful kernel sits at the noise floor; a cheat blows up.
- **Task accuracy**: no regression on any benchmark beyond a tolerance, a capability **floor**. It answers "did the model still solve real problems?"

A faithful kernel preserves both. A kernel that secretly degrades the model drops accuracy and/or blows up KL and scores zero, even if it looked fast. On a validated run a broken silu dropped GSM8K 62.5% → 0% while being 26% *faster*, and scored 0.

<Callout title="Why KL is primary, accuracy is a floor" type="warn">
Benchmark accuracy is noisy at small n: at n=12, GSM8K's ~12% standard deviation turns a two-problem flip into "−16.7%." So KL is the dense, low-variance primary gate; benchmark accuracy is a capability floor best read at ~100-200 samples. Fixing the workload *regime* (real tasks, decode-heavy lengths) does not change that a single small op (silu, rmsnorm) is a tiny fraction of end-to-end runtime, so its throughput contribution can sit below the measurement noise floor; that is a slot-size problem (bigger slots like attention and MoE), tracked separately from the gate.
</Callout>
