Prompts are **instruct-style requests** built from curated instruction templates filled with [**PG19** (Project Gutenberg)](https://www.tensorflow.org/datasets/catalog/pg19) passages. A **block-hash-derived seed** makes them deterministic: same block, same prompts, anywhere, any machine.

## Two prompt categories

Two prompt sets are used during evaluation: _speed evaluation prompts_ and _correctness evaluation prompts_.

Speed evaluation prompts are long-context generation workloads used to measure TTFT and throughput. Correctness evaluation prompts are shorter prompts used for correctness validation through teacher-forcing.
| Category | Pass | Count | Input context | `max_tokens` | Seed suffix |
| ---------- | ---------------------- | ----------------------------------------- | ---------------------- | ------------ | ----------- |
| **Speed** | Pass 1 (speed + token match) | 12 total (10 scored + 2 warmup discarded) | Long (up to ~128k cap) | 512 | (none) |
| **Correctness** | Pass 2 (correctness) | 8, no warmup | ~4k tokens | 256 | `:audit` |

Both prompt sets are independently sampled from PG19 using the current block_hash with different seed suffixes, ensuring that every challenger in the round is evaluated against the same workload.

## Why instruct format

The target model is `Qwen2.5-72B-Instruct`. Instruct-format prompts reflect real serving workloads rather than raw text continuation.

## Prompt design

Each prompt combines:

1. **Instruction template**: one of 20 curated literary-analysis tasks
2. **Context passage**: a slice of text sampled from a PG19 book

Both are selected deterministically via the block-hash seed. Templates live in [`validator/prompts.py`](https://github.com/latent-to/cacheon/blob/main/validator/prompts.py).

## Why PG19 for context

Long, continuous sequences stress prefill, attention, and KV cache. Short documents produce trivially fast responses where optimization differences are lost in noise.

| Dataset          | Typical doc length             | Why rejected or chosen                                  |
| ---------------- | ------------------------------ | ------------------------------------------------------- |
| FineWeb          | Median &lt;1K tokens           | Too short: prefill is trivial, no differentiation       |
| MMLU / HellaSwag | ~100-500 tokens                | Evaluation benchmarks, not serving workloads            |
| **PG19**         | **100K+ tokens (full novels)** | **Chosen**: long-form text forces real serving at scale |

PG19 is pinned to Hugging Face revision `a0e01428956e39868fa36ccfa0ee236ff04e6a6b`.

<Callout title="PG19 Setup Requirement" type="info">
  **PG19 must be pre-downloaded** to the local HuggingFace cache before starting the validator. If
  the dataset is missing, the validator raises an error at startup with a copy-paste download
  command rather than attempting an 11 GB download mid-evaluation.
</Callout>

## Passage quality filters

| Filter                | Speed evaluation threshold | Correctness evaluation threshold | Rejects                           |
| --------------------- | -------------------------- | -------------------------------- | --------------------------------- |
| Minimum length        | 16,000 chars               | 4,000 chars                      | Rows too short for the category   |
| Alpha character ratio | ≥ 50%                      | ≥ 50%                            | OCR noise, numeric tables         |
| Whitespace ratio      | ≤ 35%                      | ≤ 35%                            | Chapter headers, blank-heavy rows |

The sampler retries up to 1,000 times before raising an error.

## Parameter defaults

| Parameter               | Speed (Pass 1)               | Correctness (Pass 2) | Notes                                           |
| ----------------------- | ---------------------------- | -------------------- | ----------------------------------------------- |
| Scored prompts / round  | 10 (+ 2 warmup)              | 8                    | Warmup discarded before scoring                 |
| `max_tokens`            | 512                          | 256                  |                                                 |
| Context budget          | Up to 131,072 tokens (8 GPU) | ~4,096 tokens        | Speed fills toward cap; correctness stays short |
| `MIN_CONTEXT_CHARS`     | 16,000                       | 4,000                |                                                 |
| `PROMPT_ENGINE_VERSION` | 2                            | 2                    | In baseline cache key; bump on sampling changes |
| dataset `revision`      | `a0e01428...`                | `a0e01428...`        | Pinned commit SHA                               |

## Correctness and prompts

Pass 2 correctness prompts are short (~4k) by design: teacher-forcing via `prompt_logprobs` is run on a separate scoring vLLM with a matching small context cap. That pass validates output plausibility under the real model weights; it does **not** prove 128k behavior (known V1 gap).

Pass 1 collects top-5 streamed logprobs for diagnostics and baseline cache; correctness is **not** gated on first-divergence logprob analysis.

## V2 direction

V2 adds multi-turn conversations, RAG-style workloads, and concurrent request batches. V1 focuses on single-request instruct workloads with long stress context plus a short correctness pass. See [Roadmap](/docs/roadmap).
