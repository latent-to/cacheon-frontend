The validator scans the Bittensor chain for miner submissions, evaluates them against a vLLM baseline on GPU hardware, and sets one-hot weights to the current king.

## How it works

A single long-running process handles the full lifecycle:

1. **Chain scan**: polls the Bittensor chain for new miner commitments (Docker image + digest)
2. **Challenger detection**: filters for commitments not yet evaluated or rejected
3. **GPU search**: queries Targon and Lium for the cheapest available pod that meets [GPU requirements](#gpu-requirements)
4. **Automated rental**: rents a GPU pod, SSHs in, runs `setup-gpu.sh`, configures the environment, and starts `docker compose -f gpu-compose.yml up`
5. **Evaluation**: the GPU pod runs the vLLM baseline, evaluates each challenger, and uploads results to S3
6. **Teardown**: the pod is terminated after eval completes (or on failure/timeout)
7. **Weight setting**: the validator downloads results from S3 and sets one-hot weights to the current king

<Callout type="idea" title="Why this design?">

**Security:** The CPU host holds the wallets; the GPU pod has no wallet and only does eval work. S3 is the handoff layer.

**Cost:** GPUs are rented only when challengers exist and hardware is available. A configurable price cap (`CACHEON_MAX_HOURLY_PRICE`) prevents runaway spend, and the pod timeout scales automatically with the number of challengers.

</Callout>

## GPU requirements

Evaluation runs `Qwen2.5-72B-Instruct` with tensor parallelism. Use NVLink/SXM-class multi-GPU hardware where the workload is bottlenecked on cross-GPU bandwidth, and attach enough persistent disk for weights, images, and caches (see [Validator setup](/docs/validators/setup)).

The CPU orchestrator and `scripts/gpu_search.py` share one tier list in-repo (`validator.providers.TIERS`). Matching uses GPU type, GPU count, and minimum VRAM per GPU (B300 `>=` 288 GB, B200 `>=` 180 GB, H200 `>=` 141 GB, H100 `>=` 80 GB, A100 `>=` 80 GB). Within a tier, the cheapest hourly price wins; Tier B is only used when no Tier A config is available.

| Tier              | Configs (preferred order)                                                                    | When used                                                |
| ----------------- | -------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| **A** (preferred) | 2x B200, 2x B300, 4x H200 SXM, 8x H200 SXM, 4x B200                                          | Default.                                                 |
| **B** (fallback)  | 2x H200 SXM, 4x H100 SXM, 4x A100 80GB, 4x B300, 8x H100 SXM, 8x A100 80GB, 8x B200, 8x B300 | Only when Tier A has zero availability across providers. |
