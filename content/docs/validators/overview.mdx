A validator is the **referee**. It owns the entire measurement: it hosts the fixed
model on its own GPUs, swaps each miner's kernel into a single typed boundary (a
*slot*), runs the model, and measures two things against the stock (unmodified
sglang) baseline — **throughput** and **fidelity**. A miner earns if and only if the
kernel is *both* faster *and* faithful. The validator is what turns that rule into a
number that ends up as chain weights.

This page is the map. The rest of the validator guide fills in each piece.

## Responsibilities

A running validator does five things, in a loop:

1. **Run the referee.** Read miner submissions off the chain, fetch them, hash-verify
   them, and evaluate each one out-of-process. The validator never imports miner code
   in a trusted process. See [The chain loop](/docs/validators/chain-loop).
2. **Evaluate.** For each submission, bracket the candidate between two stock
   baseline launches (`B, C, B′`), time them, and check fidelity. The commands and
   their flags are in [Running evals](/docs/validators/running-evals).
3. **Score.** Turn raw tok/s into a noise-robust verdict and a scoreable result. The
   math — bookended measurement, the `1 + max(margin, k·noise)` bar, the NO-DECISION
   case, king-of-the-hill — is in [How scoring works](/docs/validators/scoring).
4. **Settle.** One champion per slot. A challenger takes the title only by beating
   the champion by the margin. Emission splits across slots that have a champion.
5. **Push weights.** Read the champion state off the ledger and emit it on chain.
   Weight *policy* is intentionally narrow: local `set-weights` reads
   `Ledger.current_weights`, while the current chain loop pushes the
   `settle_per_slot(...)` result directly. Keep those paths aligned when changing
   emissions.

## The trust posture: the validator owns all measurement

The validator **never trusts a miner-reported number.** Every datum that decides a
score is produced by the validator, or derivable from inputs the validator controls:

- **Throughput.** The token count in the numerator is driver-known a priori
  (`prompts × max_new_tokens` under `ignore_eos`), not the scheduler's self-reported
  field (`optima/eval/throughput_kl.py`, `_counted_tokens`). Timing brackets are
  `torch.cuda.synchronize`-wrapped in the driver process, which never imports the
  miner module (`optima/eval/_launch.py`, `seam.mark_driver`).
- **Fidelity.** KL is computed against a validator-run baseline launch; the in-engine
  audit re-runs the *captured stock baseline* on pre-call clones and compares under
  the slot's own verify tolerances (`optima/audit.py`). See
  [Fidelity: audit vs KL](/docs/validators/fidelity).
- **Identity.** A submission is content-addressed (`optima/bundle_hash.py`); the
  fetched archive is re-hashed after extraction and a mismatch is a hard reject
  (`optima/chain/fetch.py`, `fetch_bundle`).
- **Anti-phantom-pass.** An active candidate launch must produce seam receipts
  proving the bundle loaded *and* the miner impl was actually selected — else a
  silent stock-vs-stock run would score KL 0.0 / PASS (`optima/receipts.py`).

The miner's whole on-chain footprint is one timelock commitment carrying
`{"v":1,"h":"<content_hash>","u":"<url>"}`. Everything else is off-chain, on the
validator.

## Architecture: CPU control box + GPU box

Production splits the validator across two machines, joined by SSH. This is the
isolation pattern documented in `docs/SUBNET_BLUEPRINT.md` §8 and is the concrete
shape of the threat model:

- **CPU control box** — holds the chain keys (hotkey), the ledger, the loop. It
  never runs miner code. The coldkey stays off-box entirely; the recurring loop is
  **hotkey-only**.
- **GPU box** — runs sglang and the untrusted miner kernel. It holds **no chain
  keys, no cloud creds, and no network egress** except the SSH control channel. A
  fully malicious kernel that achieves code execution can't steal keys that aren't
  there and can't exfiltrate over a network that isn't there. It can be wiped
  between evals.

The two are wired together by the loop's `--eval-cmd` contract: the control box
shells out over SSH, the GPU box runs the arena evaluator wrapper (`optima
evaluate` plus `optima bench` when the arena requires a benchmark gate), and the
result travels back as a JSON report. See [The chain loop](/docs/validators/chain-loop) and
[Running on testnet](/docs/validators/testnet).

```
        ┌──────── CPU control box ────────┐
        │  chain keys (hotkey)            │
        │  Ledger (scores, champions)     │
        │  validator_loop.run_pass        │
        └───────────────┬─────────────────┘
                        │ ssh gpubox optima-eval.sh {bundle} {report}
        ┌───────────────▼─────────────────┐
        │  GPU box (no keys, no egress)   │
        │  sglang + miner kernel          │
        │  optima evaluate / bench        │
        └─────────────────────────────────┘
```

This mirrors the service decomposition in `docs/SUBNET_BLUEPRINT.md` §2: the chain
is the trust anchor **only** (commitments + weights); everything else — fetching,
hash verification, copy detection, the gate chain, settlement — happens on the
validator, recorded in the JSON `Ledger`.

## What's actually been run (the honest state)

Optima is arena-agnostic (see [Arenas](/docs/reference/arenas)); the slots, seam,
scoring, and chain loop are general. The first fully validated arena is
MiniMax-M3-NVFP4 on 4×B300, and the mechanism has also been validated on Qwen2.5
and gpt-oss-120b. Both halves of the M3 record are merged and live-validated:

- **The referee (PR #25).** The fused-epilogue bundles were evaluated as miner
  submissions through Optima's own scorer: shallow
  `collective.ar_residual_rmsnorm` (1.044× / 1.049×) and deep
  `collective.moe_finalize_ar_rmsnorm` (1.074× / 1.071×), all with **0
  fidelity-audit violations**. See [MiniMax-M3 case study](/docs/validators/minimax-m3-case-study) for the
  exact record regime, the eval-bridge architecture, and the last-layer veto bug.
- **The chain loop (PR #26).** On Bittensor testnet netuid 307, a miner hotkey
  committed the deep bundle via the native timelock commit-reveal; the validator
  loop fetched it, hash-verified it, evaluated it out-of-process on the real
  4×B300 referee, and scored it at SCORE 1.0717 (the third independent
  reproduction). See [Running on testnet](/docs/validators/testnet).

**One honest boundary:** weights stayed dry-run on 307. Subnet 307 is owned by
someone else and was never start-called, so its alpha pool rejects stake
(`SubtokenDisabled`) and the validator hotkey never got a permit (permits go to
the top-64 by stake). The *mechanism* is proven end to end; a real signed
`set_weights` needs a subnet where the validator has a permit — preferably one
you own.

## Economics (honest note)

Testnet pays nothing — netuid 307's alpha pool rejects stake and the validator
hotkey there never holds a permit, so `set_weights` only ever ran dry-run (see
above). Mainnet economics — a subnet Optima's own operators control, staked
validator permits, and the actual emission split miners are paid under — are
documented open items, not yet built. That last piece, the emission *policy*,
is deliberately kept behind one seam (`Ledger.current_weights`) precisely so it
can change without touching chain I/O: today it's per-slot king-of-the-hill,
and it is expected to move toward relative-improvement-plus-time-decay rather
than staying winner-take-all forever. No revenue or reward projections exist
for this subnet; none should be inferred from anything on this page.
