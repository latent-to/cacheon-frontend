The validator scans the Bittensor chain for miner submissions, evaluates them against a vLLM baseline on GPU hardware, and sets one-hot weights to the current king.

## How it works

A single long-running process handles the full lifecycle:

1. **Chain scan**: polls the Bittensor chain for new miner commitments (Docker image + digest)
2. **Challenger detection**: filters for commitments not yet evaluated or rejected
3. **GPU search**: queries Targon and Lium for the cheapest available pod that meets [GPU requirements](#gpu-requirements)
4. **Automated rental**: rents a GPU pod, SSHs in, runs `setup-gpu.sh`, configures the environment, and starts `docker compose up`
5. **Evaluation**: the GPU pod runs the vLLM baseline, evaluates each challenger, and uploads results to S3
6. **Teardown**: the pod is terminated after eval completes (or on failure/timeout)
7. **Weight setting**: the validator downloads results from S3 and sets one-hot weights to the current king

<Callout type="idea" title="Why this design?">

**Security:** The CPU host holds the wallets; the GPU pod has no wallet and only does eval work. S3 is the handoff layer.

**Cost:** GPUs are rented only when challengers exist and hardware is available. A configurable price cap (`CACHEON_MAX_HOURLY_PRICE`) and timeout (`CACHEON_GPU_TIMEOUT_MIN`) prevent runaway spend.

</Callout>

## GPU requirements

Evaluation requires multi-GPU NVLink/SXM interconnect to run `Qwen2.5-72B-Instruct` with tensor parallelism. PCIe-connected GPUs (most consumer cloud offerings) cannot load this model across cards at the required speed.

All instances must meet these baseline conditions:

- NVLink/SXM interconnect required
- Storage: 400 GB minimum
- VRAM per GPU: H200 `>=` 141 GB, H100 `>=` 80 GB, A100 `>=` 80 GB, B200 `>=` 180 GB

`gpu_search.py` applies these conditions and picks the cheapest available instance across two tiers:

| Tier              | Configs (preferred order)                        | When used                                                    |
| ----------------- | ------------------------------------------------ | ------------------------------------------------------------ |
| **A** (preferred) | 4x H200, 8x H100, 8x A100 80GB, 8x H200, 2x B200 | Default. Any can run 72B BF16 with tensor parallelism.       |
| **B** (fallback)  | 2x H200, 4x H100, 4x A100 80GB, 4x B200, 8x B200 | Only when Tier A has zero availability across all providers. |
