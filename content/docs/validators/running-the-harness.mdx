Optima today is a **command-line harness plus a local JSON ledger**. This page covers how to actually run it: install the package, install the seam, do a CPU dry-run with no GPU, then run a real GPU evaluation. There is no always-on service to deploy yet; see [Production deployment (roadmap)](#production-deployment-roadmap) at the end.

## Install the package

```bash
pip install -e .
```

This installs the `optima` package and the `optima` CLI. For a GPU box you will instead build a dedicated venv around a pinned sglang; see [GPU pods](/docs/validators/gpu-pods).

## Install the seam

sglang runs the model in a **spawned** scheduler process, a fresh Python interpreter that re-imports sglang from scratch, so patching a class in the parent process never reaches it. The only hook that runs in every interpreter, including the spawned child, is a `.pth` file. Write one line into site-packages:

```bash
SITE_PACKAGES=$(python -c 'import site; print(site.getsitepackages()[0])')
echo 'import optima.bootstrap' > "$SITE_PACKAGES/optima.pth"
```

A `.pth` line that begins with `import` is executed at interpreter startup. `optima.bootstrap` registers a post-import hook that installs the seam the moment a target sglang module loads, in the parent and in every spawned child. This is why the seam works at all; do not expect parent-process patching to do anything without it.

<Callout type="warn" title="Always invoke as a module">
  Run every GPU eval as <code>python -m optima.cli ...</code>, not a bare <code>optima</code> entry
  point. sglang uses <code>mp</code> spawn, and the spawn-safe <code>__main__</code> guard in
  <code>python -m optima.cli</code> is what keeps the child process from re-running the driver. The
  bare <code>optima</code> console script is fine for CPU-only commands (<code>slots</code>,
  <code>scan</code>, <code>verify --device cpu</code>).
</Callout>

## CPU dry-run (no GPU)

The dry-run exercises the whole non-GPU half of the pipeline: manifest parse, static scan, kernel load, and op-correctness. It needs no CUDA.

```bash
# list the slot catalog
optima slots

# static policy scan of an example bundle
optima scan examples/miner_silu_torch

# op-correctness on CPU against the slot's high-precision reference
optima verify examples/miner_silu_torch --device cpu --dtype float32

# the unit suite (scanner, manifest, KL, verify, block/moe seams, collective, ledger)
pytest tests/
```

`optima slots` prints the typed slot catalog. `optima verify` generates deterministic inputs for the slot's shapes, runs the miner kernel and the reference, and compares them; a wrong kernel fails here before any expensive GPU run.

## GPU evaluation

On a CUDA box with the seam installed and the toolchain set up ([GPU pods](/docs/validators/gpu-pods) covers `CUDA_HOME`, `nvcc`, `ninja`, and `TORCH_CUDA_ARCH_LIST`), run op-correctness on device, then the two-launch evaluation. The two adversarial examples must be caught by the gate:

```bash
# op-correctness on device
python -m optima.cli verify examples/miner_silu_triton --device cuda --dtype bfloat16

# cheap KL smoke on a generic corpus: faithful PASSes, broken FAILs
python -m optima.cli evaluate examples/miner_silu_triton \
    --model Qwen/Qwen2.5-0.5B-Instruct --no-deterministic
python -m optima.cli evaluate examples/miner_silu_broken \
    --model Qwen/Qwen2.5-0.5B-Instruct --no-deterministic

# the real scoring path: throughput on real benchmark prompts (GSM8K + MMLU,
# long CoT generation), gated on KL and task accuracy from the same run
python -m optima.cli bench examples/miner_silu_triton \
    --model Qwen/Qwen2.5-1.5B-Instruct --benchmarks gsm8k,mmlu --samples 64
```

`optima evaluate` is the cheap KL smoke on a generic corpus (a quick calibration check on a toy workload). `optima bench` is the scoring path: it drives both launches with real benchmark samples at realistic generation lengths, so the throughput it scores reflects the decode-heavy work production actually does.

<Callout type="info" title="Score big MoE in deterministic mode">
  On large MoE models, score with <code>enable_deterministic_inference</code> so the stock-vs-stock
  KL noise floor goes to ~0 and kernel drift is cleanly attributable. Always measure the
  stock-vs-stock floor first, then set the KL threshold to k× the measured floor rather than a
  hand-picked constant. Where deterministic mode is unavailable, run <code>--kl-advisory</code> and
  let the benchmark accuracy gate carry quality. See <a href="/docs/evaluation/scoring">Scoring</a>
  and the calibration findings in the source README.
</Callout>

### A full anti-copy round

Commit, reveal, evaluate against the ledger, then settle king-of-the-hill:

```bash
optima commit  examples/miner_silu_triton --hotkey alice --salt s1 --round 0 --ledger l.json
optima reveal  examples/miner_silu_triton --hotkey alice --salt s1 --round 0 --ledger l.json
python -m optima.cli evaluate examples/miner_silu_triton \
    --model Qwen/Qwen2.5-0.5B-Instruct --no-deterministic \
    --ledger l.json --hotkey alice --round 0
optima settle  --round 0 --margin 0.02 --ledger l.json
```

The ledger (`l.json`) is the local JSON stand-in for what production replaces with a database and on-chain commit-reveal.

## Production deployment (roadmap)

<Callout type="warn" title="Not shipped">
  There is no deployed Optima validator service today. The harness is a CLI plus a local JSON
  ledger. The architecture below is the target, not built. Do not configure it as if it exists, and
  note that Optima defines no deployment environment variables. The only real env vars are the
  miner and seam ones (<code>OPTIMA_ACTIVE</code>, <code>OPTIMA_BUNDLE_PATH</code>,
  <code>OPTIMA_MOE_SEAM</code>, <code>OPTIMA_ATTENTION_SEAM</code>,
  <code>OPTIMA_COLLECTIVE_SEAM</code>).
</Callout>

The target production shape (see [Architecture](/docs/validators/architecture#production-target-architecture)) splits the harness into an always-on CPU control plane and ephemeral GPU workers:

- **Always-on CPU control box.** Holds chain and cloud secrets, syncs the metagraph, reads on-chain commitments, drives the king-of-the-hill scheduler, and sets weights. Never runs miner code.
- **Ephemeral GPU eval workers.** Rented per eval, run only sglang plus the candidate kernel, hold no secrets, have no network egress, and are wiped after each eval. This is the isolation boundary that lets untrusted miners in.
- **A fleet of services around a database.** monitor, scheduler, executor, anticopy, api, and a thin validator that is the only writer of weights, coordinating through a DB rather than message brokers.

Each of those is a target, not a shipped feature. Until they are built, the VM is the trust boundary, chain integration is the local JSON ledger, and there is no public monitoring API.
