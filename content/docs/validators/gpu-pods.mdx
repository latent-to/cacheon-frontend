Optima evaluation runs on a CUDA GPU. This page covers renting one on lium.io, the toolchain sglang needs at runtime, and the arch flags per GPU. The harness was validated on a single H100.

## Renting a GPU on lium.io

We rent GPUs on [lium](https://lium.io) (Datura-ai). The CLI drives pods non-interactively. Pod names and IPs change on redeploy, so always run `lium ps` first to get the current ones.

```bash
lium ps                          # list pods + names + IPs
lium exec <pod> "nvidia-smi -L"  # run a command non-interactively
lium ssh  <pod>                  # interactive shell
lium rsync <pod> ./optima        # push a directory (use for the harness)
lium logs <pod>                  # stream logs
lium rm   <pod>                  # TERMINATE (stops billing)
```

<Callout type="warn" title="Pods bill while running">
  A single H100 is roughly $1.48/hr; multi-GPU boxes cost more. Tear down idle pods with
  <code>lium rm &lt;name&gt;</code> as soon as an experiment finishes. They bill for as long as they
  are running.
</Callout>

## Bootstrapping the harness on a fresh pod

Push the harness (not the sglang clone), build a venv around the pinned sglang, and install the seam. This is the recipe validated on the H100:

```bash
# on your machine: push the harness and examples
lium rsync <pod> ./optima
lium rsync <pod> ./examples

# on the pod:
curl -LsSf https://astral.sh/uv/install.sh | sh
cd /root/optima && uv venv --python 3.12 .venv && source .venv/bin/activate

# sglang on CUDA 13. --prerelease is needed (0.5.12 depends on the flash-attn-4
# beta, a pure-python wheel); --torch-backend routes torch to the cu130 index.
uv pip install --prerelease=allow --torch-backend=cu130 "sglang==0.5.12.post1"
uv pip install "kernels>=0.12,<0.13" datasets pytest -e .

# install the seam in every interpreter (spawned scheduler included)
SP=$(python -c 'import site; print(site.getsitepackages()[0])')
echo 'import optima.bootstrap' > "$SP/optima.pth"
```

The `kernels<0.13` pin matters: transformers 5.6 breaks against kernels 0.15 with `LayerRepository: Either a revision or a version must be specified` at `import sglang`. See [sglang tracking](/docs/validators/sglang-tracking) for the pinned version and the bump process.

## The toolchain: CUDA_HOME, nvcc, ninja

sglang JIT-compiles CUDA kernels at **runtime** (its `jit_kernel` path), so the pod needs `nvcc` and `ninja` on `PATH` and `CUDA_HOME` set. Without them, `import sglang` or the first forward pass fails to build.

```bash
export CUDA_HOME=/usr/local/cuda
export PATH=/usr/local/cuda/bin:$PWD/.venv/bin:$PATH   # nvcc + ninja for sglang's JIT
```

## TORCH_CUDA_ARCH_LIST per GPU

Set the compute arch to match the GPU so the runtime JIT targets the right hardware:

| GPU | `TORCH_CUDA_ARCH_LIST` |
|---|---|
| H100 (Hopper) | `9.0` |
| B200 (Blackwell) | `10.0` |

```bash
export TORCH_CUDA_ARCH_LIST=9.0   # 9.0 = H100, 10.0 = B200

# smoke: op-correctness then a two-launch eval
.venv/bin/python -m optima.cli verify   examples/miner_silu_triton --device cuda
.venv/bin/python -m optima.cli evaluate examples/miner_silu_triton \
    --model Qwen/Qwen2.5-0.5B-Instruct --no-deterministic
```

Once the smoke passes, run the full CPU dry-run and GPU eval recipe in [Running the harness](/docs/validators/running-the-harness).

## Model size: one H100 to multi-GPU

**gpt-oss-120b fits a single 80 GB H100** in the validated Hopper path (about 69 GB at its native quantization), so the bootstrap does not need a B200 cluster. The mechanism is validated on a single GPU up to that model size.

For larger models or to exercise the tensor-parallel and MoE surfaces, rent a multi-GPU box and run with tensor parallelism. The MoE runner backend is selected explicitly:

```bash
# gpt-oss-120b, TP=4 (multi-GPU): plain-triton MoE, custom all-reduce off
.venv/bin/python -m optima.cli bench examples/miner_rmsnorm_broken \
    --model openai/gpt-oss-120b --benchmarks gsm8k,mmlu --samples 16 \
    --tp-size 4 --moe-runner-backend triton --disable-custom-all-reduce \
    --mem-fraction 0.85
```

`--tp-size` sets the tensor-parallel degree, `--moe-runner-backend` selects the MoE kernel backend, and the broken bundle above must FAIL the gate (accuracy collapse and/or KL blowup). Multi-GPU is where the tensor-parallel collective and MoE-overlap slots become measurable.

<Callout type="info" title="Isolation caveat on rented pods">
  Current lium pods run without <code>CAP_SYS_ADMIN</code>, so in-process
  <code>unshare(CLONE_NEWNET)</code> no-egress isolation fails with <code>EPERM</code>. Use
  <code>--allow-unsafe-no-isolation</code> only for dev throughput replication. Production scoring
  should run the eval worker in a namespace-capable container or VM, or launch the candidate side
  under Docker/OCI with <code>--network=none</code> and the required GPU mounts. See
  <a href="/docs/validators/architecture#isolation-the-frontier-gap">the isolation gap</a>.
</Callout>
