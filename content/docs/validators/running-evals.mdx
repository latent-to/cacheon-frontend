---
title: Running diagnostics
---

Cacheon exposes three developer-facing checks. They answer different questions and none
of them is authoritative settlement evidence.

| Command    | Question                                                                                             | Typical environment                  |
| ---------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------ |
| `verify`   | Does each declared variant satisfy its typed slot ABI and reference checks?                          | CPU or GPU; spawned worker processes |
| `evaluate` | On this development setup, how do baseline and candidate throughput and diagnostic fidelity compare? | GPU model launch                     |
| `bench`    | Does the candidate regress selected task accuracy, KL diagnostics, or noisy throughput?              | GPU model launch                     |

The production path instead uses a registered arena service, complete materialized engine
trees, no-egress OCI sessions, pristine T, retained evidence, and independent
reproduction. See [Qualification](/docs/validators/qualification).

<Callout type="warn">
  These commands are for code development and integration. Do not run unknown bundles on a machine
  that lacks an appropriate outer sandbox. `--allow-unsafe-no-isolation` explicitly weakens a
  diagnostic and must never be part of a crownable workflow.
</Callout>

## Op-level verification

CPU smoke test:

```bash
python -m optima.cli verify ./my_bundle --device cpu --dtype float32
```

GPU verification:

```bash
python -m optima.cli verify ./my_bundle \
  --device cuda --dtype bfloat16 --seed 17
```

Collective slots need the intended rank count:

```bash
python -m optima.cli verify ./my_collective_bundle \
  --device cuda --world-size 4 --dtype bfloat16
```

Use `--model <KEY>` when a validator model profile changes activation or low-bit
correctness semantics. A context-inapplicable variant is not treated as a successful
verification.

Verification checks all declared variant domains for ambiguity before loading a row. It
also exercises required graph capture/replay for graph-bound block and collective slots.
CPU fallback is correctness-only and cannot establish GPU graph coverage.

## End-to-end throughput diagnostic

Always point at model bytes controlled by the operator:

```bash
python -m optima.cli evaluate ./my_bundle \
  --model /models/example-model \
  --num-prompts 32 \
  --max-new-tokens 64
```

The command runs a development baseline/candidate comparison and prints a diagnostic
score that is explicitly not recorded. Keep CUDA graphs enabled for realistic graph-path
testing; `--disable-cuda-graph` is a debugging control, not a scoring configuration.

Useful controls include:

- `--tp-size`, backend selections, and engine JSON for the intended development setup;
- `--timed-iters` and `--warmup-iters` for thermal/noise diagnosis;
- `--max-running-requests` for a serving-like concurrency bound;
- `--prompt-seed` for repeatable local investigation; and
- `--fidelity-mode kl|audit` for the legacy diagnostic quality path described below.

### Long-prefill diagnostics

`--input-len` generates deterministic long inputs. The timed iterations replay the same
prompts, so the radix cache can turn later iterations into decode-only measurements. Pair
long-prefill testing with a reviewed engine configuration that disables the radix cache:

```bash
python -m optima.cli evaluate ./my_prefill_bundle \
  --model /models/example-model \
  --input-len 8192 \
  --engine-kwargs-json '{"disable_radix_cache": true}'
```

That flag combination remains a development diagnostic; production workload regimes are
bound by the arena manifest.

## Capability diagnostic

```bash
python -m optima.cli bench ./my_bundle \
  --model /models/example-model \
  --benchmarks gsm8k,mmlu \
  --samples 128
```

Small sample counts are smoke tests, not a meaningful accuracy floor. `bench` reports
baseline/candidate task results, KL diagnostics, and noisy throughput. It does not emit a
qualification report or evidence set suitable for settlement.

## Diagnostic fidelity modes

`evaluate` retains two legacy development modes:

- `kl` aligns baseline and candidate rollout distributions and checks configured mean,
  tail, and argmax-disagreement thresholds. It is informative only when repeated stock
  launches are sufficiently deterministic for that calibration.
- `audit` runs an extra untimed candidate launch and samples dispatcher calls against an
  in-engine stock calculation; rollout KL becomes advisory. Because this executes in the
  candidate engine, it is not the production grading authority.

Framework-mode diagnostics switch to token matching and imply the diagnostic no-egress
namespace request. They do not authorize engine-wide setup as a registered economic
target. Cross-cutting proposals belong to the separate discovery lane.

Production fidelity comes from candidate-free pristine T after C is destroyed. See
[Fidelity and quality authority](/docs/validators/fidelity).

## Interpreting results

- A quality failure is useful development feedback, not a settlement event.
- A noisy B/C/B′ diagnostic may print `NO-DECISION`; gather better evidence rather than
  converting it to a win or loss.
- A local speedup does not establish the registered arena, target identity, graph proof,
  pristine quality evidence, whole-stack regression, or independent reproduction.
- A diagnostic pass never changes the SQLite evaluation stack or reward claims.
