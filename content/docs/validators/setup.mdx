## Prerequisites

- Ubuntu 22.04+ VPS (any provider; 1 vCPU, 2 GB RAM is enough)
- Docker and Docker Compose installed
- A Bittensor validator wallet registered on subnet 14 (mainnet) or 470 (testnet)
- Hippius S3 bucket and credentials
- Targon and Lium.io API keys for automatic GPU rental
- Sufficient balance on Targon and/or Lium: rentals fail if billing cannot cover hourly GPU charges

## Configuration

Copy the example env file and fill in the required values:

```bash
cd cacheon/validator
cp .env.example .env
```

Then, update the required values in the .env file with your specific configuration.

### Chain and identity

The following environment variables are related to the Bittensor chain and wallet identity:

| Env var                   | Required | Description                                           |
| ------------------------- | -------- | ----------------------------------------------------- |
| `CACHEON_NETWORK`         | Yes      | `finney` (mainnet) or `test` (testnet)                |
| `CACHEON_NETUID`          | Yes      | `14` (mainnet) or `470` (testnet)                     |
| `CACHEON_WALLET_NAME`     | Yes      | Bittensor wallet name                                 |
| `CACHEON_WALLET_HOTKEY`   | Yes      | Bittensor hotkey name                                 |
| `CACHEON_POLL_INTERVAL_S` | No       | Seconds between chain scans (default `600`)           |
| `CACHEON_DRY_RUN`         | No       | Set to `1` to skip `set_weights` (useful for testing) |
| `CACHEON_SUBMISSION_FEE_RAO` | No    | Minimum submission fee in RAO (`0` disables payment gate; default `0`. Mainnet operators typically set `100000000` for `0.1` TAO) |

<Callout type="info" title="Submission fee gate">
  When `CACHEON_SUBMISSION_FEE_RAO` is greater than `0`, the CPU validator rejects commitments
  missing valid `payment_tx` / `payment_block` fields. Miners pay via `miner/commit.py`; see
  [Miner registration](/docs/miners/registration).
</Callout>

<Callout type="info" title="Bittensor wallet directory">
  Mount your wallet at the path expected by the container. The default is `~/.bittensor` mapped to
  `/root/.bittensor` inside the container; set `BITTENSOR_DIR` in `.env` if yours is elsewhere.
</Callout>

### S3

The following environment variables are related to the S3 bucket:

| Env var              | Required | Description                                  |
| -------------------- | -------- | -------------------------------------------- |
| `HIPPIUS_ACCESS_KEY` | Yes      | Hippius S3 access key                        |
| `HIPPIUS_SECRET_KEY` | Yes      | Hippius S3 secret key                        |
| `CACHEON_S3_BUCKET`  | No       | S3 bucket name (default `cacheon-validator`) |

### GPU orchestration

The following environment variables are related to setting up the GPU configuration:

| Env var                    | Required  | Description                                                                                                                                                                          |
| -------------------------- | --------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `CACHEON_AUTO_RENT`        | No        | Set to `1` to enable automatic GPU pod rental (default `0`)                                                                                                                          |
| `LIUM_API_KEY`             | \*        | Lium API key (\* at least one provider key required)                                                                                                                                 |
| `TARGON_API_KEY`           | \*        | Targon API key (\* at least one provider key required)                                                                                                                               |
| `TARGON_VOLUME_UID`        | If Targon | `vol-…` persistent volume UID; mounted at `/workspace` (see [section below](#targon-persistent-volume-targon_volume_uid)). Leave unset if you have Lium as your preferredm provider. |
| `CACHEON_MAX_HOURLY_PRICE` | No        | Maximum hourly price in cents (default `2000`)                                                                                                                                       |
| `HF_TOKEN`                 | No        | Hugging Face token for model download on GPU pod                                                                                                                                     |

### Targon persistent volume (`TARGON_VOLUME_UID`)

When the orchestrator rents on **Targon**, the API requires an existing persistent volume. The validator passes `TARGON_VOLUME_UID` into the create-workload call and mounts that volume at **`/workspace`** on the pod (weights, Docker layers, and eval scratch live there). The value must start with **`vol-`**, matching Targon’s volume id format; otherwise rental fails before deploy.

Do this **before** `bash setup-cpu.sh`:

1. In Targon, open **[Volumes](https://targon.com/volumes)** and create a persistent volume sized **over 500 GB** (headroom for the model, container images, and caches). For manual pods, see [Manual GPU setup](/docs/validators/gpu-pod-setup) for `/workspace` disk expectations.
2. Copy the volume UID (for example `vol-abc123…`).
3. Set `TARGON_VOLUME_UID=vol-…` in `validator/.env` (same file `setup-cpu.sh` loads).

<Callout type="info" title="Set a Preferred Provider">
  If Targon is never used (for example only **`LIUM_API_KEY`** is set, or
  **`CACHEON_PREFERRED_PROVIDER=lium`**), leave `TARGON_VOLUME_UID` unset. On **Lium**, the client
  creates or reuses a volume named **`cacheon-sn14-volume`** when you rent.
</Callout>

## Start

The setup script creates the Python virtual environment (or activates an existing one), installs dependencies, loads the `.env` file, tears down any running validator container, and starts docker compose:

```bash
bash setup-cpu.sh
```

To run in the background:

```bash
bash setup-cpu.sh --detach
docker compose logs -f
```

Expected log output on a healthy start:

```
INFO  validator.chain: Preflight OK: uid=0, stake=..., validator_permit=True
INFO  validator.cpu_validator: Tick 1: block=... challengers=0 weights_set=no
```

When `CACHEON_AUTO_RENT=1` and challengers are detected, the validator automatically searches for GPU pods, rents the cheapest match, runs setup and eval via SSH, and tears down the pod when done.

## Dry-run mode

Set `CACHEON_DRY_RUN=1` before starting to run the full chain scan and challenger selection without calling `set_weights`. Useful for verifying chain connectivity and S3 credentials before going live.

## Verify S3 connectivity (optional)

```bash
python -m validator.sync download --state-dir /tmp/verify
ls /tmp/verify/
```

If `eval_job.json` appears, a previous eval job is on S3 and the validator will pick it up on its next tick.
