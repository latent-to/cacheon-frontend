There are two things to understand: the **submission pipeline** the harness runs today, and the **production target architecture** it is meant to grow into. The pipeline is built and validated on real GPUs. The service fleet is a target, not yet built.

## The submission pipeline

Each submission moves through eight steps. Each maps to a file and a CLI verb in the Optima harness.

```
   MINER                              VALIDATOR
   -----                              ---------
1  commit  H(bundle,hotkey,salt) ───► record commitment      commit_reveal.Ledger.commit
                                      (no bundle seen yet)
                ... commit window closes ...
2  reveal  bundle + salt        ───► verify vs commitment     commit_reveal.Ledger.reveal
                                      detect copies            (earliest commit = original)
3                                     parse manifest           manifest.load_manifest
4                                     static policy scan        sandbox.scan_source
5                                     load kernel isolated      sandbox.load_entry
6                                     op-correctness vs ref     verify.verify_entry
7                                     two-launch evaluate       eval.throughput_kl.evaluate
                                        baseline (kernel off)
                                        candidate (kernel on)
                                        -> throughput + KL
8                                     settle king-of-the-hill   commit_reveal.Ledger.settle
                                      -> weights to the chain
```

1. **Commit** (`commit_reveal.Ledger.commit`). The miner posts `H(content_hash, hotkey, salt)`, a hash that binds them to an exact bundle without revealing it. CLI: `optima commit`.
2. **Reveal** (`Ledger.reveal`). After the commit window, the miner posts the bundle plus salt. The validator recomputes the hash and checks it matches a commitment *that hotkey* made earlier. A copier who only sees a rival's bundle at reveal time has no matching prior commitment, so they cannot reveal it. CLI: `optima reveal`.
3. **Parse manifest** (`manifest.load_manifest`). Validates schema, ABI version, and path-safety: source paths must be relative, inside the bundle, no `..` escape, no absolute paths, no symlink escape. The manifest is data, not code; nothing runs yet.
4. **Static policy scan** (`sandbox.scan_source`). An AST pass over the kernel source that rejects obvious egress and code-execution patterns (`socket`, `subprocess`, `pickle.loads`, `torch.load`, `os.system`, `eval`, `__import__`) before anything is imported. It is careful not to false-positive on Triton's `tl.load` / `tl.store`. This is a tripwire, not the isolation boundary. CLI: `optima scan`.
5. **Load the kernel** (`sandbox.load_entry`). Imports the miner module and pulls out the `entry` callable. This runs miner code, so `optima verify` loads and runs it **out-of-process** and in production it must happen inside the isolated worker (see [Security model](#isolation-the-frontier-gap)).
6. **Op-correctness** (`verify.verify_entry`). Generates deterministic inputs for the slot's standard shapes, runs the miner kernel and the slot's high-precision reference, and compares with tolerance. This is the cheap gate: an outright-wrong kernel is rejected here before the expensive end-to-end run. The per-op verify **jitters count dims** per run so a kernel cannot hard-code the verify shapes. Collective slots are verified distributed by `optima.verify_collective`. CLI: `optima verify`.
7. **Two-launch evaluate** (`eval.throughput_kl.evaluate`, and the real-task scoring path `eval.capability.evaluate_capability` behind `optima bench`). Baseline (kernel off) then candidate (kernel on), producing throughput plus KL plus benchmark accuracy. This is the heart of scoring; see [Overview](/docs/validators/overview#two-launches) and [Scoring](/docs/evaluation/scoring). CLI: `optima evaluate` / `optima bench`.
8. **Settle** (`Ledger.settle`). Applies **king-of-the-hill**: a challenger becomes champion only if it beats the current best by a margin; copies and non-improvers earn nothing. `optima settle --per-slot` runs a champion per slot and splits emission, so a specialist who owns one slot is paid. Emits weights for the chain. CLI: `optima settle`.

<Callout type="info" title="Anti-copy is cumulative across rounds">
  Copy detection (`optima/copy_fingerprint.py`) fires on the exact content hash **or** a
  reformat-invariant, AST-normalized fingerprint, computed per slot over each op's transitive
  bundle-local import closure. A reflowed, recommented, or renamed copy with a fresh hash is still
  demoted; the earliest committer wins. A structural skeleton fingerprint additionally flags
  rename-plus-constant-tweak near-copies as an advisory at reveal (surfaced for review, never
  auto-demoted).
</Callout>

## The seam: getting a kernel into a running model

Step 7 hides the hardest piece of engineering. `sglang.Engine` forces `mp.set_start_method("spawn")` and runs the model in a **separate scheduler process**, a brand-new Python interpreter that re-imports sglang from scratch. Patching a class in the parent never reaches the child, so naive monkeypatching silently does nothing.

The fix is a `.pth` file plus a post-import hook. A line in a `.pth` file under site-packages that begins with `import` runs at interpreter startup, in the parent **and** in every spawned child:

```bash
echo 'import optima.bootstrap' > $SITE_PACKAGES/optima.pth
```

`optima.bootstrap` registers a meta-path finder that watches for the import of the seam-target modules and, the moment one finishes loading, runs `seam.activate()` against it. `activate()` installs the dispatcher into the target class (for example `SiluAndMul.forward_cuda`), then decides whether to load the miner kernel from env: `OPTIMA_ACTIVE=1` plus `OPTIMA_BUNDLE_PATH=<dir>` loads and registers the kernel (candidate), otherwise the registry stays empty and the dispatcher falls back to stock (baseline). The same process code serves both launches; the validator picks baseline versus candidate by setting env before each launch.

The dispatcher (`optima/dispatch.py`) is the one place a miner kernel is called during inference. The validator owns the output allocation, the eligibility check, and the fallback; the miner's `entry` only ever sees pre-allocated tensors and, on any exception, the dispatcher falls back to the stock kernel.

**Tamper-resistant timing.** The `.pth` runs in every process, including the validator's driver process that holds the clock. To keep a malicious kernel away from the timer, the driver calls `seam.mark_driver()` *before* importing sglang; `activate()` sees that flag and installs a pass-through seam in the driver that never imports the miner module. The kernel loads only in the spawned scheduler child. The two are different OS processes communicating over IPC, so the kernel cannot reach the timer. This is what "out-of-process timing" means.

The five seam adapters live in one table (`optima/seams.py`): the bootstrap watch-list, `seam.activate()`, and the `optima compat` canary all derive from it. Chokepoints today: `SiluAndMul` and `RMSNorm` (ops), `RadixAttention.forward` and `FusedMoE.forward` (blocks), and `GroupCoordinator.all_reduce` (collective).

## State today: a local JSON ledger

The harness is **one CLI plus a local JSON ledger**. `commit_reveal.Ledger` persists commitments, scores, the champion, and copy verdicts to a JSON file. This is deliberately a stand-in so the whole mechanism is testable without a chain or a GPU (the commit-reveal ledger has its own unit tests). It is the toy version of what production splits into a database and a chain layer.

## Production target architecture

<Callout type="warn" title="Target, not built">
  Everything in this section is the **target** architecture, distilled from a study of a live
  Bittensor subnet (Affine). None of it is deployed. Today Optima is one CLI and a local JSON
  ledger. There is no always-on service, no database, and no public monitoring API. Do not treat the
  services below as existing infrastructure.
</Callout>

A production subnet is a **fleet of cooperating services around a database**, with the chain used only as the trust anchor (it holds miner commitments and weights; everything else lives in the DB). The service decomposition:

| Service | Job | Touches chain? |
|---|---|---|
| **validator** | the only thing that sets weights on chain; reads computed weights from the DB and emits them every ~180 blocks | yes, writes weights |
| **monitor** | syncs the metagraph, reads on-chain miner commitments, seeds the challenger queue | yes, reads |
| **scheduler** | the king-of-the-hill engine: a block-tick loop that picks a challenger, runs the battle, decides, updates the champion | no |
| **executor** | runs the actual evaluations (the two-launch throughput + KL run), writes results to the DB | no |
| **anticopy** | builds fingerprints and computes copy verdicts | no |
| **api** | read-only leaderboard / dashboard backend | no |

```
   Bittensor chain
     │   ▲
     │ commitments        weights │
     ▼   │
  monitor ──► ┌───────────────┐ ◄── validator ──► (sets weights on chain)
              │   DB: state   │
  scheduler ◄─┤ (single       ├──► api (leaderboard)
     │        │  writer)      │
     ▼        └───────────────┘
  executor ──► GPU box: sglang + miner kernel
  (two-launch eval)
```

A key principle borrowed from Affine: **no message brokers**. Everything coordinates through the DB, and state is implicit, so the system is restart-safe (re-derive from what rows are missing, do not replay). Exactly one service writes the scores and weights tables (single-writer), which prevents races.

Today's harness maps cleanly onto this target: `commit_reveal.Ledger` is the DB plus the king-of-the-hill part of the scheduler; `eval.throughput_kl.evaluate` is the executor; `cli.py` is the seam of all of them. What is new in production is the chain plumbing (monitor + validator), a real database (Postgres to start, with `miners`, `scores`, `snapshots`, and `system_config` tables), and the service split. Bittensor's native commit-reveal replaces the custom transport, so the on-chain half is commit + fetch and `commit_reveal.py` keeps only the off-chain scoring half.

### Isolation, the frontier gap

The largest not-yet-built piece is **isolation for untrusted miner kernels**. The kernel is Python that runs in the scheduler process; the static scan is a tripwire, not a boundary. The target pattern (also from Affine): chain and cloud secrets live only on a CPU control box, the GPU box runs only sglang and holds no secrets, the GPU box has no network egress and is wiped between evals, and each eval gets a per-eval CUDA context plus a watchdog for GPU DoS and out-of-bounds writes. Until that is built, the VM is the trust boundary and untrusted miners are not supported. `--framework-mode` and `--isolate` fail closed if the candidate process cannot prove no-egress network isolation; `--allow-unsafe-no-isolation` is for local debugging only.
