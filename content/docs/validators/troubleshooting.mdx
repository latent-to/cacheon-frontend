## Docker

### Docker not using GPU

Verify NVIDIA Container Toolkit is installed and configured:

```bash
docker run --rm --gpus all nvidia/cuda:12.8.0-base-ubuntu22.04 nvidia-smi
```

If this fails, reinstall the toolkit and restart Docker:

```bash
sudo nvidia-ctk runtime configure --runtime=docker
sudo systemctl restart docker
```

### Not all 4 GPUs visible

Check that all 4 H200 GPUs appear in `nvidia-smi`. If only a subset is visible, verify GPU topology and CUDA device ordering:

```bash
nvidia-smi topo -m
```

## vLLM baseline

### vLLM baseline not starting

Common causes:

1. **Insufficient GPU memory**: Qwen2.5-72B in BF16 needs ~140 GB total across 4 GPUs. Check `nvidia-smi` on all GPUs.
2. **Tensor parallel mismatch**: ensure `--tensor-parallel-size 4` matches the number of available GPUs.
3. **Model path wrong**: the mounted volume must contain the full HF snapshot. Check `ls /models` inside the container.
4. **Port conflict**: Another process on port 8000. Check with `ss -tlnp | grep 8000`.
5. **CUDA version mismatch**: Ensure the vLLM image supports your CUDA driver version.

## Miner containers

### Miner container pull fails

- Verify the image reference and digest are correct
- Ensure the machine has internet access for pulling images (network is only blocked inside miner containers during eval, not on the host)
- Check disk space: `df -h`

### Miner container times out

The default startup timeout is 10 minutes. 72B model loading on 4x H200 takes 2-5 minutes with weights mounted. If the miner's server legitimately needs more startup time, this is a problem with their implementation.

### GPU cleanup not working

If scores seem inconsistent between runs, check that GPU state is being properly reset. Verify all 4 GPUs are being cleaned:

```bash
nvidia-smi --query-gpu=memory.used --format=csv
```

All GPUs should show near-zero memory usage between evaluations (aside from the vLLM baseline).

## Chain interaction

### `set_weights` fails

1. **Wallet not registered**: verify with `btcli wallet overview`
2. **Insufficient stake**: weight-setting requires minimum stake
3. **Network issues**: the loop retries on the next poll

### Stale state after code update

If the state format changes between versions, delete the state directory:

```bash
rm -rf /workspace/cacheon/state
```

The loop rebuilds by evaluating all revealed commitments fresh.

### First eval takes very long

The first eval on a cold machine involves pulling Docker images. Model weights should already be downloaded (see [machine setup](/docs/validators/machine-setup)). The vLLM baseline also needs time to warm up on the first request.
