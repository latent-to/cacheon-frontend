---
title: CLI reference
---

The CLI is a thin operator and contributor surface over Cacheon's typed APIs. A
command's output has only the authority listed here; human-readable output is not a
qualification, settlement, or release receipt. The local contribution loop is `scan`
plus `verify` — there is no local command that grants qualification or settlement
authority.

```bash
python -m optima.cli <command> [options]
python -m optima.cli <command> --help
```

Using the module form is recommended on GPU hosts because SGLang starts child
processes. The installed `optima` console script targets the same parser.

## Command inventory

| Command            | Audience         | Authority                | Purpose                                                                   |
| ------------------- | ---------------- | -------------------------- | -------------------------------------------------------------------------- |
| `slots`             | all              | read-only                  | Print the registered slot ABI                                             |
| `compat`            | operator         | diagnostic                 | Check the installed SGLang seam surface against the pin                   |
| `chain-compat`      | operator         | diagnostic                 | Check the installed Bittensor SDK surface without chain access            |
| `scan`              | contributor      | local gate                 | Parse a bundle and apply recursive static policy                          |
| `verify`            | contributor      | local gate                 | Check declared slot behavior against validator-owned references           |
| `chain-package`     | contributor      | packaging                  | Build a canonical archive and print its content hash                      |
| `chain-submit`      | contributor      | chain mutation              | Commit a bundle hash and HTTPS fetch location through timelock reveal      |
| `chain-status`      | all              | read-only                  | Inspect public subnet, registration, and reveal state                     |
| `chain-register`    | operator         | chain mutation             | Burn-register a hotkey and run the SDK preflight                          |
| `chain-validate`    | validator        | production intake          | Consume finalized reveals; a deployment may inject qualification services |
| `set-weights`       | validator        | production control plane   | Reconcile the journaled global reward projection                          |
| `model-provision`   | release operator | production artifact        | Seal model bytes into a content-addressed publication and receipt         |
| `release-verify`    | release consumer | production verification    | Reopen a signed Engine release under an externally trusted key            |
| `release-context`   | release consumer | production build input     | Materialize a deterministic OCI context from a verified release           |

<Callout type="warn" title="No local qualification path">
  There is no local command that grants qualification or settlement authority. Complete engine
  qualification begins at the deployment-injected arena boundary. The direct evaluator, benchmark
  command, standalone hash command, and the local JSON commit/reveal/ledger simulator have been
  removed from the source tree; they are not an alternate production interface, and older examples
  referencing `evaluate`, `bench`, `hash`, `commit`, `reveal`, `ledger`, or `legacy-settle` no
  longer apply to the current checkout.
</Callout>

## Local contribution loop

### `slots`

```bash
python -m optima.cli slots
```

Reads the live `SLOTS` registry. It does not import contribution code or require
a GPU. See the [slot catalog](/docs/reference/slots-table).

### `scan`

```bash
python -m optima.cli scan path/to/bundle
```

Loads `manifest.toml` as data and checks the complete declared tree. A clean AST
scan is defense in depth; it does not make contribution code trusted.

### `verify`

```bash
# CPU sanity check
python -m optima.cli verify examples/miner_silu_torch \
  --device cpu --dtype float32

# CUDA slot verification
python -m optima.cli verify path/to/bundle \
  --device cuda --dtype bfloat16 --model MiniMax-M3

# Collective verification at the arena topology
python -m optima.cli verify path/to/bundle \
  --device cuda --world-size 4
```

Useful options are `--dtype`, `--device`, `--seed`, `--world-size`, `--tp-size`,
and `--model`. Verification proves only the registered component contract. It
does not prove a serving speedup or grant settlement authority.

## Submission commands

### Package

```bash
python -m optima.cli chain-package path/to/bundle --out bundle.tar.gz
```

Building the archive also prints its canonical content hash. Host the archive
at a stable **HTTPS** URL. The commitment binds that hash; validators fetch and
independently rehash the extracted tree.

### Submit

```bash
python -m optima.cli chain-submit path/to/bundle \
  --url https://artifacts.example/bundle.tar.gz \
  --network <network> --netuid <netuid> \
  --wallet <wallet> --hotkey <miner-hotkey> \
  --blocks-until-reveal 10
```

Use `--dry-run` to construct and print the payload without signing or sending
it. A submitted commitment is unreadable until its reveal delay elapses; the
validator acts only on finalized, valid reveals.

### Inspect chain state

```bash
python -m optima.cli chain-status \
  --network <network> --netuid <netuid>
```

Adding `--wallet` and `--hotkey` reports registration information for that
identity.

## Validator commands

### `chain-validate`

Intake-only mode is complete and intentionally available without a GPU arena:

```bash
python -m optima.cli chain-validate \
  --network <network> --netuid <netuid> \
  --intake-only --once
```

It performs finalized ordering, hardened fetch, rehashing, private retention,
and immutable worker publication. Full qualification additionally requires
deployment code to inject an `ArenaServiceRegistry` and select its closed
service with `--arena-id`. A reviewed deployment wrapper calls
`cmd_chain_validate(args, arena_registry=registry)` (or `run_validator(...)`)
after constructing that registry. Its operator-facing invocation may retain
the same arguments:

```bash
python -m optima.cli chain-validate \
  --network <network> --netuid <netuid> \
  --arena-id <registered-arena>
```

The shell command above is **not sufficient through the stock entrypoint**:
`python -m optima.cli` has no way to inject the registry and will refuse it.
The repository does not ship a production `ArenaServiceProvider`, and the
documentation does not provide a shell callback escape hatch.

Storage and loop controls are `--intake-db`, `--private-root`,
`--publication-root`, `--interval`, and `--once`. These paths contain durable
authority; back them up and apply the documented permissions.

### `set-weights`

```bash
python -m optima.cli set-weights \
  --network <network> --netuid <netuid> \
  --half-life-blocks <blocks> \
  --discovery-lifetime-blocks <blocks> \
  --discovery-pool-ppm <ppm> \
  --refresh-blocks <blocks> \
  --dry-run
```

The command recomputes a pure global projection from settled state. With
`--dry-run`, it returns the proposed payload without creating a journal intent
or submitting. Without `--dry-run`, reconciliation records the intent in the
SQLite journal and may sign/submit it. An SDK success return leaves the intent
pending; later chain observation is what confirms it. `--release-hold REASON`
appends an audited release of a held publication and does not submit it.

## Release commands

### Provision model bytes

```bash
python -m optima.cli model-provision \
  /srv/models/model /srv/optima/model-publication \
  --expected-content-digest <sha256>
```

The result is an immutable content-addressed tree plus a receipt. `--workers`
controls hashing parallelism.

### Verify a release

```bash
python -m optima.cli release-verify /srv/optima/releases/<digest> \
  --expected-public-key <ed25519-public-key> \
  --descriptor-digest <expected-digest>
```

Release verification requires a Cacheon installation with the `release` extra.

The expected public key is an external trust input; accepting a key discovered
inside the release would not authenticate its signer.

### Materialize a container context

```bash
python -m optima.cli release-context /srv/optima/releases/<digest> ./context \
  --expected-public-key <ed25519-public-key> \
  --descriptor-digest <expected-digest>
```

This first reopens the complete signed release and then emits a deterministic
OCI context. Release construction and signing are programmatic APIs; there is
no public `release-create` command.

## Exit behavior

Most commands return `0` on success and a non-zero code on refusal or failed
checks. Scripts should rely on the process status and retain structured/durable
receipts where a command emits them. Human-readable text is not settlement
evidence.
