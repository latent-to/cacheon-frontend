## Authority order

When sources disagree, apply this order:

1. executable code, closed registries, and schemas at the pinned commit;
2. the [product contract](https://github.com/latent-to/cacheon/blob/main/docs/PRODUCT_CONTRACT.md) and [slot contract](https://github.com/latent-to/cacheon/blob/main/docs/SLOT_CONTRACT.md);
3. authenticated retained evidence and immutable publications;
4. this ledger and the [changelog](/docs/history/changelog);
5. campaign notes, plans, audit snapshots, console output, and older narrative reports.

The evidence that proves one layer cannot be substituted for another:

| Evidence                       | Establishes                                               | Does not establish                                           |
| ------------------------------ | --------------------------------------------------------- | ------------------------------------------------------------ |
| Unit/focused suite             | Implemented invariants under test fixtures                | Real GPU behavior or production operations                   |
| `scan` / `verify`              | Static policy and exercised component contract            | Model throughput, pristine quality, or settlement            |
| One qualification PASS         | Registered B/C/B′/T decision under one authority          | Crown or release readiness                                   |
| Two independently bound PASSes | Settlement eligibility for the exact contribution/context | Integration or serving authorization                         |
| Signed release verification    | Descriptor, artifact, model, and signature consistency    | Reproducible registry build or successful production serving |

## Implemented surface

### Product and referee separation

- `EvaluationStackManifest` represents the complete incumbent and permits hostile proposal
  references only inside isolated evaluation.
- Qualification owns B/C/B′ execution, candidate-free pristine T quality, three-way
  decisions, and reproduction. Its durable attempt retains aggregate B/C/B′ speed
  witnesses plus referenced graph, raw-quality, and T/reference products; richer raw
  B/C/B′ session/device frames are validated in-run but are not serialized into the
  attempt.
- Settlement updates an arena-scoped evaluation stack with one target transition
  transactionally; reward projection and chain publication are separate state machines.
- `EngineReleaseManifest` accepts integrated contributions only. Promotion must reopen
  crown evidence and bind reviewed source, selected payload, provenance, security,
  compatibility, and tests.
- A serving release has no chain, wallet, intake, settlement, or live proposal dependency.

### Sealed direct artifacts

The closed artifact-provider registry contains one authoritative, crownable provider: `cutlass.cute.cubin.v1`.

Its contract is:

- candidate CuTe factory code runs only in a GPU-hidden, no-network hermetic prebuild
  child;
- the child must emit exactly one CUDA ELF CUBIN — never PTX, a host object, shared
  library, Python launcher, or runtime JIT input;
- the manifest carries declarative bindings, specialization, launch, resource, and
  lifecycle data rather than candidate host callbacks;
- the host seals build, tree, compiler, compile-profile, architecture, CUBIN, kernel ABI,
  and launch-plan identities;
- each rank binds logical exports by physical kernel ordinal only after selecting its CUDA
  device and initializing process groups; and
- launch materialization, pointers, scalars, packed fields, TMA descriptors, FastDivmod
  values, streams, generated storage, cleanup, and evidence remain validator-owned.

All registered targets allow the provider feature and all 11 slots expose declarative
call ABIs. That is schema coverage, not empirical proof of every slot/provider pair.
Direct-artifact collective declarations and group/peer resources are schema vocabulary,
not an executable standard path: `build_cute_cubin` supplies no group
capability/handle resolvers, so such plans fail closed. The fused-epilogue collective
evidence uses the separate hardened native-build/overlay lane.

See [Sealed direct artifacts](/docs/architecture/direct-artifacts).

## Empirical qualification evidence

The strongest retained performance authority is the joined blockscore qualification on a
TP4 arena hosted by an 8×B200 system:

| Field                         | Primary | Reproduction |
| ----------------------------- | ------- | ------------ |
| Charged-basis speedup         | 1.0561× | 1.0487×      |
| Timed-section-only diagnostic | 1.0932× | 1.0866×      |
| Qualification decision        | PASS    | PASS         |

Settlement accepted the pair after all seven required authority/evidence digests differed,
used the lower value, **1.0487×**, and committed a generation-1 crown. Restart reopening
passed. These checks prove distinct bound qualification products; they do not by themselves
prove separate operators, hosts, or infrastructure failure domains. The charged basis
includes conditioning and timed work; the timed-section result and the standalone
19.6–22.4% serving bracket use different denominators and are not interchangeable scores.

Evidence boundaries:

- The run used an SM100/B200 worker image from an earlier branch, before the evaluator
  trim and strict-validation branch. The relevant production pipeline was preserved, but
  current `f8be20a4` has not been rerun on that GPU arena.
- Its SGLang source build was `0.0.0.dev1+g56e290315`, not the repository constant
  `0.5.13.post1`. `compat` reports this mismatch without failing, so the run does not
  validate the source pin as a consensus boundary.
- A real projection with digest prefix `fb20bc7b` was submitted on subnet 307. The latest
  retained record still marked the timelock reveal pending; final chain readback is not
  claimed.
- Earlier fused-epilogue and MSA measurements remain useful mechanism evidence in the
  [MiniMax-M3 case study](/docs/validators/minimax-m3-case-study) and
  [changelog](/docs/history/changelog), but they do not replace this authority or become
  current-schema crowns retroactively.

## Current checkout hardening and release status

The strict-validation commits centralize repeated validation mechanisms in
`optima/_strict.py`. Module-specific exception types and policy bounds remain local;
shared helpers enforce canonical digests, exact integers, identifiers, exact fields,
duplicate-key rejection, CUDA driver integers, and environment flags. Most identity
authorities now reject the all-zero digest. Identity computation itself is unchanged.

The checked-out branch is **not release-ready**. The serving-wheel allowlist omits the
direct-artifact runtime module closure, including provider, ABI, device-launch/runtime,
CUDA, and CuTe modules. Commit `f8be20a4` added `optima._strict` and a clean extracted-wheel
smoke for `_strict`, the seam, bootstrap, the SGLang plugin, and `release_runtime`; that
narrower regression passes. Clean-wheel `import optima.manifest` still fails first on
missing `optima.artifact_provider`, and other release-path imports fail on omitted modules
such as `optima.cuda_cubin`. The 1,951-test suite does not exercise the complete
manifest-reachable installed-wheel import graph.

Direct-CUBIN serving has an independent fail-closed blocker: `release_runtime.py` does
not propagate the signed native build's `OPTIMA_CUTE_COMPILE_PROFILE_DIGEST` into the
engine process, while the CUBIN runtime requires that exact profile identity.

Release preparation also treats native output generically: it reopens a caller-supplied
`NativeArtifactPublication` inventory and binds its `NativeBuildSpec`; it does not run the
provider prebuild or provider-specifically reopen the CuTe index and compile profile. A
release pipeline must retain and verify those provider-specific authorities before the
generic release identity can be accepted.

Release authorization also requires closure of the open review surface around:

- effective SGLang argv authority, including abbreviations, duplicate/loader flags, and
  management interfaces;
- binding the signed container-reproducibility statement to the actual builder output,
  not only a later mutable tag readback;
- authenticated or disabled management routes when serving uses host networking;
- release/session binding for serving receipts and complete archive/model/build TOCTOU
  accounting; and
- production key custody, rotation, rollback, registry retention, and deployment policy.

No final deterministic registry pair, authorized image, TP8 serving run, or complete
all-rank serving receipt set is claimed for current HEAD. See [Releases](/docs/architecture/releases)
and [Release workflow](/docs/engine/release-workflow) for the gates this closes.

## Operational invariants and open controls

- One authoritative executor manager must own each protected device/resource pool. Two
  independent managers do not share the quiescence lock.
- Settlement restart authenticates retained attempt bytes and stored PASS dispositions; it
  is not a full causal regrade. Preserve the private provider/plan authorities needed to
  reconstruct the exact `CausalQualificationInput` if regrade is an operational promise.
- Queue-age admission backpressure is not automatic disposal. An operator must explicitly
  release a hold with an audited reason or use the typed expiry transition after its
  minimum age; stale non-terminal rows can otherwise block overlapping settlement.
- Generation-zero or orphan arena stacks can block global projection. The repository has
  no typed retirement API; direct database deletion is not a supported operating path.
- Weight reconciliation refreshes the metagraph and resumes an exact retained pending
  intent across later heads. Its observations are not an atomic chain snapshot, and its
  journal deadline is not timelock-reveal-aware.
- FlashInfer dependency overlays build each logical architecture in a separate hermetic
  child because architecture selection is import-time state. Runtime installs only the
  row for the active device. During autotuning, both the deep producer and fused epilogue
  consumer remain stock so a candidate cannot influence tactic selection.
- The DSL-JIT carve-out admits only the registered CuTe form under closed binding and
  contribution-local reachability rules. It is defense in depth inside OCI isolation,
  not a proof that arbitrary Python or native code is safe.
