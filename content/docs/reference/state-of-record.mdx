## Post-main B300 intake findings

Three source changes after PR #56 materially refine the current boundary:

1. `804147c4`
   admits CuTe's `cute.compile(<@cute.jit function>, ...)` form under exact
   whole-module alias analysis. Only an external, allowlisted `cutlass.cute`
   alias bound once is eligible; shadowing, rebinding, local vendoring, other
   `.compile` receivers, and `builtins` access fail closed.
2. `3cff4731`
   moves candidate loading out of watched-module activation. `seam.activate()`
   now arms pass-through adapters but never imports miner code;
   `load_candidate_bundle()` runs only at positively identified
   `run_scheduler_process` entry. Detokenizer/output-path children therefore
   remain candidate-free, and active-member coverage must equal the scheduler
   rank count exactly.
3. `5325468a`
   extracts tracing-JIT admission into the stdlib-only validator-owned
   `DSL_JIT_ENTRYPOINTS` table in `optima/dsl_jit_policy.py`. The shared policy
   retains whole-module single-binding and contribution-local shadow checks and
   also refuses a string/bytes literal as the first positional compile
   argument. CuTe is the only admitted row; Triton needs no explicit
   `.compile` carve-out.

These commits are implemented, focused-tested, and included in the current
working-tree complete suite, but they are not merged into the `main` base named
above and do not erase the outstanding B300 empirical qualification and
release-image proofs below.

## Current product boundary

The source now implements the product/referee separation:

- Cacheon Engine is represented by `EngineReleaseManifest` and signed release artifacts, without chain runtime dependencies.
- The referee uses `EvaluationStackManifest`, registered arenas, isolated complete-engine execution, pristine reference quality, two-pass reproduction, transactional settlement, and journaled reward projection.
- A crown cannot enter a release until `promote_integrated_contribution()` reopens evidence and produces an approved integration record over reviewed source at a reviewed Git commit. The selected payload remains byte-for-byte bound to its crowned digest; only packaging outside that selected closure and later materialization may be normalized.
- Serving consumes integrated source and signed release identity, never a live proposal URL or chain row.

See [Architecture overview](/docs/architecture/overview), [Product model](/docs/architecture/product-model), and [Release architecture](/docs/architecture/releases).

## Hardened-path validation

The final hardening architecture has meaningful structural validation, but not a final empirical performance crown.

### What was exercised

- The exact finalized reveal and publication path was reopened from a live testnet event.
- A structural candidate completed two independent `PASS` authorities, settled into a generation-1 crown, and reopened correctly after restart.
- The structural candidate encoded a synthetic **100-parts-per-million** improvement. It was chosen to prove authority, evidence, and state transitions; it was not an empirical accelerator result.
- A live weight projection ran in dry-run mode and correctly retained `submitted=false`.
- OCI host primitives, materialized engines, native publication, streaming sessions, B/C/B′ lifecycles, pristine T, causal qualification, discovery, settlement, integration, and release components have focused tests and are covered by the current recorded 1,724-passed/18-skipped/0-failed suite.
- Release construction validates external Ed25519 trust, deterministic runtime artifacts, exact native/model/tree bindings, canonical SBOM/provenance, fail-closed runtime environment, registry manifest reopening logic, host inspection policy, and serve-receipt coverage in unit or focused tests.

### What this does not prove

The structural proof does not establish a real throughput win, B300 topology behavior, final calibration constants, an actual registry-reproducible image, or successful TP8 serving. It establishes that the authority and persistence machinery can carry an exact two-pass candidate through its state transitions without substituting pre-hardening evidence.

## Current honest nonclaims

At this snapshot, Cacheon does **not** claim any of the following:

- a real B300 candidate has passed the complete final B/C/B′/T path, independent reproduction, settlement, restart, and crown-backed weight publication;
- the signed release path has built two real registry images with the same raw manifest digest and then served one through the final host/runtime path;
- a final release has completed TP8 serving and all-rank active/fired/completed receipt verification;
- a production `ArenaServiceProvider` is included;
- SM103/CuTe compilation, NVLink/P2P/custom collective behavior, topology calibration, and TP4 half/role swaps are proven for the final path;
- emissions half-life, discovery lifetime/pool, margins, noise envelopes, or refresh policy are production-calibrated;
- mainnet operations, hosted artifact durability, monitoring, backups, rollout, and signing-key procedures are complete.

## B300 validation debt

The remaining hardware-specific proof is concentrated rather than architectural. A final B300 campaign must establish:

1. the exact SM103/CuTe and native-artifact build/load identities;
2. NVLink, P2P, NCCL, custom collective, and fused-epilogue behavior under the registered topology;
3. graphs-on verification and adversarial graph replay for real candidates;
4. calibrated B/B′ drift, speed margin, T quality, hidden-task, and retry policies;
5. concurrent TP4 halves where registered, followed by a role/half swap;
6. one real candidate through first `PASS`, independent reproduction, settlement, crown restart, and crown-backed weights;
7. integration of that crowned delta into reviewed source;
8. an actual signed release, reproducible registry pair, authorized image, TP8 serve, and complete seam receipts.

None of these items requires changing the product boundary. A failure may still expose an implementation defect, but the work is validation of the current architecture rather than permission to reuse a legacy crown.

## Operational caveats

### Executor manager single-writer assumption

The OCI executor's quiescence lock aliases one manager instance's `transaction_lock` ([`oci_backend.py`](https://github.com/latent-to/optima/blob/main/optima/eval/oci_backend.py)). Safety therefore assumes one authoritative executor manager per protected device/resource pool in a process. Constructing independent managers over the same pool would not create a shared lock and is outside the documented operating invariant.

### Evidence-record range

Tracked referee evidence records currently cover PRs #40 through #54. No `pr-0055` or `pr-0056` record exists, and [`check_referee_evidence.py`](https://github.com/latent-to/optima/blob/main/scripts/check_referee_evidence.py) freezes the expected range below 55. Extending the registry requires an explicit schema/range update rather than placing unchecked files beside it.

### Legacy code

Legacy developer and simulator code remains in the tree, including [`eval/_launch.py`](https://github.com/latent-to/optima/blob/main/optima/eval/_launch.py), [`eval/capability.py`](https://github.com/latent-to/optima/blob/main/optima/eval/capability.py), [`eval/throughput_kl.py`](https://github.com/latent-to/optima/blob/main/optima/eval/throughput_kl.py), and [`commit_reveal.py`](https://github.com/latent-to/optima/blob/main/optima/commit_reveal.py). Production authority roots are separated from these paths, but they have not been deleted.

The local simulator commands are `commit`, `reveal`, `ledger`, and `legacy-settle --development-only`. They are not alternate interfaces to production settlement.
