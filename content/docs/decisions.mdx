---
title: Key decisions
description: A single-page log of the major design decisions for Cacheon V1, why they were made, and what was considered and rejected.
---

This page exists so contributors and miners can answer "why does it work this way?" without re-reading the whole architecture doc. Decisions are listed roughly in order of how load-bearing they are.

## Inference framework: HuggingFace, not vLLM

**Decision:** Use HuggingFace Transformers for the harness.

**Reasoning:** vLLM manages its own KV cache internally via PagedAttention, and replacing it would mean fighting undocumented internals that change between versions. HuggingFace gives exact hook points — every attention layer is a plain `forward()` method. Both baseline and miner run on the same stack, so relative comparisons are valid.

**Rejected:** vLLM. PagedAttention is not hookable cleanly. Any custom policy would require a vLLM fork and would measure against vLLM-specific behavior that doesn't generalize.

## Incentive: king-of-the-hill, winner-take-all

**Decision:** One reigning champion. All emission to the king. Challengers must beat the king's score by more than a small, time-decaying margin (see [Defender's moat](#defenders-moat-decaying-epsilon--copycat-dq) below) to dethrone them.

**Reasoning:** Creates a sharp optimization target — miners optimize for "best," not "good enough to be top-N." GPU eval only runs when new challengers exist, which amortizes the cost of expensive H100-hours.

**Rejected:** Top-N split (e.g. 60/25/10/5%). Distributes incentive more broadly but weakens the signal. Miners optimize to land in the top bracket, which is noisier than optimizing to be best.

## Defender's moat: decaying epsilon + copycat DQ

**Decision:** The dethrone rule is two-stage.

1. **Byte-identical copies are disqualified.** A challenger whose `policy.py` `source_hash` (SHA-256 of the file bytes, computed on the CPU validator) matches the current king's hash, on a **different hotkey**, with a **strictly later** `commit_block`, is stored with `disqualify_reason = "duplicate_of_king"` and score `0`. The king keeps the throne. Earliest commit wins by construction. (The GPU may still run; the rewrite happens when the result is recorded in `ValidatorState.record_evaluation`.)
2. **Non-copies must exceed a decaying threshold.** A non-DQ'd challenger must strictly exceed `king_score × (1 + ε(age))` to take the crown, where:

   ```
   ε(age) = epsilon_initial × max(0, 1 − age / decay_blocks)
   age    = current_block − king.crowned_at_block
   ```

   Defaults: `epsilon_initial = 0.01` (1%), `decay_blocks = 50_400` (~7 days at ~12 s/block). Tunable via `CACHEON_KING_EPSILON_INITIAL` and `CACHEON_KING_EPSILON_DECAY_BLOCKS` env vars (`validator/config.py`).

**Reasoning:** Two problems motivated this:

- **Float noise.** Re-running the same policy on a different H100 run can shift `memory_reduction` or `latency_improvement` by tenths of a percent. A strict `>` rule would let that noise dethrone a genuinely-winning king on a coin flip.
- **Copycat submissions.** Without stage 1, the dominant strategy is "wait for someone to win, then copy their `policy.py` to a fresh hotkey and register." Free incentive with zero R&D.

The epsilon starts at 1% on the block the king is crowned, which is wide enough to swallow hardware noise and discourage near-duplicates, and narrow enough that real improvements still dethrone quickly. It decays linearly to 0 over a week, so a stale king with no real challengers doesn't accumulate a moat they didn't earn. After the decay window, the threshold equals `king_score` and the rule is strict `stored.score > king.score` again.

Byte-identical copies are caught in stage 1 by `source_hash` — faster and more decisive than relying on the score-noise band alone. See `validator/state.py`'s `_effective_dethrone_threshold` and `record_evaluation` for the full implementation.

**Rejected:**

- Strict `>` only: floats are noisy, this was too easy to game.
- Fixed-forever ε: rewards kings who got lucky once and never improve.
- Score smoothing (EMA): blurs which commit beat which, hurts auditability.

## Commitment model: one per hotkey, permanent

**Decision:** Each hotkey maps to exactly one policy commit. No re-submission on the same hotkey.

**Reasoning:** Commitment pressure — miners test locally before registering. Clean on-chain record: one hotkey = one policy version at one pinned commit SHA.

**Rejected:** Unlimited resubmission on the same hotkey. Turns the subnet into a free hyperparameter search service. Removes skin in the game.

## Submission transport: on-chain commit pointer

**Decision:** Miners publish policy source to a public Hugging Face repo and commit `{"repo": "<namespace>/<name>", "revision": "<40-char hex sha>"}` on-chain.

**Reasoning:** Policy code lives where code lives (on Hugging Face). On-chain record is a lightweight pointer that the harness resolves and pins to the exact commit SHA. Cleaner and more auditable than uploading bytes directly.

**Rejected:** Direct `.py` file upload on-chain. Clunky, size-limited, harder to audit.

## Eval hook shape: `write` + `attend`, not three hooks

**Decision:** The policy gets two hooks — `write(K, V, ...)` and `attend(Q, ...)`. `attend` returns the final `AttentionOutput`.

**Reasoning:** Three separate hooks (`write`, `score`, `aggregate`) force materialization of the `[batch, heads, q_len, kv_len]` score tensor, which kills fused approaches like TurboQuant at long context. They also create a circular dependency for eviction — H2O needs attention weights from the current step to decide what to evict in `write`, but weights are only computed in `score`.

`write` + `attend` lets the miner own the full pass. The policy object holds state across calls; all internal computation stays inside the class.

**Rejected:** Separate `score` and `aggregate` hooks. Kills TurboQuant-class strategies outright.

## Quality enforcement: hard KL gate, not soft penalty

**Decision:** `if KL > 0.1 nats: score = 0`, no partial credit.

**Reasoning:** A soft penalty creates incentives to trade output quality for efficiency in ways that are hard to detect. Over many rounds, the subnet would drift toward policies that are slightly worse at everything. A hard gate says: your job is to match baseline quality first, then optimize.

**Rejected:** `score -= k × KL` with some tunable `k`. Always leaks quality.

## Score weights: 60% memory, 40% latency

**Decision:** `score = 0.6 × memory_reduction + 0.4 × latency_improvement`.

**Reasoning:** Memory is the primary bottleneck at scale — KV cache memory determines max context length and max batch size. Latency matters but is secondary and can often be covered by hardware. Both weights are tunable once we see real submissions.

**Rejected:** 50/50. Undersells how load-bearing memory is at long context. Rejected for now, could shift toward 50/50 if V2 expands to latency-sensitive workloads.

## Memory measurement: CUDA allocator delta around policy run

**Decision:** Memory score uses `policy_memory_bytes` — the CUDA allocator delta measured by the harness after the full generation run (prefill + all decode steps), not `policy.memory_bytes()` (self-reported) or `torch.cuda.max_memory_allocated()` (peak VRAM).

```python
torch.cuda.synchronize(device)
gc.collect()
mem_before = torch.cuda.memory_allocated(device)
# ... prefill + decode ...
torch.cuda.synchronize(device)
gc.collect()
mem_after  = torch.cuda.memory_allocated(device)
policy_memory_bytes = max(0, mem_after - mem_before)
```

**Reasoning:** Self-reporting (`memory_bytes()`) is gameable — a policy can return `0` regardless of what it stores, inflating its apparent memory reduction. The CUDA allocator delta is non-gameable: it measures what the policy actually allocated on the GPU. `torch.cuda.empty_cache()` is intentionally omitted from the measurement block — it's expensive (forces allocator to release free blocks to the OS, adding hundreds of ms per run) and unnecessary (`memory_allocated()` counts live bytes only, not cached-free blocks). Measuring after the full run rather than after prefill is safe because the AST sandbox blocks all async/finalizer tricks, and the harness has no miner hook between the last `attend()` return and measurement.

**Supersedes (2026):** Earlier versions used `policy.memory_bytes()` (self-reported cache size). That was replaced first by `torch.cuda.max_memory_allocated()` (peak VRAM), then by the allocator-delta approach here. `memory_bytes()` has been removed from the `KVCachePolicy` interface and from `sandbox.REQUIRED_METHODS`.

**Rejected:**

- `policy.memory_bytes()` self-report: gameable, cannot be audited at runtime.
- `torch.cuda.max_memory_allocated()` as the memory axis: correct for "did this process fit on the GPU," wrong for "how much did you compress the KV cache." Peak VRAM includes model weights, transient attention buffers, and implementation artifacts.
- Measuring after prefill only: same result because the sandbox prevents timing attacks, but end-of-run is simpler and captures decode-appended cache entries too.

## KL evaluation: teacher-forced, not autoregressive comparison

**Decision:** KL divergence is computed by running the miner's policy on the **baseline's token sequence** in a single forward pass (`score_policy_on_sequence`), not by comparing logits from two independent autoregressive runs.

**Reasoning:** Two independent greedy decode runs produce slightly different logits due to quantization error or CUDA non-determinism. If the argmax flips even one token early in the sequence, the two runs diverge catastrophically — the miner feeds token₁' while the baseline fed token₁, and from that point the models see completely different inputs. KL explodes to 10-20 nats despite the policy being correct. Teacher-forcing eliminates this: both paths see identical tokens, so KL reflects only the numerical difference in attention computation.

**Rejected:** Comparing autoregressive logits directly. Produces false positives (correct policies DQ'd) due to greedy-argmax sensitivity. Even an identity passthrough policy can fail the KL gate this way due to Flash Attention FP non-determinism across separate invocations.

## Baseline latency: CUDA kernel warm-up

**Decision:** On baseline cache misses, `pod_eval.py` runs a throwaway single-prompt, single-token forward pass before the timed baseline.

**Reasoning:** The baseline is the first inference after model load. CUDA pays one-time costs: kernel JIT compilation, memory allocator warm-up, and clock ramping. Every miner after that runs on a warm GPU and appears artificially faster. A do-nothing passthrough policy was scoring 0.91 on latency improvement purely from warmup savings. The throwaway pass absorbs these costs (~1 second) so baseline and miners start from the same GPU state.

**Rejected:** Running baseline twice and using the second run's latency. Correct but doubles baseline time (~5 min → ~10 min) on cache misses. The single-token warm-up gets 95% of the benefit at 1% of the cost.

## Operating model: single-validator V1

**Decision:** One validator (the subnet owner), one GPU pod, one person setting weights.

**Reasoning:** Different hardware produces different KL values. An H100 and an A100 running the same policy produce slightly different FP results. If multiple validators ran independent eval, they'd disagree on rankings and produce conflicting weight signals. Centralized eval on a reference machine eliminates that variance.

The tradeoff — trust concentration — is real. V2's verifiable rollouts (publishing prompts, baseline logits, KL values) are the mitigation: anyone can audit results after the fact, even if they can't run eval themselves.

**Rejected (for V1):** Decentralized validation. Deferred to V2 once rollout publishing is in place.

## Security: CPU/GPU split, SSH transport

**Decision:** Wallet keys and chain access live on a CPU server. GPU pod only runs inference.

**Reasoning:** If the GPU pod is compromised (it runs arbitrary miner code after sandboxing, so it's a realistic attack surface), the attacker gets model weights and in-flight eval results — not wallet keys, not weight-setting capability. SSH between the two machines narrows the blast radius.

**Rejected:**

- **Everything on one machine**: Miner code and wallet keys on the same host is reckless.
- **HTTP service on the pod**: More moving parts, open port, auth layer. SSH is simpler and the pod's only interface becomes the SSH key.

## Sandbox: two layers — AST + subprocess isolation

**Decision:** Layer 1 is static AST analysis (fast feedback). Layer 2 is firejail subprocess isolation (security boundary).

**Reasoning:** Neither alone is enough. AST-only can be bypassed with enough cleverness. Subprocess-only gives miners terrible error messages—miner gets random timeouts at 300s rather than an `import os rejected` error in 5ms. Layered defense covers both.

**Rejected:**

- **AST-only**: Not airtight.
- **Subprocess-only**: Bad UX for miners.

## Triton: blocked in V1

**Decision:** Triton kernels are not allowed in V1 submissions.

**Reasoning:** Triton can read and write arbitrary GPU memory, and its kernel bytecode is harder to audit than PyTorch ops. PyTorch is sufficient to implement every known KV optimization approach correctly for V1. The kernel-level speedups Triton unlocks are a V2 concern with additional isolation (Triton IR static analysis + constrained allocator).

**Rejected:** Allow Triton in V1. Sandboxing cost not worth it for V1's scope.

## Prompts: PG19, block-hash seeded, 131K chars

**Decision:** PG19 passages, seeded from the Bittensor block hash, truncated to ~131K chars (~32K tokens).

**Reasoning:** KV-cache benchmarking needs long, continuous sequences. At 2K tokens, the Qwen2.5-7B GQA cache is ~0.7% of peak memory — memory reduction is invisible to scoring. At 32K tokens it's ~12%, large enough for compression gains to register. Block-hash seeding makes prompt selection deterministic and auditable; PG19's full-novel passages force the cache to grow.

**Rejected:**

- **FineWeb. Median documents &lt;1K tokens**: cache too small for signal.
- **Synthetic prompts (infilling, retrieval)**: Deferred to V2+ once V1 is running on real text.

## Attention weight sanity checks

**Decision:** Occasionally validate that attention weights sum to ~1 per query row.

**Reasoning:** Catches degenerate policies that collapse attention to a single token. Such policies can sometimes pass the KL gate by accident on easy prompts; the sanity check rejects them before they get scored.

**Rejected:** Check every round. Overhead is not worth it; sampled checks are sufficient.

---

All of these are open to revision. The rule is: change them when you have a real reason, not to be neat. Every decision here was expensive to make — reverting one without evidence costs more than keeping it.
