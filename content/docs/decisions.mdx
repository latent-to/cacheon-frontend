---
title: Key decisions
description: A single-page log of the major design decisions for Cacheon V1, why they were made, and what was rejected.
---

This page exists so contributors and miners can answer "why does it work this way?" without re-reading the whole architecture doc. Decisions are listed roughly in order of how load-bearing they are.

## Model: Qwen2.5-72B-Instruct on 4x H200

**Decision:** Start with 72B on multi-GPU from V1.

**Reasoning:** 72B on 4x H200 is a genuinely hard infrastructure challenge. It requires tensor parallelism, NVLink communication optimization, and distributed KV cache management. This is the "tip of the spear" positioning: the competition is inherently difficult, which makes winning meaningful. 14B on a single GPU is too easy and doesn't produce differentiation worth competing for. The mechanism is kept simple (one model, one endpoint, no spec-dec, no quantization) to offset the model-size complexity.

**Rejected:** Starting with 14B. Too easy to serve fast; insufficient optimization surface. Starting with 7B. Trivial.

## Endpoint: `/v1/chat/completions` only

**Decision:** Require only the chat completions endpoint. Do not require `/v1/completions`.

**Reasoning:** The target model is an instruct model (Qwen2.5-72B-Instruct) and the long-term positioning is "fastest backend for agentic workflows." Chat completions is the standard endpoint for instruct models. Requiring both endpoints doubles the API surface miners must implement. Raw completions (no chat template) is less aligned with production serving patterns.

**Rejected:** Requiring both `/v1/completions` and `/v1/chat/completions`. Unnecessary surface area.

## Baseline: vLLM, not HuggingFace Transformers

**Decision:** The baseline is a pinned vLLM instance serving the same model on the same hardware with tensor parallelism.

**Reasoning:** vLLM is the de facto standard for production LLM serving. It implements PagedAttention, continuous batching, FlashAttention, and other optimizations. Beating it requires genuine innovation. A HuggingFace Transformers baseline would be too easy to beat and would not produce a useful signal about production readiness.

**Rejected:** HuggingFace Transformers as baseline. Too slow. Beating vanilla HF is not impressive; beating vLLM is.

## Correctness: first-divergence logprob analysis

**Decision:** Per-prompt first-divergence gate with logprob sanity checks.

**Reasoning:** With multi-GPU tensor parallelism, floating-point reduction order varies slightly between runs. At positions where two tokens are nearly tied in probability, this noise can flip the greedy argmax. Once one token diverges, the entire rest of the sequence cascades. A global token-match threshold falsely disqualifies legitimate miners running the exact same model.

The first-divergence approach checks only the first mismatch per prompt: if the baseline's token is in the miner's top-5 logprobs with a gap `<= 0.5`, and the miner is using greedy decoding, the divergence is from TP noise and the prompt passes. This catches quantized models and wrong architectures while forgiving legitimate numerical variance.

**Rejected:** Global token-match threshold (fails on cascade). Full-vocabulary KL via API logprobs (impractical data volume).

## V1 restrictions: no spec-dec, no quantization

**Decision:** Speculative decoding and weight quantization are explicitly banned in V1.

**Reasoning:** Both techniques can produce legitimate token divergence that complicates correctness checking. Speculative decoding with draft model rejection sampling is mathematically equivalent to greedy decoding, but implementation bugs can cause subtle output differences. Quantization inherently changes the computation. Banning both in V1 lets us ship a tight correctness gate without false positives. V2 relaxes these restrictions with more nuanced correctness checking.

## Aggregation: median across prompts

**Decision:** Per-prompt TTFT and throughput are aggregated using the median across 10 prompts.

**Reasoning:** Median is robust to outliers (a single GC pause, memory allocation spike, or thermal throttle does not tank the score). Mean would let one bad prompt disproportionately affect the result. Trimmed mean is an alternative but more complex for no clear V1 benefit.

## Warmup: discard first 1-2 prompts

**Decision:** Before scored prompts, the harness sends 1-2 warmup prompts that are discarded.

**Reasoning:** First inference after model load is slower on every framework (CUDA graph compilation, KV cache pre-allocation, JIT warmup, GPU clock ramping). Including startup costs in the score would reward miners who optimize for fast warmup rather than fast steady-state serving. The competition should measure steady-state performance.

## Memory: constraint, not score axis

**Decision:** Memory efficiency is not directly scored. The constraint is: serve the model on the allocated GPUs. If you OOM, you fail.

**Reasoning:** External memory measurement of a containerized process is unreliable and gameable. Memory efficiency shows up indirectly through speed: better memory layout enables faster access patterns and the ability to handle longer contexts. Docker resource limits enforce the hard constraint.

## Prompts: hybrid instruct format

**Decision:** Instruct-style prompts built from curated templates filled with PG19 context passages.

**Reasoning:** The target model is an instruct model and the long-term story is "fastest backend for agent workflows." Raw PG19 text continuation does not reflect real serving workloads. Hybrid prompts test realistic request shapes (summarization, QA, agent traces) while preserving the deterministic, long-context properties of PG19 that differentiate servers on prefill.

**Rejected:** Raw PG19 continuation only. Not aligned with instruct model or production workloads.

---

All of these are open to revision. The rule is: change them when you have a real reason, not to be neat.
