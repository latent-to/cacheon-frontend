---
title: Key decisions
description: A single-page log of the major design decisions for Cacheon V1, why they were made, and what was rejected.
---

This page exists so contributors and miners can answer "why does it work this way?" without rereading the validators and evaluation sections. Decisions are listed roughly in order of how load-bearing they are.

## Model: Qwen2.5-72B-Instruct

**Decision:** Start with 72B on multi-GPU from V1.

**Reasoning:** 72B on 8-GPU TP=8 (H200/B200/B300) is a genuinely hard infrastructure challenge. It requires tensor parallelism, NVLink communication optimization, and distributed KV cache management. This is the "tip of the spear" positioning: the competition is inherently difficult, which makes winning meaningful. 14B on a single GPU is too easy and doesn't produce differentiation worth competing for. The mechanism is kept simple (one model, one endpoint) to offset the model-size complexity.

**Rejected:** Starting with 14B. Too easy to serve fast; insufficient optimization surface. Starting with 7B. Trivial.

## Endpoint: `/v1/chat/completions` only

**Decision:** Require only the chat completions endpoint. Do not require `/v1/completions`.

**Reasoning:** The target model is an instruct model (Qwen2.5-72B-Instruct) and the long-term positioning is "fastest backend for agentic workflows." Chat completions is the standard endpoint for instruct models. Requiring both endpoints doubles the API surface miners must implement. Raw completions (no chat template) is less aligned with production serving patterns.

**Rejected:** Requiring both `/v1/completions` and `/v1/chat/completions`. Unnecessary surface area.

## Baseline: vLLM, not HuggingFace Transformers

**Decision:** The baseline is a pinned vLLM instance serving the same model on the same hardware with tensor parallelism.

**Reasoning:** vLLM is the de facto standard for production LLM serving. It implements PagedAttention, continuous batching, FlashAttention, and other optimizations. Beating it requires genuine innovation. A HuggingFace Transformers baseline would be too easy to beat and would not produce a useful signal about production readiness.

**Rejected:** Hugging Face Transformers as the baseline. It is significantly slower than modern inference engines, and outperforming vanilla Transformers is not a meaningful benchmark. Cacheon uses vLLM because beating vLLM demonstrates a genuine inference improvement.

## Two vLLM images: generation baseline vs scoring baseline

**Decision:** Speed uses `vllm/vllm-openai:v0.22.0` as the generation baseline. Correctness teacher-forcing runs on a separate `vllm/vllm-openai:v0.9.2` container via `prompt_logprobs`.

**Reasoning:** These are different workloads. The generation baseline should reflect current production vLLM performance (torch.compile, full 32k context on mainnet). Teacher-forcing re-scores each miner's output text under canonical Qwen weights; that `prompt_logprobs` path is much more memory-hungry at long context than streaming generation. Newer vLLM builds (including v0.22.0 when used for scoring) OOM or return incomplete logprobs on ~6k-input teacher-forcing for Qwen2.5-72B on 8-GPU TP=8. v0.9.2 is the newest build validated for stable `prompt_logprobs` on that hardware, including B200 (`sm_100`). Eval prompts stay at ~6k input; the scoring container caps context at ~7168 tokens and runs with lower GPU memory utilization so correctness checks do not become validator-side infra failures.

**Rejected:** A single vLLM image for both speed and correctness. Either unreliable scoring (`scoring_infra_fail` DQs) or a downgraded generation baseline miners are not actually competing against. Bumping the scoring image before a newer vLLM handles long `prompt_logprobs` reliably on 8-GPU TP=8.

## Correctness: token match pre-filter + teacher-forcing

**Decision:** One prompt set (10 scored + 2 warmup). Speed and correctness use the same miner outputs. A token-match pre-filter disqualifies submissions with aggregate match below 10% (mean of per-prompt rates) before teacher-forcing runs. Teacher-forcing on a separate scoring vLLM (`v0.9.2`, `prompt_logprobs`) checks that output text is plausible under Qwen. To pass, mean logprob must be at least `-4.0`, min logprob per token must stay above `-12.0`, and all 10 scored prompts must pass.

**Reasoning:** Global token match alone produced false disqualifications when greedy decoding diverged early and cascaded across long outputs. A 10% aggregate pre-filter filters obvious failures (wrong model, broken inference) while allowing reasonable variance on long eval prompts. Teacher-forcing validates plausibility under canonical weights via the scoring vLLM's tokenizer, without requiring byte-identical stream tokens or baseline alignment. Cross-engine token-count comparison was removed after false disqualifications. Eval prompts stay at ~6k input to match the scoring container's context budget (see Two vLLM images above).

## Aggregation: median across prompts

**Decision:** Per-prompt end-to-end speed improvements are aggregated using the median across the 10 scored eval prompts.

**Reasoning:** Median aggregation prevents isolated events like GC pauses, memory allocation spikes or thermal throttling from distorting the final score. Using the mean would allow one bad prompt disproportionately affect the result. Trimmed mean is an alternative but more complex for no clear V1 benefit.

## Warmup: discard first 2 prompts

**Decision:** Before scored prompts, the harness sends 2 warmup prompts that are discarded.

**Reasoning:** First inference after model load is slower on every framework (CUDA graph compilation, KV cache pre-allocation, JIT warmup, GPU clock ramping). Including startup costs in the score would reward miners who optimize for fast warmup rather than fast steady-state serving. The competition should measure steady-state performance.

## Memory: constraint, not score axis

**Decision:** Memory efficiency is not scored directly. The requirement is simply that the model must run successfully on the allocated GPUs. If the server runs out of memory, your evaluation fails.

**Reasoning:** External memory measurement of a containerized process is unreliable and gameable. Memory efficiency shows up indirectly through speed: better memory layout enables faster access patterns and the ability to handle longer contexts. Docker resource limits enforce the hard constraint.

## Prompts: hybrid instruct format

**Decision:** Instruct-style prompts built from curated templates filled with PG19 context passages.

**Reasoning:** The target model is an instruct model and the long-term story is "fastest backend for agent workflows." Raw PG19 text continuation does not reflect real serving workloads. Hybrid prompts test realistic request shapes (summarization, QA, agent traces) while preserving the deterministic, long-context properties of PG19 that differentiate servers on prefill.

**Rejected:** Raw PG19 continuation only. Not aligned with instruct model or production workloads.

---

All of these are open to revision. The rule is: change them when you have a real reason, not to be neat.
