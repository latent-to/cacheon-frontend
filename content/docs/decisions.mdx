title: Key decisions
---

This page exists so contributors and miners can answer "why does it work this way?" without rereading the validator and evaluation sections. Decisions are listed roughly in order of how load-bearing they are.

## Submission unit: a kernel for one slot, not a whole container

**Decision:** A miner implements **one registered target** (a typed slot, or an atomic boundary spanning a few related slots) inside a fixed model the validator controls. The validator owns the weights, the graph, the tokenizer, the sampler, the prompts, the timing, and the reference. The miner only fills an output tensor the validator allocated, strictly upstream of the sampler.

**Reasoning:** This is the single most important design decision, and it is what makes the referee cheat-resistant. Because the miner never produces the final tokens or the logprobs, the "run gibberish and fetch the real answer from an API" attack has **no output to substitute**. The validator produces the tokens from the rest of the trusted model.

**Rejected:** Whole-model or whole-container submission (the Phase 1 approach). If the miner controls the entire forward pass, they can gut the model (drop layers, aggressively quantize) and then make the output look correct by fetching real answers from an API. The kernel-slot design structurally removes that attack class.

## Fidelity reference: a local stock-kernel run, not a production API

**Decision:** Speed and fidelity are measured by two different local references, launched separately so neither corrupts the other. Speed compares the candidate against a bracket of the frozen incumbent, launched immediately before and after it (B, C, B′), on the same box. Fidelity compares the candidate against a separately launched pristine reference (T) run over sealed prompt trajectories, decoupled from the timed launches so speed-oriented timing changes cannot alter measured correctness.

**Reasoning:** The references are local, exact, free, deterministic, and adversary-independent. Splitting speed from fidelity also fixed a real failure mode: coupling them let timing-driven nondeterminism from the speed launches occasionally fail an honest candidate's fidelity check. A separately launched pristine reference removed that false-failure path.

**Rejected:** A production API (for example a hosted model endpoint) as the per-round reference. It is non-reproducible (samples drift even at temperature 0 due to quantization and sampler differences), sparse (APIs return only top-k logprobs, so a true distribution distance is not computable), a network dependency inside the trusted loop, and consensus-breaking (two validators hitting the API get different samples and compute different scores, which breaks Bittensor consensus). Referencing a second trusted GPU pod is compatible but is just the same idea with more plumbing.

## Fidelity metric: per-token KL plus benchmark accuracy

**Decision:** Fidelity is judged by **per-token KL divergence** against the stock reference run (the dense, low-variance primary gate), backed by **task accuracy** on real benchmarks (GSM8K, MMLU) as a capability floor. The KL gate also caps the argmax-disagreement rate and a coverage or tail-mass guard, so a sparse cheat that keeps mean KL low by flipping a few tokens is still caught.

**Reasoning:** Because the validator owns an exact local reference run, it can compare the full per-token output distribution directly rather than only checking that outputs are plausible. KL is dense and low-variance, so it makes a good primary gate; benchmark accuracy is noisier and needs large sample counts, so it serves as a capability floor rather than the primary signal. The KL threshold is per-slot and calibrated to the model's own nondeterminism noise floor, because attention's flash-style softmax reorders arithmetic and sits at a higher floor than elementwise ops.

**Rejected:** A teacher-forcing plausibility gate (the Phase 1 approach), which checks that an output is plausible under the model's logprobs but does not measure exact distribution match. Cacheon's two-launch design provides an exact reference, so it can measure the distribution difference directly.

## Engine: a pinned sglang version

**Decision:** All validators run the **same pinned sglang version** (`PINNED_SGLANG`), asserted by an `optima compat` canary. The seam patches this unmodified, pinned engine at runtime; sglang is never forked or reconfigured.

**Reasoning:** Bittensor consensus (Yuma) requires every validator to measure the same submission and get the same number. That requires pinning the engine version for the measurement environment. Runtime patching (rather than an engine config flag) also lets a miner introduce a genuinely novel kernel while every validator still runs the same pinned package. The pin binds the **validator**, not the miner: a submission targets a frozen slot contract, never a specific sglang version, so miners are decoupled from the pin and ride a version bump without resubmitting.

**Rejected:** Letting each validator or miner pick their own engine version. Two validators on different runtimes get different throughput and different numerics, which breaks Yuma consensus.

## Anti-copy: commit-reveal

**Decision:** Every proposal moves through `commit → timelocked reveal → intake → qualification → settlement`, continuously and per-proposal rather than in a synchronized round. Each miner first commits `H(content_hash, hotkey, salt)` on chain, then reveals `(content_hash, salt)` once the timelock clears; a reveal is accepted only if it matches a commitment that same hotkey posted earlier.

**Reasoning:** Submissions are evaluated in the open, so a copier could otherwise resubmit the current incumbent's bundle. A copier who only sees a rival's bundle at reveal time has no matching prior commitment for it and cannot reveal it. Copy detection additionally fires cumulatively across proposals on either the exact content hash or a reformat-invariant AST fingerprint.

**Rejected:** Evaluating in the open with no commitment (trivially copyable), and byte-hash-only deduplication (a reformatted or renamed copy dodges a plain hash).

## Emissions: per-target crowns, not a fixed split

**Decision:** Each registered target tracks its own current best contribution independently. A candidate crowns a target only after two independent qualification passes both clear the bar `1 + max(min_margin, noise_multiplier × noise)`, where `min_margin` and `noise_multiplier` come from a frozen calibration and `noise` is the measured spread between two incumbent launches. The first pass moves a candidate to `reproduction_pending`; a second, independently attempted pass must reproduce it before anything crowns, and the settled speedup is the lower of the two. Reward is age-decayed credit on that margin, not a flat emission share: `credit = floor((speedup_ppm - 1,000,000) × half_life / (half_life + age_blocks))`, so credit halves every half-life, counted from the block the winning proposal was submitted, not from when settlement happened to run.

**Reasoning:** A per-target crown still rewards specialists: a miner who owns one target is paid, rather than everything going to a single best end-to-end bundle. The calibrated bar, not a plain `>` or a flat percentage, prevents rankings from flipping on timing noise, and requiring two independent passes means a single lucky measurement can't crown anything on its own. Age-decayed credit rewards being first and being better, and decays a stale advantage instead of paying an unchanging share forever.

**Rejected:** A single global champion across all targets (does not reward specialization), a flat percentage margin applied regardless of measured noise (miscalibrates on quieter or noisier hardware), and a one-pass crown (a single noisy measurement could crown a kernel that never reproduces).

## Scoring regime: CUDA graphs on

**Decision:** Throughput is scored with **CUDA graphs on**. Op seams are graph-captured directly; a block or collective kernel must declare `graph_safe: true` in its metadata to run inside the graph, otherwise it falls back to the trusted baseline under capture.

**Reasoning:** Graphs-off cripples the baseline roughly 4.5 to 6.5 times, so a graphs-off "win" is meaningless. Beating sglang, vLLM, or TensorRT with graphs on is the entire point, so scoring only ever happens in that regime. An undeclared kernel that cannot be captured falls back rather than being allowed to wedge the graph.

**Rejected:** Graphs-off scoring, which would inflate apparent speedups against a crippled baseline and reward kernels that cannot run in the regime that matters in production.

## Submission language: Triton and CuteDSL source, not raw CUDA

**Decision:** Miners submit **source**, never a compiled artifact. The default path is Triton or CuteDSL. A target that explicitly permits it can also accept declared `.cu`/`.cuh` CUDA source, built by a validator-owned, reviewed builder rather than the miner's own toolchain. A target that permits a registered direct-artifact provider can accept CuTe source that the validator itself compiles into a sealed CUBIN during an isolated, no-network prebuild; the miner never supplies compiled output, only the source the validator compiles.

**Reasoning:** Source kernels keep submissions inspectable and narrow the attack surface: the host launch code still runs in-process, so isolation is still required, but there is no opaque compiled artifact to hide behind. Inspectable source is also what makes an accepted win reusable IP rather than a one-off patch.

**Rejected:** Raw CUDA extensions or prebuilt binaries supplied directly by the miner. Whether the source is Triton, CuteDSL, or reviewed CUDA, and whether it runs directly or through a sealed validator-compiled CUBIN, the validator builds the executable artifact from visible source — never the miner.

All of these are open to revision. The rule is: change them when you have a real reason, not to be neat.
